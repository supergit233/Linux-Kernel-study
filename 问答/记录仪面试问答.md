# Linux 驱动岗位面试日志

## 项目：5G/WiFi 联网执法记录仪

## 一、项目整体架构

### 面试官问题

你简历上写了这个 5G/WiFi 联网执法记录仪项目。请你从 Linux 驱动角度介绍一下这个项目的整体架构。

重点说明：

- Hi3516 在系统里负责什么？
    
- IMX586 摄像头数据怎么进入系统？
    
- WiFi 模组通过什么接口接入？
    
- ICM-20948 陀螺仪通过什么接口接入？
    
- 5G/4G 模组在 Linux 中一般表现为什么设备？
    
- 你负责或重点研究的是哪些部分？
    

---

### 你的回答

这个项目是基于 Hi3516 + Linux 5.10，摄像头通过 MIPI 连接，视频流通过 VI 接收 sensor 数据，输出 YUV 格式给 VPSS 处理，处理完后给到 VENC 进行 H.265/H.264 编码，然后通过 WiFi/4G 用 RTSP 将 H.264 传到本地，同时保存 H.265 格式为 MP4。

WiFi 通过 SDIO 接口，被 MMC 枚举识别后，通过用户态 WiFi 程序连接网络。

4G 通过 USB 接口，在 `/dev` 下枚举出 `ttyUSB*` 设备，通过指定的 AT 接口进行调试，走无线网卡模式。

陀螺仪通过 I2C 接口传输数据，时间戳同步后基于轴向转换进行补偿。目前是在 VENC 编码后进行防抖补偿，我全面负责。

---

### 面试官点评

整体架构回答方向是对的，但有一个关键错误：

你说“在 VENC 编码后进行防抖补偿”，这个说法面试中风险很大。

一般视频防抖补偿应该发生在**编码前的 YUV 阶段**，而不是编码后的 H.264/H.265 码流阶段。

原因是：

H.264/H.265 是压缩码流，里面不是完整的一帧一帧原始图像，而是经过帧间预测、块划分、运动估计、熵编码之后的数据。你不能直接对压缩码流做旋转、平移、裁剪补偿。

如果真的要在编码后补偿，就需要：

```text
H.264/H.265 解码 → 得到 YUV → 做防抖变换 → 再重新编码
```

这样代价很高，嵌入式实时场景一般不会这么做。

更合理的链路是：

```text
Sensor → MIPI CSI → VI → ISP → VPSS/VGS 防抖补偿 → VENC 编码 → RTSP 推流 / MP4 存储
```

---

### 推荐回答

这个项目基于 Hi3516 + Linux 5.10。Hi3516 作为主控 SoC，负责系统运行、外设驱动、音视频采集处理、编码、存储和网络传输。

摄像头 IMX586 通过 MIPI CSI-2 接入，控制通道走 I2C，数据通道走 MIPI。Sensor 输出 RAW 数据后，由 Hi3516 的 VI 模块接收，再经过 ISP 处理成 YUV 数据，然后送到 VPSS 做缩放、裁剪、分路等处理，最后送到 VENC 做 H.264/H.265 硬件编码。

WiFi 模组通过 SDIO 接入，Linux 中由 MMC/SDIO 子系统枚举，驱动 probe 成功后注册成网络设备，用户态可以看到 `wlan0`，然后通过 `wpa_supplicant` 连接 AP。

4G/5G 模组通过 USB 接入，Linux 下通常会枚举出 `ttyUSB*` 串口设备和一个网络设备。`ttyUSB*` 用于 AT 指令调试，网络数据可以走 ECM、RNDIS、MBIM、QMI 等模式。

ICM-20948 陀螺仪通过 I2C 接入，驱动读取角速度和加速度数据，结合视频帧时间戳做同步，计算图像补偿量。防抖补偿发生在编码前的 YUV 阶段，通过 VPSS/VGS 或软件做旋转、平移、裁剪，然后再送 VENC 编码。

我重点研究的是 V4L2 视频采集链路、IMX586 sensor 驱动、SDIO WiFi 驱动移植、I2C 陀螺仪数据读取和防抖链路调试。

## 二、V4L2 subdev 与 sensor 驱动

### 面试官问题

你说摄像头通过 MIPI 接入。IMX586 这种 sensor 驱动，在 Linux 里一般不是直接生成 `/dev/video0`，而是作为 V4L2 subdev 接入。

请讲一下：

- V4L2 subdev 是什么？
    
- sensor 驱动主要负责什么？
    
- sensor 的 I2C 和 MIPI 分别负责什么？
    
- `s_stream()` 是干什么的？
    
- 设置分辨率、帧率、曝光、增益，大概通过什么方式完成？
    

---

### 你的回答

subdev 是作为一个抽象的独立模块，提供一种功能，多个 subdev 首尾连接，在 media controller 下组成整套视频链路 V4L2 device，对外暴露一个或多个 video device 接口。

sensor 驱动主要负责摄像头的控制、初始化。

I2C 控制摄像头，MIPI 传输视频数据。

`s_stream()` 不知道。

设置分辨率是在用户态程序通过 ioctl 命令，在 open、查询能力后设置图像格式 `VIDIOC_S_FMT`。

---

### 面试官点评

你对 subdev 的理解基本正确。

V4L2 subdev 可以理解成视频链路里的一个子模块，比如：

```text
sensor
MIPI CSI receiver
ISP
decoder
bridge
```

这些模块通过 media controller 组成一条 media pipeline，最终由 video device 暴露 `/dev/videoX` 给用户态。

但是 sensor 驱动不只是“控制、初始化”。更完整地说，sensor 驱动负责：

```text
1. 上电、复位、时钟、pinctrl、regulator
2. I2C 读写 sensor 寄存器
3. chip id 检测
4. 配置分辨率、帧率、RAW 格式、MIPI lane 数、MIPI 速率
5. 曝光、增益、翻转、测试图等 control
6. 实现 V4L2 subdev ops，比如 s_stream、set_fmt、get_fmt
7. 电源管理，比如 runtime PM、suspend/resume
```

I2C 和 MIPI 的关系你答对了：

```text
I2C  = 控制面，用来读写 sensor 寄存器
MIPI = 数据面，用来传输 RAW 图像数据
```

`s_stream()` 是 sensor subdev 中非常核心的接口。它用于启动或停止 sensor 输出图像流。

典型逻辑：

```c
static int imx586_s_stream(struct v4l2_subdev *sd, int enable)
{
    if (enable) {
        /*
         * 写 mode 寄存器表
         * 配置分辨率、帧率、MIPI 参数、曝光默认值等
         * 最后写 streaming on，比如 0x0100 = 0x01
         */
    } else {
        /*
         * 写 streaming off，比如 0x0100 = 0x00
         */
    }

    return 0;
}
```

用户态调用 `VIDIOC_STREAMON` 后，最终会沿着：

```text
video device → bridge/CSI/VI → sensor subdev
```

调用到 sensor 的：

```text
s_stream(1)
```

让 sensor 真正开始输出 MIPI 数据。

设置分辨率你只答了一半。用户态确实可以通过 `VIDIOC_S_FMT` 设置格式，但对 sensor subdev 来说，底层通常进入：

```text
.set_fmt()
.get_fmt()
```

然后驱动从 mode 表里选择最接近的分辨率和 bus format。真正写寄存器生效，很多时候是在 `s_stream()` 里完成的。

曝光、增益通常不是靠 `VIDIOC_S_FMT`，而是靠 V4L2 controls：

```text
V4L2_CID_EXPOSURE
V4L2_CID_ANALOGUE_GAIN
V4L2_CID_DIGITAL_GAIN
V4L2_CID_VBLANK
V4L2_CID_HBLANK
```

---

### 推荐回答

V4L2 subdev 是 V4L2 框架中对视频链路子模块的抽象。像 sensor、MIPI CSI、ISP、decoder 都可以作为 subdev。多个 subdev 通过 media controller 连接成一条 pipeline，最终由 video device 暴露 `/dev/videoX` 给用户态。

IMX586 sensor 驱动主要负责 sensor 的上电、复位、时钟、I2C 寄存器读写、chip id 检测、分辨率和帧率配置、MIPI 输出参数配置、曝光增益控制，以及实现 V4L2 subdev ops。

I2C 是控制通道，用来配置 sensor 寄存器；MIPI CSI-2 是数据通道，用来传输 sensor 输出的 RAW 图像数据。

`s_stream()` 是 sensor subdev 的启动/停止流接口。`s_stream(1)` 时，驱动通常会写入当前 mode 的寄存器表，并打开 streaming；`s_stream(0)` 时关闭 sensor 输出。

用户态通过 `VIDIOC_S_FMT` 设置分辨率和像素格式，底层会调用到 video device 或 subdev 的 format 相关回调；曝光、增益一般通过 V4L2 controls 设置，最终进入 sensor 驱动的 `.s_ctrl()`，再通过 I2C 写 sensor 寄存器。

## 三、V4L2 用户态采集流程

### 面试官问题

用户态通过 V4L2 采集一帧图像，大概需要哪些 ioctl？从打开 `/dev/video0` 到拿到一帧数据，完整流程是什么？

重点讲：

```text
open
VIDIOC_QUERYCAP
VIDIOC_S_FMT
VIDIOC_REQBUFS
VIDIOC_QUERYBUF
mmap
VIDIOC_QBUF
VIDIOC_STREAMON
VIDIOC_DQBUF
VIDIOC_QBUF
VIDIOC_STREAMOFF
```

---

### 你的回答

从 open 打开 video0 开始；用户查询下当前支持的格式；设定想要的格式。如果是分辨率、格式就走 ioctl；曝光、增益等摄像头配置走 ioctl -> v4l2 control。

然后向内核请求想要的内核缓冲区；核实查看得到的缓冲区；`mmap` 用户态内存映射得到的内核缓冲区，同时入队；开始传输。

循环操作：等待 `buffer_done` 完成，将 buf 放入 done_list，开始出队，紧接出队后空 buf 入队。

得到想要的数据后，结束传输。

---

### 面试官点评

这轮回答整体不错，流程基本正确。

可以修正几个表达：

“查询当前支持的格式”可以拆开说：

```text
VIDIOC_QUERYCAP：查询设备能力
VIDIOC_ENUM_FMT：枚举支持的像素格式
VIDIOC_ENUM_FRAMESIZES：枚举分辨率
VIDIOC_ENUM_FRAMEINTERVALS：枚举帧率
```

“曝光、增益走 ioctl -> v4l2_ctl”要注意：

```text
v4l2-ctl 是用户态工具
它本质也是调用 ioctl
驱动里对应的是 V4L2 control framework
```

`mmp` 应该是 `mmap`。

“出队后空 buf 入队”说得对，这是 V4L2 streaming 模式的核心。

---

### 推荐回答

用户态先：

```c
fd = open("/dev/video0", O_RDWR);
```

然后通过：

```text
VIDIOC_QUERYCAP
```

查询设备是否支持 video capture、streaming 等能力。

接着通过：

```text
VIDIOC_S_FMT
```

设置图像格式、分辨率、像素格式。

然后使用：

```text
VIDIOC_REQBUFS
```

向内核申请一组 buffer。

再用：

```text
VIDIOC_QUERYBUF
```

查询每个 buffer 的 offset 和 length。

然后通过：

```text
mmap
```

把内核 buffer 映射到用户空间。

接着调用：

```text
VIDIOC_QBUF
```

把这些 buffer 入队，告诉驱动这些 buffer 可以拿去采集。

之后调用：

```text
VIDIOC_STREAMON
```

启动采集。

采集过程中，用户态一般通过 `poll/select/epoll` 等待帧完成，然后调用：

```text
VIDIOC_DQBUF
```

取出已经填充好的 buffer。用户处理完这帧数据后，再调用：

```text
VIDIOC_QBUF
```

把空 buffer 放回驱动队列，循环使用。

结束采集时调用：

```text
VIDIOC_STREAMOFF
```

停止采集，并释放 mmap、关闭 fd。

## 四、V4L2 controls 与 ioctl 的关系

### 面试官问题

你刚才提到了曝光、增益。V4L2 controls 跟用户态 ioctl 是什么关系？它们有什么区别？

---

### 你的回答

ioctl 是统一的控制入口，不同的 IO 命令指向不同的 subdev，其中指向 sensor 的 IO 命令通过 vdev 的转发后进入 sensor 的驱动。这里根据命令不同，驱动会采用不同的 ops。这里不暴露给用户态的 ops 就是 v4l2 control。

---

### 面试官点评

这个回答里有几个概念混在一起了。

第一，ioctl 不是“不同命令直接指向不同 subdev”。

用户态通常操作的是：

```text
/dev/video0
```

ioctl 先进入 video device 的 ioctl 处理函数。这个 ioctl 最后是否转发到 sensor subdev，要看命令类型和驱动框架。

例如：

```text
VIDIOC_S_FMT
```

可能先由 video device、CSI、VI 处理，再传递给 sensor subdev 的 `.set_fmt()`。

```text
VIDIOC_STREAMON
```

会启动整条 video pipeline，最终可能调用 sensor 的 `.s_stream(1)`。

```text
VIDIOC_S_CTRL
```

如果这个 control 属于 sensor，比如曝光、增益，才会进入 sensor control handler 的 `.s_ctrl()`。

更准确地说：

```text
ioctl 进入 video device
V4L2/media 框架根据 pipeline、entity、ctrl handler
把请求分发到对应模块
```

第二，V4L2 controls 不是“不暴露给用户态的 ops”。

恰恰相反，V4L2 controls 是专门**暴露给用户态的参数控制机制**。

用户态可以执行：

```bash
v4l2-ctl -d /dev/video0 --list-ctrls
v4l2-ctl -d /dev/video0 --set-ctrl exposure=100
v4l2-ctl -d /dev/video0 --set-ctrl analogue_gain=16
```

程序里则可以调用：

```c
ioctl(fd, VIDIOC_S_CTRL, &ctrl);
```

controls 暴露的是“参数”，比如：

```text
曝光
增益
水平翻转
垂直翻转
测试图案
白平衡
```

ops 是驱动内部的实现函数，比如：

```text
.s_ctrl()
```

---

### 推荐回答

ioctl 是用户态访问 V4L2 设备的统一系统调用入口。用户态不会直接调用驱动函数，而是通过不同的 ioctl 命令和 `/dev/videoX` 交互。

V4L2 controls 是 V4L2 框架中专门用于管理可调参数的机制，比如曝光、模拟增益、数字增益、白平衡、翻转、测试图等。

用户态可以通过：

```text
VIDIOC_G_CTRL
VIDIOC_S_CTRL
VIDIOC_G_EXT_CTRLS
VIDIOC_S_EXT_CTRLS
```

这些 ioctl 来读写 control。

驱动里会通过：

```c
v4l2_ctrl_handler_init()
v4l2_ctrl_new_std()
v4l2_ctrl_new_std_menu()
```

注册 control。

当用户设置曝光时，V4L2 core 会根据 control id 找到对应的 `v4l2_ctrl`，最终调用驱动注册的：

```c
v4l2_ctrl_ops->s_ctrl()
```

然后驱动在 `.s_ctrl()` 中通过 I2C 写 sensor 对应寄存器。

## 五、设置 sensor 曝光的完整路径

### 面试官问题

如果用户态执行：

```bash
v4l2-ctl -d /dev/video0 --set-ctrl exposure=1000
```

请你从用户态到 sensor 寄存器，把完整路径讲一下。

重点说明：

- `v4l2-ctl` 本质调用什么？
    
- 内核通过什么找到对应 control？
    
- 驱动里哪个回调被调用？
    
- 最后怎么真正改变 sensor 曝光？
    
- 曝光值是不是直接写 1000 到寄存器？
    

---

### 你的回答

`v4l2-ctl` 本质在调用对应驱动提前写好的 ops 函数，`ctrl_ops`。

`VIDIOC_S_CTRL` 让 V4L2 core 知道这是一个设置 control 的操作，`ctrl_id` 会去找这个设备提供的功能，找到后会用 `v4l2_ctrl_ops` 的 `.s_ctrl` 回调。

这个回调会根据输入的 id 执行对应的寄存器写入操作，将 value 按照寄存器的地址规则写入。

改变 sensor 曝光，是通过 I2C 框架写入寄存器。如果摄像头像前面说的 MIPI 传数据，I2C 控制。

曝光值是不是直接写，要看设备寄存器的定义是否支持直接写入。

---

### 面试官点评

这轮回答基本合格。

需要修正一点：

用户态的 `v4l2-ctl` 不是直接调用驱动 ops，而是调用 ioctl。

更准确是：

```text
v4l2-ctl
→ open("/dev/video0")
→ ioctl(fd, VIDIOC_S_CTRL / VIDIOC_S_EXT_CTRLS)
→ V4L2 core
→ 找 ctrl handler
→ 找到对应 v4l2_ctrl
→ 调用 v4l2_ctrl_ops->s_ctrl()
→ sensor 驱动通过 I2C 写寄存器
```

曝光值是否直接写 1000，取决于 sensor 数据手册和驱动定义。

很多 sensor 的曝光单位不是“毫秒”，而是：

```text
line
```

也就是曝光行数。

驱动可能要根据：

```text
像素时钟 pclk
一行长度 line_length
帧长度 frame_length
vblank
最大曝光限制
寄存器位宽
```

把用户传入的值转换成 sensor 寄存器值，再拆成高低字节写入多个寄存器。

---

### 推荐回答

用户执行：

```bash
v4l2-ctl -d /dev/video0 --set-ctrl exposure=1000
```

后，`v4l2-ctl` 会打开 `/dev/video0`，然后通过：

```text
VIDIOC_S_CTRL
```

或者：

```text
VIDIOC_S_EXT_CTRLS
```

把 control id 和 value 传给内核。

V4L2 core 根据 control id，在当前 video device 或 subdev 绑定的 `v4l2_ctrl_handler` 中找到对应 control。

如果这个 control 属于 sensor，比如曝光，最终会调用 sensor 驱动注册的：

```c
v4l2_ctrl_ops->s_ctrl()
```

在 `.s_ctrl()` 中，驱动根据：

```c
ctrl->id
```

判断这是：

```text
V4L2_CID_EXPOSURE
```

然后把用户传入的曝光值转换成 sensor 寄存器需要的格式，最后通过 I2C 写入曝光相关寄存器。

曝光值不一定是直接把 1000 写进寄存器。很多 sensor 的曝光单位是 line，驱动需要根据行长、帧长、时钟、vblank 限制等转换，再拆成高低字节写入多个寄存器。

## 六、SDIO WiFi 从上电到 wlan0

### 面试官问题

你说 WiFi 模组通过 SDIO 接入，被 MMC 枚举识别。那 Linux 里 SDIO WiFi 从上电到出现 `wlan0`，大概经历哪些步骤？

重点说明：

- DTS 里需要配置什么？
    
- SDIO host controller 和 WiFi function 怎么匹配？
    
- MMC/SDIO 枚举大概做了什么？
    
- WiFi 驱动的 `probe()` 什么时候被调用？
    
- firmware 什么时候加载？
    
- 最后为什么会出现 `wlan0`？
    

---

### 你的回答

首先设备树层面，看 SDIO bus 有没有开。在下面新建一个设备节点，里面可以先降低频率，提高识别成功率；加上一个固定在板子上的属性，让内核主动去询问。

## 八、SDIO WiFi 中断与收包流程

### 面试官问题

WiFi 芯片收到 AP 发来的数据后，Linux 驱动是怎么知道“有数据来了”的？从中断到网络协议栈，大概流程是什么？

- `cap-sdio-irq` 和 out-of-band GPIO 中断有什么区别？
    
- 驱动中断处理里为什么不能做太重？
    
- 驱动如何读取 RX buffer？
    
- 最后如何把数据交给 Linux 网络协议栈？
    

---

### 你的回答

---

### 面试官补充讲解

WiFi 芯片收到 AP 发来的无线数据后，不会一直等主控轮询，而是通过中断通知 host：

```text
我 RX buffer 里有数据了，你来读。
```

常见有两种中断方式。

第一种是 SDIO in-band interrupt。

也就是通过 SDIO 总线本身的中断机制通知主机。DTS 里常见配置：

```dts
cap-sdio-irq;
```

意思是 SDIO host 支持 SDIO IRQ，WiFi function 可以通过 SDIO 协议触发中断。

第二种是 OOB GPIO interrupt。

OOB 是 out-of-band，也就是不用 SDIO 总线中断，而是 WiFi 模组额外拉一根 GPIO，例如：

```text
WL_WAKE_HOST
```

WiFi 芯片收到数据后拉高或拉低这个 GPIO，触发 SoC 的 GPIO 中断。

两者区别：

```text
in-band SDIO IRQ：走 SDIO 总线协议，硬件连接少
OOB GPIO IRQ   ：额外一根中断脚，响应直接，也常用于低功耗唤醒
```

从中断到协议栈的流程大概是：

```text
AP 发送无线数据
→ WiFi 芯片接收
→ 芯片内部解调、校验、解密
→ 数据放入 RX buffer
→ 通过 SDIO IRQ 或 WL_WAKE_HOST GPIO 通知 host
→ Linux SDIO host 收到中断
→ 调用 WiFi 驱动注册的中断处理函数
→ 中断处理函数确认 RX ready
→ 唤醒 bottom half / workqueue / RX thread
→ RX 线程通过 CMD53 读取 RX buffer
→ 驱动解析私有头、802.11/802.3 数据
→ 构造 sk_buff
→ 调用 netif_rx() 或 napi_gro_receive()
→ Linux 网络协议栈处理 IP/TCP/UDP
→ 用户态 socket 收到数据
```

为什么中断里不能做太重？

因为中断上下文要求尽快返回，不能长时间占用 CPU，不能做可能睡眠的操作。SDIO 大块读取、协议解析、分配 skb 这些操作通常放到底半部、workqueue、tasklet、NAPI 或专门 RX 线程中完成。

---

### 推荐回答

WiFi 芯片收到 AP 数据后，会把数据放入芯片内部 RX buffer，然后通过 SDIO in-band interrupt 或单独的 WL_WAKE_HOST GPIO 中断通知主控。

如果走 SDIO in-band interrupt，需要 host 支持 `cap-sdio-irq`；如果走 OOB GPIO interrupt，则是 WiFi 模组单独用一根 GPIO 唤醒主控。前者走 SDIO 协议，硬件连接简单；后者是额外中断脚，常用于低功耗唤醒。

Linux 侧收到中断后，驱动中断处理函数一般只读取中断状态，确认是 RX ready，然后唤醒 bottom half、workqueue 或 RX 线程。真正的数据读取不会放在中断里做。

RX 线程通过 SDIO CMD53 从 WiFi 芯片 RX buffer 中读取数据，解析芯片私有头和网络帧，然后构造成 `sk_buff`，最后通过 `netif_rx()` 或 NAPI 接口提交给 Linux 网络协议栈。协议栈继续处理 IP/TCP/UDP，最终用户态应用通过 socket 读到数据。

## 九轮之后的额外追问：mmc0:0001 到 mmc0:0001:1 失败

### 你的问题

`mmc0` 是 host 成功，`mmc0:0001` 是 card 成功，`mmc0:0001:1` 是 func 成功。

`mmc0:0001` 到 `mmc0:0001:1` 之间出错，具体会是什么问题？我之前看说修改设备树可以解决。

---

### 讲解

你的理解是对的：

```text
mmc0             = SDIO host controller 成功注册
mmc0:0001        = SDIO card 被识别成功
mmc0:0001:1      = SDIO function 1 被枚举成功
```

`mmc0:0001` 出现，说明主控已经能和 SDIO 设备进行基础通信。

但是从 card 到 function，内核还要继续读取：

```text
CCCR：SDIO card 公共控制寄存器
FBR ：Function Basic Register，每个 function 的基础信息
CIS ：Card Information Structure，里面有 vendor/device/class 等信息
```

读取成功后，内核才会创建：

```c
struct sdio_func
```

然后 sysfs 中才会出现：

```text
/sys/bus/sdio/devices/mmc0:0001:1
```

如果 card 有了，但 function 没有，说明：

```text
SDIO 基础响应有了
但读取 function 信息、切换总线参数、供电/复位/时钟/宽度等某一步失败
```

常见原因如下。

### 1. bus-width 配置错误或 4-bit 不稳定

很多 SDIO 设备早期 1-bit 阶段能识别 card，所以有：

```text
mmc0:0001
```

但后面切到 4-bit 后，读取 FBR/CIS 失败，于是没有：

```text
mmc0:0001:1
```

调试时可以先改成：

```dts
bus-width = <1>;
```

如果 1-bit 下 function 能出来，4-bit 不行，重点查：

```text
D1/D2/D3 线
pinctrl 复用
上拉
电平
走线和信号质量
```

### 2. max-frequency 过高

有些板子低速阶段能识别 card，但继续初始化或升频后通信不稳定，读取 CIS 失败。

可以先降频：

```dts
max-frequency = <25000000>;
```

甚至：

```dts
max-frequency = <10000000>;
```

如果降频后 `mmc0:0001:1` 出来了，就说明是时序或信号质量问题。

### 3. pinctrl 配置不完整

SDIO 必须保证这些线都配置正确：

```text
CLK
CMD
D0
D1
D2
D3
```

尤其是 4-bit 模式下，D1-D3 也必须正确复用成 SDIO 功能。

通常：

```text
CMD、D0-D3 需要上拉
CLK 通常不需要上拉
```

如果只配好了 CMD/D0，1-bit 阶段可能正常，但 4-bit 或 function 读取阶段就可能失败。

### 4. 电源时序 / reset GPIO / WL_REG_ON 不稳定

有些 WiFi 芯片刚上电时可以对 SDIO 有部分响应，但内部还没完全 ready。host 太早枚举，就可能 card 出现了，但 function 读取失败。

可以通过 `mmc-pwrseq-simple` 控制上电时序：

```dts
wifi_pwrseq: wifi-pwrseq {
    compatible = "mmc-pwrseq-simple";
    reset-gpios = <&gpioX Y GPIO_ACTIVE_LOW>;
    post-power-on-delay-ms = <200>;
};

&sdio0 {
    status = "okay";
    non-removable;
    bus-width = <4>;
    max-frequency = <25000000>;
    mmc-pwrseq = <&wifi_pwrseq>;
};
```

关键点：

```text
reset-gpios / WL_REG_ON 控制 WiFi 释放复位
post-power-on-delay-ms 给 WiFi 芯片启动时间
```

### 5. non-removable / cap-sdio-irq / keep-power-in-suspend

这些一般不是 card 到 function 失败的首要原因，但会影响板载 WiFi 的枚举和稳定性。

常见配置：

```dts
&sdio0 {
    status = "okay";
    bus-width = <4>;
    max-frequency = <25000000>;
    non-removable;
    cap-sdio-irq;
    keep-power-in-suspend;
};
```

含义：

```text
non-removable         ：告诉内核这是板载设备，不依赖插卡检测
cap-sdio-irq          ：host 支持 SDIO 中断
keep-power-in-suspend ：休眠时保持供电
```

---

### 这个问题的面试推荐回答

如果 `mmc0:0001` 出现，说明 SDIO host 已经识别到了 card。但如果没有 `mmc0:0001:1`，说明 card 后续 function 枚举失败。内核在这个阶段会读取 CCCR、FBR、CIS 等信息，并创建 `sdio_func`。

失败原因可能是读取 function 信息时通信不稳定，比如 4-bit 模式切换失败、频率过高、D1-D3 引脚复用或上拉配置错误，也可能是 WiFi 上电时序不对，芯片还没 ready。

我会先在 DTS 里把：

```dts
bus-width = <1>;
max-frequency = <10000000>;
```

然后确认：

```text
pinctrl
WL_REG_ON
reset-gpio
mmc-pwrseq
power-on-delay
```

如果这样 function 能出来，再逐步恢复 4-bit 和高频率定位问题。

关键结论是：

```text
card 能出来但 function 出不来，通常不是驱动 id table 的问题，
而是 SDIO function 枚举阶段通信或板级配置问题。
```

# Linux 驱动岗位面试日志（二）

## 第十轮：V4L2 buffer 机制

### 面试官问题

用户态已经通过 `mmap` 拿到了内核 buffer 的地址，为什么还要调用 `VIDIOC_QBUF` 入队？

重点回答：

```text
1. mmap 到底做了什么？
2. QBUF 的作用是什么？
3. 驱动内部有哪些 buffer 队列？
4. DMA 写满一帧后 buffer 状态怎么变？
5. DQBUF 是从哪里取 buffer？
```

### 你的回答

`mmap` 是让用户态跟内核态指向同一个地址，减少 user-kernel 之间的 copy。

但数据 `STREAMON` 开始需要视频链路提供空 buf，V4L2 提供了 `wait_list`、`done_list` 两个链表，表示空闲和装满的 buffer，用于视频流周转。

`QBUF` 就是将空 buf 放入 `wait_list`，让视频流可以将数据放入这些准备好的 buf。没有这一步，视频数据没有地方放。

写满一帧后，`buf_done` 将 buf 放入 `done_list` 并通知出队，`DQBUF` 从 `done_list` 取出。

### 点评

你的核心理解是对的：

```text
mmap 只是映射
QBUF 才是把 buffer 交给驱动参与采集
DQBUF 是取出已经完成的 buffer
```

需要修正的点是：

不能简单说“用户态和内核态指向同一个地址”，更准确是：

```text
用户态虚拟地址  ┐
               ├── 映射到同一块物理 buffer / DMA buffer
内核态虚拟地址  ┘
```

用户态和内核态的虚拟地址通常不是同一个，但它们映射的是同一块底层 buffer。

### 推荐回答

`mmap` 只是把内核申请的采集 buffer 映射到用户空间，让用户态可以直接访问这块 buffer，减少一次从内核到用户态的数据拷贝。但 `mmap` 本身并不表示这个 buffer 已经交给驱动使用。

`VIDIOC_QBUF` 的作用是把某个 buffer 入队，告诉 V4L2 驱动：这个 buffer 当前是空的，可以交给硬件 DMA 或采集链路写入数据。

驱动内部一般会维护 buffer 状态和队列，比如：

```text
空闲队列
等待采集队列
active 队列
done 队列
```

用户调用 `QBUF` 后，buffer 从用户持有状态变成 queued 状态。启动 `STREAMON` 后，驱动把 queued buffer 交给硬件 DMA。

当硬件采集完一帧后，中断或回调里会调用类似 `vb2_buffer_done()` 的接口，把 buffer 状态标记为 DONE，并放入 done 队列，同时唤醒等待的用户进程。

用户态调用 `VIDIOC_DQBUF` 时，就是从 done 队列里取出已经填充好的 buffer。处理完后再调用 `QBUF` 把它重新交回驱动循环使用。

## 第十一轮：V4L2 buffer 和 DMA

### 面试官问题

你刚才说硬件 DMA 会往 buffer 里写数据。那这个 buffer 是谁分配的？为什么硬件 DMA 能写进去？

重点回答：

```text
1. VIDIOC_REQBUFS 之后内核做了什么？
2. buffer 为什么要满足 DMA 要求？
3. 什么是物理地址、虚拟地址、DMA 地址？
4. 用户态 mmap 看到的地址能不能直接给 DMA 用？
5. CPU 和 DMA 同时访问 buffer 会不会有 cache 一致性问题？
```

### 你的回答

buf 是用户通过 `REQBUFS` 指令向内核申请，内核根据实际情况分配，通过 `mmap` 让用户态指向内核分配好的 buf，之后 DMA 就可以直接写。

### 点评

你答对了前半部分：

```text
buffer 是用户通过 VIDIOC_REQBUFS 请求
由内核或 videobuf2 框架分配
用户态通过 mmap 映射后访问
```

但“之后 DMA 就可以直接写”还不够。面试官会继续问：

```text
DMA 写的是用户态地址？
内核虚拟地址？
还是物理地址 / DMA 地址？
```

关键点：

> DMA 硬件不能直接使用用户态虚拟地址。

用户态看到的地址，比如：

```text
0x7fxxxxxx
```

只是用户进程虚拟地址。硬件 DMA 真正需要的是：

```text
DMA 地址 / 总线地址
```

这个地址由内核通过 DMA API 或 V4L2/videobuf2 框架管理。

### 地址区别

```text
用户虚拟地址：
    用户态 mmap 后访问 buffer 用

内核虚拟地址：
    内核驱动访问 buffer 用

DMA 地址 / 总线地址：
    硬件 DMA 控制器访问 buffer 用
```

三者不是一回事。

### 推荐回答

`VIDIOC_REQBUFS` 后，V4L2 驱动或 videobuf2 框架会根据驱动选择的内存模型分配 buffer。对于视频采集硬件，通常会分配适合 DMA 的内存，比如物理连续的 DMA buffer，或者通过 scatter-gather/IOMMU 映射的 buffer。

用户态通过 `mmap` 拿到的是用户虚拟地址，它只是方便用户程序访问数据，不能直接给硬件 DMA 用。

硬件 DMA 真正使用的是内核通过 DMA API 得到的 DMA 地址或总线地址。驱动会把这个 DMA 地址配置到 VI/CSI/DMA 控制器寄存器中，硬件采集到数据后写入这块 buffer。

如果 CPU cache 和 DMA 同时访问这块 buffer，还要考虑缓存一致性。DMA 写完后 CPU 读之前，需要保证 cache 失效或同步，否则 CPU 可能读到旧数据。通常通过 coherent DMA buffer 或 DMA API 的 cache sync 机制解决。

## 第十二轮：V4L2 多 buffer 的意义

### 面试官问题

为什么 V4L2 采集一般要申请多个 buffer，比如 3 个或 4 个？只申请 1 个 buffer 行不行？

重点回答：

```text
1. 多 buffer 解决什么问题？
2. 硬件采集和用户处理为什么需要并行？
3. 用户处理太慢会发生什么？
4. buffer 数量是不是越多越好？
5. 实际项目中怎么选 buffer 数量？
```

### 你的回答

多个 buf 组成流水线，可以有效缓解数据阻塞问题。不然一个 buf 要写，还需要先等它从 `done_list` 出队，然后再入队。

用户处理太慢，会有大量 buf 在 `done_list` 等待。如果全部 buf 都写满并进入 `done_list`，就没有空 buf 在 `wait_list`，数据就没有地方写入，要么暂停处理，要么丢掉数据等待有空 buf。

理论上只对 stream 数据流转这一环节来说，buffer 越多越好。但更多 buf 需要更多符合条件的内存分配，DMA 需要完整的连续内存，内核可能不能分配很多内存。

buf 数量主要用于用户处理数据和视频流出数据之间缓冲，主要看这两个模块之间的处理速度差距，一般先设 4 到 6 个。

### 点评

这题回答得很好。

V4L2 streaming 采集本质上是生产者消费者模型：

```text
硬件/驱动采集帧  = 生产者
用户态处理帧    = 消费者
buffer 队列      = 中间缓冲池
```

如果只有一个 buffer：

```text
硬件写 buffer
→ 用户 DQBUF 取走
→ 用户处理
→ 用户 QBUF 还回来
→ 硬件才能写下一帧
```

这样硬件采集会被用户处理阻塞。

多个 buffer 可以形成流水线：

```text
硬件正在写 A buffer
用户正在处理 B buffer
C/D buffer 还在队列中等待下一帧
```

### 推荐回答

V4L2 申请多个 buffer 是为了让硬件采集和用户态处理形成流水线。硬件可以在一个 buffer 上 DMA 写入新帧，用户态同时处理上一个已经完成的 buffer。

如果只申请一个 buffer，用户态处理这帧时，硬件没有空 buffer 可以写下一帧，采集链路会被阻塞，实时视频流很容易丢帧。

如果用户处理太慢，done 队列会堆积，最后所有 buffer 都被填满或被用户态持有，驱动没有空 buffer 可用，后续帧可能被丢弃或采集等待。

buffer 数量不是越多越好，因为视频 buffer 通常是 DMA buffer，占用连续内存或 IOMMU 映射资源。buffer 多了会增加内存占用和端到端延迟。

实际项目中一般根据分辨率、帧率和用户处理耗时选择，常见是 3 到 6 个，调试初期可以先用 4 个。

## 第十三轮：I2C sensor 驱动 probe 流程

### 面试官问题

一个 I2C sensor 驱动的 `probe()` 一般会做哪些事情？

重点回答：

```text
1. 怎么拿设备树资源？
2. 怎么检查 sensor 是否存在？
3. 怎么初始化 V4L2 subdev？
4. 怎么注册 controls？
5. 怎么和 media controller 建立关系？
6. probe 里会不会直接开始出图？
```

### 你的回答

上电识别芯片，看是否有回应，`register_subdev` 函数注册为 subdev 模块。

controls 不知道是什么。

要通过 entity、link、pad 跟 media 建立联系，会注册 pad，media 系统之后会调用 `media_entity_pads_init` 等函数将 entity、link、pad 联系起来。

不会直接出图，出图前需要检查整个链路是否通了。

### 点评

你知道一些关键名词：

```text
subdev
media entity
pad
link
chip id
probe 里不直接出图
```

但流程还不完整。一个 sensor probe 通常包括：

```text
1. 分配私有结构体
2. 从设备树获取 clock、GPIO、regulator、pinctrl、endpoint
3. 上电 sensor
4. I2C 读取 chip id
5. 初始化 v4l2_subdev
6. 初始化 media entity / pad
7. 注册 V4L2 controls
8. 异步注册 subdev
9. 下电或进入 runtime PM
```

### 推荐回答

I2C sensor 驱动 probe 里首先会分配私有结构体，然后从设备树获取资源，比如 I2C client、xvclk、reset/pwdn GPIO、regulator、电源、pinctrl、MIPI endpoint 等。

然后驱动会按照 sensor 的上电时序打开 regulator、使能时钟、释放 reset，再通过 I2C 读取 chip id，确认 sensor 存在且通信正常。

确认成功后，会初始化 `v4l2_subdev`，通常用 `v4l2_i2c_subdev_init()` 绑定 I2C client 和 subdev ops，设置 entity function 为 camera sensor。

接着初始化 V4L2 controls，比如曝光、模拟增益、vblank、hblank、翻转、测试图案等，并把 ctrl handler 挂到 subdev 上。

然后初始化 media entity pad。sensor 一般有一个 source pad，通过 `media_entity_pads_init()` 注册，后续通过 media controller 和 MIPI CSI 的 sink pad 建立 link。

最后通过异步注册接口注册 subdev，比如 `v4l2_async_register_subdev()`。

probe 里一般不会直接开始出图，因为此时用户还没有设置格式，buffer 没有准备，MIPI/VI/ISP pipeline 也没有启动。真正开始输出图像是在用户态 `VIDIOC_STREAMON` 后，框架调用 sensor 的 `s_stream(1)`。

## 第十四轮：sensor chip id 读取失败排查

### 面试官问题

如果 IMX586 sensor 驱动 probe 失败，dmesg 显示读取 chip id 失败，你会怎么排查？

重点回答：

```text
1. I2C 设备树地址是否正确
2. I2C 控制器和 pinctrl 是否正常
3. sensor 电源是否正常
4. reset / pwdn GPIO 电平是否正确
5. MCLK / XVCLK 是否输出
6. 上电时序和延时是否符合要求
7. 用哪些命令或工具辅助排查
```

### 你的回答

那就是设备没有被成功识别，一般是设备树、通信接口、引脚的问题。先简单看下软件部分，比如设备树、pinctrl，然后检查硬件接的是否正确。

### 点评

方向对，但太泛。需要说清楚：

```text
查什么
怎么查
看到什么现象说明什么问题
```

chip id 读取失败，本质是：

```text
CPU 通过 I2C 访问 sensor 指定寄存器失败
```

排查链路：

```text
设备树配置
→ I2C 控制器
→ pinctrl
→ 电源
→ reset/pwdn
→ MCLK
→ I2C 总线
→ sensor 寄存器
```

### 推荐回答

如果 sensor 驱动 probe 时读取 chip id 失败，我会先确认这是 I2C 通信失败，还是读到了错误的 ID。

第一层先查设备树，确认 sensor 挂在哪个 I2C 控制器下，`reg = <0x1a>` 这类 I2C 地址是否正确，`compatible` 是否匹配驱动，I2C 控制器 `status` 是否打开。

第二层查 pinctrl，确认 I2C 的 SCL/SDA 引脚是否复用到 I2C 功能，上拉是否正确，电平域是否匹配。

第三层查 sensor 上电，包括 AVDD、DVDD、DOVDD 这些电源是否起来，电压是否正确，上电顺序和延时是否满足要求。

第四层查 reset 和 pwdn GPIO，确认电平极性是否写反，sensor 是否真的退出 reset/powerdown。

第五层查 MCLK/XVCLK，sensor 一般需要外部时钟，比如 24MHz。如果没有 MCLK，I2C 有时也可能无法正常响应。

第六层用工具辅助验证，比如看 `dmesg`，用 `i2cdetect` 看 I2C 地址是否有 ACK，用示波器或逻辑分析仪看 SCL/SDA 有没有波形、有没有 ACK。

如果地址能 ACK 但 chip id 错，就查寄存器地址、读写时序、寄存器地址宽度，比如 8bit/16bit 地址、8bit/16bit 数据是否配置错。

## 第十五轮：sensor 能识别但不出图

### 面试官问题

假设 sensor 的 chip id 能读到，`s_stream(1)` 也调用了，但是应用层拿不到图像，或者画面黑屏/花屏，你会怎么排查？

重点回答：

```text
1. sensor 是否真的 streaming on
2. MIPI lane 数、lane 顺序、速率是否匹配
3. RAW10/RAW12 是否一致
4. 分辨率、帧率、hts/vts 是否匹配
5. VI/CSI/ISP 是否配置正确
6. buffer 是否正常 QBUF/DQBUF
7. dmesg 或硬件统计里看什么错误
```

### 你的回答

这个我不懂。

### 讲解

chip id 能读到，只能说明：

```text
I2C 控制通道正常
```

但不代表：

```text
MIPI 数据通道正常
```

不出图要查这几层：

```text
sensor 是否出流
MIPI 是否收到数据
CSI/VI/ISP 是否配置正确
buffer 是否正常流转
```

### 推荐回答

如果 chip id 能读到，说明 I2C 控制通道基本正常，但不代表 MIPI 数据通道正常。

我会先确认 sensor 是否真的 stream on，比如 `s_stream(1)` 是否执行成功，mode 寄存器表是否写入成功，最后的 streaming 寄存器是否打开。

然后检查 MIPI 参数，包括 lane 数、lane 顺序、link frequency、MIPI 速率是否和 sensor mode 表、DTS endpoint、CSI 接收端配置一致。

接着检查数据格式，比如 sensor 输出 RAW10 还是 RAW12，Bayer 顺序是 RGGB 还是 BGGR，media bus code 是否和 CSI/VI/ISP 端一致。

再检查分辨率、帧率、hts/vts、pixel rate 是否匹配，确认 VI/CSI/ISP pipeline 是否启动，media link 是否 enable。

用户态侧还要确认 `REQBUFS/QBUF/STREAMON/DQBUF` 流程是否正常。

如果 `DQBUF` 一直阻塞，说明没有 frame done，可能是 sensor 没出流或 MIPI/VI 没收到完整帧。

如果能 `DQBUF` 但花屏，重点查格式、stride、Bayer 顺序和 RAW 位宽。

最后看 dmesg 或平台寄存器统计里的 MIPI 错误，比如：

```text
CRC error
ECC error
lane sync error
FIFO overflow
frame end timeout
packet data type error
```

## 第十六轮：Linux I2C 三个核心对象

### 面试官问题

Linux I2C 设备驱动里，`i2c_driver`、`i2c_client`、`i2c_adapter` 分别代表什么？它们之间是什么关系？

重点回答：

```text
1. i2c_adapter 是什么？
2. i2c_client 是什么？
3. i2c_driver 是什么？
4. 设备树里的 I2C 节点怎么变成 i2c_client？
5. driver 和 client 怎么匹配后进入 probe？
```

### 你的回答

`i2c_adapter` 是控制器，提供物理总线。

`i2c_client` 是挂在 I2C 总线上的具体设备。

`i2c_driver` 是对应设备的驱动。

### 点评

这题回答基本正确。

更准确一点：

设备树方式下，主要通过：

```text
of_match_table 里的 compatible
```

和设备树节点里的：

```dts
compatible = "sony,imx586";
```

匹配。

### 推荐回答

`i2c_adapter` 表示 SoC 里的 I2C 控制器，也就是一条 I2C 总线，负责真正产生 I2C 时序。

`i2c_client` 表示挂在某条 I2C 总线上的一个具体从设备，比如 IMX586 sensor。设备树中 I2C 控制器节点下面的子节点会被 I2C core 解析成 `i2c_client`，其中 `reg` 表示 7-bit I2C 地址，`compatible` 用于和驱动匹配。

`i2c_driver` 是设备对应的驱动，里面有 `probe/remove` 和匹配表。对于设备树方式，I2C core 会用设备节点的 `compatible` 和驱动里的 `of_match_table` 匹配；匹配成功后调用驱动的 `probe()`，并把对应的 `i2c_client` 传进去。

驱动之后通过 `i2c_transfer()` 或 SMBus 接口，借助 client 所属的 adapter 和设备通信。

## 第十七轮：I2C 读写 sensor 寄存器

### 面试官问题

IMX586 这种 sensor 驱动里，通常会封装 `read_reg()` / `write_reg()` 函数。请你讲一下：

```text
1. 为什么 sensor 读写寄存器不用普通内存读写？
2. I2C 写一个 16-bit 地址、8-bit 数据的 sensor 寄存器，怎么构造 buffer？
3. I2C 读寄存器为什么通常要发两段 message？
4. i2c_transfer() 返回值怎么判断？
5. 如果读写偶尔失败，怎么处理？
```

### 你的回答

第一、第二个不知道。

发两段是因为第一段转到要读的位置，也就是假写，第二段开始读。

`transfer` 返回值不清楚。

### 点评

你答对了一点：

```text
I2C 读寄存器通常两段 message：
第一段假写寄存器地址
第二段读数据
```

其它部分需要补强。

### 讲解

sensor 是挂在 I2C 总线上的外部芯片，不是 SoC 内部 MMIO 外设。CPU 不能直接通过地址总线访问它的寄存器，只能通过 I2C 协议发送：

```text
设备地址 + 寄存器地址 + 数据
```

所以不能用：

```c
readl();
writel();
```

而是要用：

```c
i2c_transfer()
i2c_master_send()
regmap
```

### 写 16-bit 地址、8-bit 数据

比如写：

```text
寄存器地址：0x0202
数据：0x03
```

buffer 通常是：

```text
buf[0] = 0x02    // register high byte
buf[1] = 0x02    // register low byte
buf[2] = 0x03    // value
```

### 读寄存器为什么两段

第一段：

```text
write: reg high byte + reg low byte
```

告诉 sensor 要读哪个寄存器。

第二段：

```text
read: value
```

真正读取数据。

### `i2c_transfer()` 返回值

`i2c_transfer()` 成功返回的是完成的 message 数量，不是字节数。

```text
写一个 msg：成功返回 1
读两个 msg：成功返回 2
```

如果返回负数，是错误码。如果返回数量不是期望值，也要当成失败。

### 推荐回答

Sensor 是外部 I2C 从设备，不是 SoC 内部 MMIO 外设，所以不能用 `readl/writel` 直接访问寄存器，而是要通过 I2C 协议读写。

如果 sensor 是 16-bit 寄存器地址、8-bit 数据，写寄存器时一般构造 3 字节 buffer：高 8 位寄存器地址、低 8 位寄存器地址、数据值，然后通过 `i2c_transfer()` 或 `i2c_master_send()` 发送。

读寄存器通常用两个 `i2c_msg`。第一段是写寄存器地址，也就是假写，告诉 sensor 要读哪个寄存器；第二段设置 `I2C_M_RD`，真正读取数据。

`i2c_transfer()` 成功返回的是完成的 message 数量，不是字节数。写一个 msg 成功应返回 1，读两个 msg 成功应返回 2。

如果 I2C 读写偶尔失败，可以对关键寄存器增加 retry 和短延时，并打印错误码和寄存器地址。但如果频繁失败，就要排查 I2C 速率、上拉、电源、reset、MCLK、pinctrl 和总线波形。

## 第十八轮：ICM-20948 轮询和中断

### 面试官问题

ICM-20948 这种 I2C 传感器，既可以轮询读数据，也可以用 data ready 中断。你说一下轮询和中断方式的区别，以及项目里为什么更适合用中断？

重点回答：

```text
1. 轮询方式怎么做？
2. 中断方式怎么做？
3. 中断处理函数里能不能直接通过 I2C 读很多数据？
4. 为什么 IMU 数据需要时间戳？
5. 如果 IMU 频率 200Hz、视频 30fps，怎么对齐？
```

### 你的回答

轮询就是内核一直主动询问是否有数据，一直发起读操作。

中断是陀螺仪数据准备好后向内核发出信号，内核接收到信号发出读取操作。

用中断可以减少资源占用，CPU 不用一直发起操作。

中断应该尽可能短，可以用中断线程去读取数据。

IMU 的数据是用于纠正视频被记录下的那一时刻的姿态。为了让数据对齐视频采集时间，需要一个时间戳。视频记录一个系统单调时间，IMU 也同时记录同一系统的时间，将这两个时间对应。

### 点评

这题回答不错。

核心点都说到了：

```text
轮询是定时主动读
中断是数据 ready 后通知主控
中断减少无效读取
硬中断里不要做重活
IMU 和视频帧需要同一时间基准
```

### 推荐回答

轮询方式是主控按照固定周期主动读取 IMU 数据，比如 200Hz 时每 5ms 读一次寄存器。它实现简单，但 CPU 会周期性唤醒，没数据也可能读，资源占用更高，时间点也不一定正好对应 IMU 数据 ready 的时刻。

中断方式是配置 ICM-20948 的 data ready 中断，当陀螺仪和加速度计采样完成后，通过 INT 引脚触发 SoC 的 GPIO 中断。驱动收到中断后，再读取 IMU 数据。这样有数据才读，CPU 和 I2C 总线占用更低，时间戳也更准确。

I2C 读操作可能睡眠，所以不适合在硬中断里做大量读写。一般硬中断里只确认中断或唤醒线程，真正的 I2C 读取放到 threaded irq 或 workqueue 中完成。

IMU 数据需要时间戳，是因为防抖要知道视频某一帧曝光时刻设备的姿态。视频帧和 IMU 都要使用同一个单调时间基准打时间戳。

比如 IMU 是 200Hz，视频是 30fps，可以在每帧视频时间戳附近找到前后两组 IMU 数据，通过插值或积分得到该帧对应的姿态补偿量。

## 第二十轮：嵌入式 Linux 启动流程

### 面试官问题

你说做过 Hi3516 平台系统移植。那从上电开始，一个嵌入式 Linux 系统是怎么启动起来的？

重点回答：

```text
1. BootROM 是什么？
2. U-Boot 做什么？
3. Kernel 怎么被加载？
4. DTB 什么时候用？
5. rootfs 怎么挂载？
6. init 进程什么时候启动？
7. 移植时主要改了哪些地方？
```

### 你的回答

上电后首先是 BootROM 启动，启动串口、时钟、DDR，然后从启动介质将 U-Boot、DTS 搬到内存。

U-Boot 准备启动参数，解压内核镜像，跳转到内核入口。

内核开始解析设备树与内存，初始化驱动框架，创建内核线程，初始化准备启动用户态，挂载 rootfs，挂载完文件系统启动 init。

U-Boot 适配：修改 defconfig 添加要适配的目标板子，然后修改对应的 Kconfig、Makefile、board.c。对于换了板载硬件的在这里改，然后编译完得到 boot.bin。

之后添加 bootcmd 用于指示如何搬运内核，bootargs 用于指示串口跟文件系统地址。

内核可以先用相近的 defconfig，然后 menuconfig 进行功能配置，编译得到内核镜像。

文件系统制作好，跟内核镜像、DTS 放入 SD 卡，先前的 bootargs 指向这个 SD 卡，然后上机供电。

### 点评

整体框架有，但需要修正几处。

第一，BootROM 通常不负责完整初始化 DDR。更常见的是：

```text
BootROM
→ 加载 SPL / 前级 bootloader
→ SPL 初始化 DDR
→ 加载完整 U-Boot 到 DDR
```

第二，BootROM 通常不负责加载 DTS/DTB。DTB 一般由 U-Boot 加载，或者和内核镜像打包在一起。

第三，不是所有平台最终都叫 `boot.bin`。有的平台是：

```text
u-boot.bin
u-boot.img
u-boot-dtb.bin
SPL + u-boot.img
厂商打包后的 boot.bin
```

面试时可以说“最终生成板子可烧录的 boot 镜像”。

### 推荐回答

嵌入式 Linux 上电后，首先执行芯片内部固化的 BootROM。BootROM 根据启动引脚或 efuse 配置选择启动介质，比如 SPI Flash、eMMC、SD 卡，然后从固定位置加载 SPL 或 U-Boot 前级镜像。

SPL 或 U-Boot 早期阶段会完成 DDR 初始化、基础时钟、串口等初始化，然后把完整 U-Boot 加载到 DDR 中运行。

U-Boot 启动后，会初始化控制台、存储、网络等外设，读取环境变量，根据 `bootcmd` 从 Flash、SD 卡、eMMC 或 TFTP 加载 kernel 镜像和 DTB 到内存。

然后 U-Boot 通过 `bootm`、`bootz` 或 `booti` 跳转到内核入口，并把 `bootargs` 和 DTB 地址传给内核。

Kernel 启动后会解压自身，建立页表，初始化内存管理、调度器、中断、驱动模型，然后解析设备树，根据设备树创建 platform device、i2c client、spi device 等，再匹配驱动。

最后内核根据 `bootargs` 里的 `root=` 参数挂载根文件系统，启动第一个用户态进程 `/sbin/init`、`/init` 或 systemd/busybox init，系统进入用户空间。

## 第二十一轮：U-Boot 新板子如何被选中

### 面试官问题

你说 U-Boot 适配要改 defconfig、Kconfig、Makefile、board.c。那一个新板子在 U-Boot 里要能被 `make xxx_defconfig` 选中，通常需要哪些文件参与？

重点回答：

```text
1. configs/xxx_defconfig 是干什么的？
2. board/vendor/board/Kconfig 是干什么的？
3. board/vendor/board/Makefile 是干什么的？
4. 顶层或 SoC 级 Kconfig 怎么找到 board Kconfig？
5. board.c 主要放什么？
6. make xxx_defconfig 报找不到 board，可能是哪层没接上？
```

### 你的回答

`configs/xxx_defconfig` 是表明 target。

然后在 `arch/arm` 下有 Kconfig，里面写 config target 并 source 一个指定路径，这个路径就是编译用到的板级文件。

文件下的 Kconfig 会指定头文件与板级目录，Makefile 会指定编译产物。

`board.c` 会放板载硬件的初始化函数代码。

source 的路径不对就找不到 board。

### 点评

方向对，但需要更精确。

`configs/xxx_defconfig` 是板子的默认配置文件，里面会选中某个目标板，例如：

```config
CONFIG_TARGET_MYBOARD=y
CONFIG_DEFAULT_DEVICE_TREE="myboard"
```

`arch/arm/Kconfig` 或 SoC 级 Kconfig 的作用是 source 到板级 Kconfig。

`board/vendor/board/Kconfig` 真正定义板级配置，例如：

```kconfig
if TARGET_MYBOARD

config SYS_BOARD
    default "myboard"

config SYS_VENDOR
    default "mycompany"

config SYS_CONFIG_NAME
    default "myboard"

endif
```

`board/vendor/board/Makefile` 不是指定最终产物，而是指定哪些 `.c` 文件参与编译，例如：

```makefile
obj-y += board.o
```

### 关系图

```text
configs/myboard_defconfig
    |
    | 选择 CONFIG_TARGET_MYBOARD=y
    v
arch/arm/Kconfig 或 arch/arm/mach-xxx/Kconfig
    |
    | source "board/mycompany/myboard/Kconfig"
    v
board/mycompany/myboard/Kconfig
    |
    | 设置 SYS_VENDOR="mycompany"
    | 设置 SYS_BOARD="myboard"
    | 设置 SYS_CONFIG_NAME="myboard"
    v
board/mycompany/myboard/Makefile
    |
    | obj-y += board.o
    v
board/mycompany/myboard/board.c
```

### 推荐回答

一个新板子要能被 `make xxx_defconfig` 选中，首先需要在 `configs/` 下有对应的 `xxx_defconfig`。这个 defconfig 里会选中目标板配置，比如 `CONFIG_TARGET_MYBOARD=y`，并设置默认设备树、启动地址、启动介质等默认选项。

然后 U-Boot 的 Kconfig 系统里必须定义这个 `TARGET_MYBOARD`。通常是在 `arch/arm/Kconfig` 或 SoC 级 Kconfig 中 `source "board/mycompany/myboard/Kconfig"`，让板级 Kconfig 被包含进来。

在 `board/mycompany/myboard/Kconfig` 里，会定义目标板，并设置 `SYS_VENDOR`、`SYS_BOARD`、`SYS_CONFIG_NAME` 等，告诉 U-Boot 板级目录和配置头文件对应关系。

`board/mycompany/myboard/Makefile` 用来指定这个板级目录下哪些文件参与编译，比如 `obj-y += board.o`。

`board.c` 里放板级初始化代码，比如 GPIO、电源、复位、网口 PHY、DRAM 信息、启动阶段的特殊硬件初始化等。

如果 `make xxx_defconfig` 或编译时找不到 board，可能是 defconfig 里的 `CONFIG_TARGET_xxx` 没有对应 Kconfig 定义，也可能是 `arch/arm/Kconfig` 没有 source 到板级 Kconfig，或者 `SYS_VENDOR/SYS_BOARD` 配错导致 U-Boot 找不到 `board/vendor/board/` 目录。

## 插入问题：arch/arm/Kconfig 的 source 和 board Kconfig 是否重复

### 你的问题

`arch/arm/Kconfig` 的：

```kconfig
source "board/mycompany/myboard/Kconfig"
```

跟：

```text
board/mycompany/myboard/Kconfig
```

的内容不是重复了吗？

不重复。它们是**包含关系**和**定义关系**。

```text
arch/arm/Kconfig：
    只是把某个板子的 Kconfig 文件包含进来
```

## 第二十二轮：bootcmd 和 bootargs

### 面试官问题

U-Boot 里 `bootcmd` 和 `bootargs` 分别是什么？

重点回答：

```text
1. bootcmd 是什么时候执行的？
2. bootcmd 一般负责做什么？
3. bootargs 是给谁用的？
4. bootargs 常见参数有哪些？
5. 内核卡在挂载 rootfs，优先查 bootargs 里哪些参数？
```

### 你的回答

`bootargs` 提供串口跟文件系统，内核的保存地址并告知如何使用。

`bootcmd` 提供将内核、DTS 搬运到指定地址，用 `booti` 启动内核镜像。

### 点评

方向对，但有一点要纠正：

> `bootargs` 不是提供“内核的保存地址”。

内核镜像和 DTB 加载到内存哪个地址，主要由 `bootcmd` 里的 `load/fatload/ext4load/tftp` 命令决定。

`bootargs` 是传给 Linux kernel 的启动参数。

### bootcmd

`bootcmd` 是 U-Boot 自动启动时执行的一串命令。比如倒计时结束后，U-Boot 执行：

```bash
run bootcmd
```

它一般负责：

```text
从 Flash / eMMC / SD / TFTP 加载 kernel
加载 DTB
必要时加载 initramfs
调用 bootm / bootz / booti 启动内核
```

### bootargs

`bootargs` 是 U-Boot 传给 Linux 内核的命令行参数。

常见内容：

```text
console=ttyAMA0,115200
root=/dev/mmcblk0p2
rootfstype=ext4
rootwait
rw
init=/sbin/init
earlycon
mtdparts=...
```

### 推荐回答

`bootcmd` 是 U-Boot 自动启动时执行的命令，主要负责从启动介质或者网络加载 kernel 镜像和 DTB 到内存指定地址，然后通过 `bootm`、`bootz` 或 `booti` 跳转启动内核。

`bootargs` 是 U-Boot 传给 Linux kernel 的启动参数，主要告诉内核控制台串口、根文件系统位置、文件系统类型、挂载方式、是否等待 root 设备等。

比如：

```bash
setenv bootcmd 'fatload mmc 0:1 0x82000000 Image; fatload mmc 0:1 0x83000000 hi3516.dtb; booti 0x82000000 - 0x83000000'
```

表示从 mmc 加载内核和 dtb，然后启动。

```bash
setenv bootargs 'console=ttyAMA0,115200 root=/dev/mmcblk0p2 rootwait rw rootfstype=ext4'
```

表示内核控制台使用 `ttyAMA0`，根文件系统在 `/dev/mmcblk0p2`，等待 mmc 设备出现，以读写方式挂载 ext4。

如果内核启动后卡在挂载 rootfs，我会优先检查：

```text
root= 是否指向正确分区
rootfstype= 是否正确
rootwait 有没有加
内核是否编进了对应存储控制器驱动
内核是否支持对应文件系统
rootfs 里是否存在 /sbin/init 或指定 init 程序
```

## 第二十三轮：Linux 设备树作用

### 面试官问题

Linux kernel 移植时会改 DTS。那设备树在 Linux 启动中到底起什么作用？

重点回答：

```text
1. 设备树描述什么？
2. 设备树不应该描述什么？
3. 内核什么时候解析 DTB？
4. compatible、reg、interrupts、clocks、pinctrl 分别有什么用？
5. 为什么驱动代码里一般不直接写死 GPIO 号、寄存器地址、I2C 地址？
```

### 你的回答

内核通过 `dev->of.node` 解析设备树，刚开始就会解析设备树。

设备树描述硬件设备的连接方式。

`compatible` 用于跟驱动匹配。

`reg` 表示参数个数。

`interrupts` 表示中断号。

`clocks` 表示时钟。

`pinctrl` 表示引脚复用的规则。

驱动不写死可以一套驱动不同设备树来适配不同硬件，不用频繁改驱动代码。

### 点评

整体方向对，但有两个关键点要修正。

第一，`reg` 不是“参数个数”，而是描述设备的地址资源。

对 MMIO 设备：

```text
reg = <寄存器物理基地址 大小>
```

对 I2C 设备：

```text
reg = <I2C 7-bit 地址>
```

第二，“刚开始就会解析设备树”可以更准确：

内核启动早期会先解析 DTB，拿到内存、chosen、bootargs、CPU 等基础信息；后面在初始化平台总线和各子系统时，会根据设备树节点创建 platform device、i2c_client、spi_device 等设备对象，再和驱动匹配。

### 推荐回答

设备树主要用于描述板级硬件信息，比如外设寄存器地址、中断、时钟、GPIO、pinctrl、电源、总线设备以及设备之间的连接关系。它不应该描述应用层业务逻辑，比如录像策略、网络地址、防抖算法参数等。

U-Boot 启动内核时会把 DTB 地址传给 kernel。内核启动早期会解析 DTB，先获取 memory、chosen、bootargs、CPU 等基础信息，后续在 platform bus、I2C、SPI 等子系统初始化时，根据设备树节点创建对应的 device，再通过 `compatible` 和驱动的 `of_match_table` 匹配，匹配成功后调用 `probe()`。

`compatible` 用来匹配驱动。

`reg` 用来描述设备地址资源，比如 MMIO 基地址和大小，或者 I2C 从设备地址。

`interrupts` 描述中断号和触发方式。

`clocks` 描述设备使用的时钟。

`pinctrl` 描述引脚复用和上下拉等电气配置。

驱动代码里一般不写死 GPIO 号、寄存器地址、I2C 地址，是为了让驱动和板级硬件解耦。同一个驱动可以通过不同 DTS 适配不同板子，减少改驱动代码的次数。

## 第二十四轮：设备树 reg、#address-cells、#size-cells

### 面试官问题

为什么有些节点的 `reg` 是两个数，有些是一个数，甚至有些是四个数？这由什么决定？

重点回答：

```text
1. #address-cells 是什么？
2. #size-cells 是什么？
3. 父节点如何决定子节点 reg 的格式？
4. I2C 设备为什么常见 reg = <0x1a>？
5. MMIO 外设为什么常见 reg = <0x12040000 0x1000>？
```

### 你的回答

这个不太清楚。

### 讲解

设备树里 `reg` 的格式不是固定的，它由父节点的这两个属性决定：

```dts
#address-cells
#size-cells
```

一句话记忆：

> 子节点 `reg` 里“地址占几个 cell、大小占几个 cell”，由父节点的 `#address-cells` 和 `#size-cells` 决定。

### 什么是 cell

设备树里的一个 `<...>` 数字默认是一个 cell，通常是 32 bit。

比如：

```dts
reg = <0x12040000 0x1000>;
```

这里有两个 cell：

```text
0x12040000
0x1000
```

### `#address-cells`

表示子节点地址部分用几个 cell 表示。

```dts
#address-cells = <1>;
```

表示子节点地址用 1 个 cell。

```dts
#address-cells = <2>;
```

表示子节点地址用 2 个 cell，常见于 64 位地址。

### `#size-cells`

表示子节点 size 部分用几个 cell 表示。

```dts
#size-cells = <1>;
```

表示 size 用 1 个 cell。

```dts
#size-cells = <0>;
```

表示子节点 `reg` 里没有 size 字段。

### I2C 设备例子

```dts
&i2c1 {
    #address-cells = <1>;
    #size-cells = <0>;

    imx586@1a {
        compatible = "sony,imx586";
        reg = <0x1a>;
    };
};
```

父节点 `i2c1` 规定：

```text
地址占 1 个 cell
大小占 0 个 cell
```

所以子节点格式是：

```text
reg = <地址>
```

因此：

```dts
reg = <0x1a>;
```

表示 IMX586 的 I2C 7-bit 地址是 `0x1a`。

### MMIO 外设例子

```dts
soc {
    #address-cells = <1>;
    #size-cells = <1>;

    uart0: serial@12040000 {
        compatible = "arm,pl011";
        reg = <0x12040000 0x1000>;
    };
};
```

父节点规定：

```text
地址占 1 个 cell
大小占 1 个 cell
```

所以：

```dts
reg = <0x12040000 0x1000>;
```

表示：

```text
寄存器物理基地址：0x12040000
寄存器区域大小：0x1000
```

### 四个数的例子

```dts
soc {
    #address-cells = <2>;
    #size-cells = <2>;

    device@0,12040000 {
        reg = <0x0 0x12040000 0x0 0x1000>;
    };
};
```

父节点规定：

```text
地址占 2 个 cell
大小占 2 个 cell
```

所以格式是：

```text
reg = <地址高32位 地址低32位 大小高32位 大小低32位>
```

这里表示：

```text
地址 = 0x00000000_12040000
大小 = 0x00000000_00001000
```

### 推荐回答

`reg` 的格式不是固定的，而是由父节点的 `#address-cells` 和 `#size-cells` 决定。`#address-cells` 表示子节点地址字段占几个 cell，`#size-cells` 表示 size 字段占几个 cell。一个 cell 通常是 32 bit。

比如 I2C 控制器节点一般设置：

```dts
#address-cells = <1>;
#size-cells = <0>;
```

所以 I2C 设备的：

```dts
reg = <0x1a>;
```

只有一个地址，表示 7-bit I2C 地址。

SoC MMIO 总线节点一般设置：

```dts
#address-cells = <1>;
#size-cells = <1>;
```

所以：

```dts
reg = <0x12040000 0x1000>;
```

表示寄存器物理基地址和大小。

如果父节点是：

```dts
#address-cells = <2>;
#size-cells = <2>;
```

那么 `reg` 就会有四个 cell，用两个 cell 表示 64 位地址，两个 cell 表示 64 位大小。

## 你可以背的系统移植总述

> 系统上电后先执行芯片内部 BootROM，BootROM 根据启动介质加载 SPL 或前级 bootloader。SPL 完成 DDR 等早期初始化后加载完整 U-Boot。U-Boot 初始化串口、存储、网络等外设，根据 `bootcmd` 加载 kernel 和 DTB 到内存，并通过 `booti/bootz/bootm` 启动内核，同时通过 `bootargs` 传递控制台、rootfs 等参数。Kernel 启动后解析 DTB，初始化内存、调度、中断和驱动模型，根据设备树创建 platform device、i2c client、spi device 等对象，匹配驱动并调用 probe。最后根据 `root=` 参数挂载根文件系统，启动 `/sbin/init` 或 `/init`，进入用户空间。
