# HI3516CV610 板级落地

## 导读

### 本章定位

这一章把标准 SDIO 主线落到 HI3516CV610 板级环境里，重点看 DTS 如何选择控制器、`nebula,sdhci` 如何接入标准 SDHCI/MMC 框架，并把这些板级事实重新对照回 [[06-典型Function驱动例子]] 里已经建立好的最小 function driver 顺序。

### 核心对象

- `sdio0 / sdio1 / mmc0` DTS 节点
  - 板级选择和控制器用途描述
- `struct device_node`
  - host 节点和后续可能存在的 function child node 都通过它进入内核对象模型
- `nebula,sdhci`
  - HI3516CV610 私有 host compatible
  - 含义不是“另一套完全私有的 MMC/SDIO 框架”，而是“这套控制器以 `sdhci` 风格接入 Linux 标准 SDHCI 框架”
- `struct sdhci_host`
  - SDHCI 通用控制器对象
  - 它位于板级 host 驱动和 `mmc_host` 之间，负责承接标准 SDHCI 寄存器模型、通用中断流程和 host 通用操作
- `struct mmc_host`
  - 上交给 MMC core 的标准 host 对象

### 关键函数

- `sdhci_pltfm_init()`
- `sdhci_nebula_add_host()`
- `sdhci_add_host()`
- `sdio_add_func()`
- `mmc_of_find_child_device()`
- runtime PM 相关初始化
- SDHCI 通用 `enable_sdio_irq`

### 主流程

DTS 启用 `sdio0` -> platform driver 匹配 `nebula,sdhci` -> 初始化 SDHCI host -> 注册 MMC host -> 上层 MMC/SDIO core 枚举卡 -> 创建 `sdio_func` -> 如果存在对应 child node 则挂到 `func->dev.of_node` -> 进入具体 function driver probe。

## 这一章按什么逻辑展开

这一章按“先用 `06` 给出的标准 function driver 顺序作参照，再看 3516 从设备树到 host、再到 function 的真实落地”的逻辑展开。

这样拆的原因是：

- 如果板级章节一上来就只读 `sdhci_nebula.c`，很容易和前面标准 MMC/SDIO 主线脱节
- `06` 已经把标准 function driver 的最小顺序站稳，这一章更适合按那条顺序倒推：板级 DTS 怎样把 host 准备好，标准枚举链又怎样继续走到 `sdio_func` 和具体 driver
- 当前公开的 HI3516CV610 DTS 已经把 host 级路径交代清楚；如果后续还要让具体 function driver 使用 `of_node`，就继续沿标准 `mmc_of_find_child_device()` 这条线往下补 child node

所以本章后面的结构是：

1. 先固定和当前板子直接相关的 DTS 与源码文件
2. 先把它和 [[06-典型Function驱动例子]] 的最小顺序对起来
3. 再看 `mmc0/sdio0/sdio1` 在板级上怎么取舍
4. 再看 `nebula,sdhci` 如何落到 `sdhci_host` / `mmc_host`
5. 最后把板级接入重新接回标准 SDIO 枚举、`sdio_func` 创建和 function driver probe 主线

## 0. 先说清 `sdhci` 是什么

`sdhci` 是 `Secure Digital Host Controller Interface`，可以先把它理解成：

- 一套标准化的 SD/MMC/SDIO host 控制器寄存器接口
- Linux 内核里围绕这套接口做出来的一层通用 host 框架

它解决的问题是：

- 不同 SoC 的 SD/MMC/SDIO 控制器虽然来自不同厂商
- 但如果寄存器布局和控制方式大体符合 SDHCI 规范
- 就没有必要每个厂商都从零重写一整套 MMC host 主线

于是 Linux 里形成了这条分层关系：

```text
厂商私有 platform driver
-> sdhci_host
-> mmc_host
-> mmc core
-> sdio core
```

这条关系可以这样理解：

1. `厂商私有 platform driver`
   - 负责当前 SoC 的时钟、复位、寄存器偏移、quirk、DTS 对接
2. `sdhci_host`
   - 负责复用 Linux 现成的 SDHCI 通用逻辑
   - 例如很多标准寄存器访问、通用中断处理、通用 host 能力管理
3. `mmc_host`
   - 作为 MMC core 真正接收的标准 host 对象
4. `mmc core / sdio core`
   - 再往上就是前面几章已经讲过的标准枚举、probe、I/O、IRQ 主线

所以在这一章里：

- `sdhci` 不是 card，也不是 function
- `sdhci` 讲的是 host controller 这一层的标准化抽象
- `nebula,sdhci` 的意思是：HI3516CV610 这套控制器以厂商私有 glue driver 的方式，接在标准 `sdhci` 框架下面

这也是为什么本章后面读 `sdhci_nebula.c` 时，要始终分清两层：

- `nebula`
  - 当前 SoC 的私有适配层
- `sdhci`
  - Linux 已经提供好的通用 host 框架层

## 2. DTS 里最关键的事实

### 2.1 SoC 级 DTS 定义了三类节点

在 `hi3516cv610.dtsi` 中可以看到：

- `mmc0: eMMC@0x10030000`
- `sdio0: SDIO@0x10030000`
- `sdio1: SDIO1@0x10040000`

最重要的观察：

- `mmc0` 和 `sdio0` 共享同一套控制器基地址 `0x10030000`
- 这不是两个物理控制器，而是同一套硬件在 DTS 层按不同用途暴露

### 2.2 当前板级选择了 `sdio0`

在 `hi3516cv610-demb.dts` 中：

- `&mmc0 { status = "disabled"; }`
- `&sdio0 { status = "okay"; }`
- `&sdio1 { status = "disabled"; }`

这表示当前板型把 `0x10030000` 这套控制器作为 `sdio0` 用，而不是 eMMC。

## 3. `sdio0` 节点当前配置了什么

本节先看设备树描述了什么，再看内核如何消费这些属性。

在 `hi3516cv610.dtsi` 里，`sdio0` 这几个属性最值得关注：

- `compatible = "nebula,sdhci"`
- `reg = <0x10030000 0x1000>`
- `interrupts = <GIC_SPI 42 IRQ_TYPE_LEVEL_HIGH>`
- `clock-names = "mmc_clk", "mmc_hclk"`
- `resets = <&clock 0x35c0 16>, <&clock 0x35c0 17>, <&clock 0x35c0 18>, <&clock 0x35c4 1>`
- `reset-names = "crg_reset", "crg_tx", "crg_rx", "dll_reset"`
- `max-frequency = <50000000>`
- `crg_regmap = <&clock>`
- `iocfg_regmap = <&ioconfig0>`
- `bus-width = <4>`
- `cap-sd-highspeed`
- `full-pwr-cycle`
- `disable-wp`
- `no-mmc`
- `devid = <1>`

这说明当前板级意图很明确：

- 按 4bit SD/SDIO 总线使用
- 最高先按 50MHz high-speed 路线跑
- 明确不是 eMMC 模式

### 3.1 DTS 属性如何被内核消费

设备树不是“给人看的配置清单”，而是 platform bus、OF 子系统、MMC core 和具体 host/function driver 共同消费的资源描述。`sdio0` 这类 host 节点至少要按下面方式归位：

| DTS 属性 | 描述了什么 | 内核如何消费 |
| --- | --- | --- |
| `compatible = "nebula,sdhci"` | 这套控制器由 HI3516CV610 的 nebula SDHCI host 驱动管理 | OF/platform 匹配后进入 `sdhci_nebula.c` 的 platform driver probe |
| `reg = <0x10030000 0x1000>` | 控制器寄存器物理地址和大小 | platform resource 被 host 驱动映射成寄存器访问基址 |
| `interrupts = <GIC_SPI 42 IRQ_TYPE_LEVEL_HIGH>` | host controller 自己的硬件中断线 | OF IRQ 解析成 Linux IRQ，供 SDHCI host 处理命令完成、数据完成、错误和 SDIO IRQ 上报 |
| `clock-names = "mmc_clk", "mmc_hclk"` | 控制器工作时钟和总线时钟 | clock framework 按名字查找并打开时钟，影响 CMD/CLK 是否能跑起来 |
| `resets` / `reset-names` | 控制器复位线，包括 CRG、TX、RX、DLL 等复位资源 | reset/clock 相关代码在 host 初始化阶段释放或控制复位，错误时通常停在 host probe 或 CMD 无响应 |
| `max-frequency = <50000000>` | host 允许的最高 SDIO 时钟 | MMC core/host 能力限制最终影响初始化后时钟选择 |
| `crg_regmap = <&clock>` | host 驱动访问 CRG/clock 相关寄存器所需的 regmap | `sdhci_nebula.c` 这类厂商 host 驱动可通过 regmap 做控制器私有时钟/复位/模式配置 |
| `iocfg_regmap = <&ioconfig0>` | host 驱动访问 IO 配置寄存器所需的 regmap | 板级管脚、电气属性或复用配置异常时，这类资源会影响 CMD/DAT/CLK 信号是否真正可用 |
| `bus-width = <4>` | 板级布线支持 4bit SDIO 数据线 | MMC core 配置总线宽度时会参考，影响吞吐和兼容性 |
| `cap-sd-highspeed` | 板级允许 SD high-speed 能力 | 参与 host/card 能力协商，影响是否切到 high-speed |
| `full-pwr-cycle` | host 支持完整掉电再上电 | 电源管理和重新初始化场景会参考 |
| `disable-wp` | 不使用写保护检测 | 避免 MMC/SD 写保护路径依赖不存在的板级信号 |
| `no-mmc` | 这一路不按 MMC/eMMC 模式使用 | 帮助 host/core 按 SD/SDIO 方向理解当前控制器用途 |
| `devid = <1>` | 厂商驱动内部区分 sdio0 | 供 `sdhci_nebula.c` 做控制器实例、日志或寄存器差异判断 |

如果后续在 host 节点下面补 function child node，它的消费方式又是另一层：

| child node 属性 | 描述了什么 | 内核如何消费 |
| --- | --- | --- |
| `wifi@1` / `reg = <1>` | 这个子节点对应 SDIO function 1 | `mmc_of_find_child_device()` 根据 function number 把 node 绑定到 `func->dev.of_node` |
| `compatible = "realtek,rtl8189fs"` | function driver 可用于识别具体模组或私有配置 | function driver 可以从 `func->dev.of_node` 读取，通常不替代 `sdio_device_id` 匹配 |
| `interrupt-parent` / `interrupts` | 模组额外拉出的 out-of-band IRQ | function driver 解析成普通 Linux IRQ，再用 `devm_request_threaded_irq()` 注册 |

注意：host 节点的 `interrupts = <GIC_SPI 42 IRQ_TYPE_LEVEL_HIGH>` 是控制器 IRQ，不是 WiFi/BT function 的 out-of-band IRQ；function 级外部中断应该写在对应 child node 下面。

```c
&sdio1 {
	status = "okay";
	non-removable;
	bus-width = <1>;
	max-frequency = <400000>;
	no-1-8-v;
	rtl8189fs: wifi@1 {
	    compatible = "realtek,rtl8189fs";
		reg = <1>; 
		interrupt-parent = <&gpioX>;     //这才是真正的 out-of-band IRQ 
										 //挂在具体 function child node 上
        interrupts = <Y IRQ_TYPE_LEVEL_LOW>;
	    };
};
```

但它当前还没有继续写出某个具体 WiFi/BT 模组的 function child node。  
所以这里要分清两层：

- host 级 DTS
  - 当前这份 3516 DTS 已经完整给出
- function 级 child node
  - 如果后续具体 function driver 还要通过 `func->dev.of_node` 取 `compatible`、`interrupts` 等属性，就要继续在 host 节点下面补对应 child node

这正是 [[06-典型Function驱动例子]] 和本章接起来的地方：  
[[06-典型Function驱动例子]] 先把标准 function driver 顺序站稳，本章再说明 HI3516CV610 板级怎样把 host 节点和后续 child node 消费路径真正接起来。

## 4. `sdio1` 的意义

`sdio1` 在 SoC DTS 里也定义好了：

- 基地址 `0x10040000`
- `devid = <2>`

但当前板级把它关掉了。也就是说：

## 5. `nebula,sdhci` 驱动是怎么接进标准框架的

文件：

- `drivers/vendor/mmc/sdhci_nebula.c`

它做的事不是重写一套 MMC/SDIO core，而是：

1. 通过 `of_match_table` 匹配 `compatible = "nebula,sdhci"`
2. 调 `sdhci_pltfm_init()` 创建标准 `sdhci_host`
3. 填入自己的 `sdhci_ops`
4. 调 `sdhci_nebula_add_host(host)`

放进新学习架构的流程示意里，每一行要看清职责边界：

```text
compatible = "nebula,sdhci"          // DTS 声明这套控制器要绑定 nebula SDHCI host driver
-> platform driver probe             // platform bus 匹配成功后进入 host 驱动 probe
-> sdhci_pltfm_init()                // 创建标准 sdhci_host，并挂上平台私有数据
-> sdhci_nebula_add_host()           // 完成 nebula 私有寄存器、时钟、复位和 host 参数准备
-> sdhci_add_host()                  // 进入 SDHCI core，注册标准 host 能力和中断处理
-> mmc_add_host()                    // 把 mmc_host 交给 MMC core，并触发后续 rescan
```

这意味着：

- 硬件适配是私有的
- 但往上还是标准 `sdhci -> mmc core -> sdio core`

所以只要 host 驱动没把接口搞坏，上层 `sdio.c / sdio_bus.c / sdio_io.c / sdio_irq.c` 的阅读方法和标准 Linux 一样。

进一步把它和 [[06-典型Function驱动例子]] 接起来，可以写成：

```text
HI3516CV610 DTS 选中 sdio0
-> nebula host driver probe
-> sdhci_host / mmc_host 注册完成
-> MMC/SDIO core 开始标准枚举
-> sdio_add_func() 创建并注册 sdio_func
-> 如果有匹配的 DT child node，则通过 mmc_of_find_child_device() 挂到 func->dev.of_node
-> 进入具体 function driver probe
```

也就是说，本章真正补的是：

- [[06-典型Function驱动例子]] 里 `probe()` 之前，板级和 host 这一长段到底是谁先把路铺好的

## 6. `sdhci_nebula.c` 里值得特别关注的点

### 6.1 它是标准 SDHCI 平台化写法

代码里可以看到：

- `struct sdhci_ops g_sdhci_nebula_ops`
- `struct sdhci_pltfm_data g_sdhci_nebula_pdata`
- `sdhci_pltfm_init()`

这说明它不是完全自创框架，而是在复用 Linux 的 SDHCI 抽象。

### 6.2 它支持 runtime PM

在 `probe()` 里有一段明显的 runtime PM 逻辑：

- `pm_runtime_get_noresume()`
- `pm_runtime_enable()`
- `pm_runtime_put_autosuspend()`

这里的 `runtime PM` 不是整机休眠，而是设备在系统正常运行过程中，按“空闲/忙碌”状态动态进入低功耗和恢复。

放回本章语境里看，它管的不是 `sdio_func`，而是这一层 host 控制器：

```text
nebula platform driver
-> sdhci_host
-> mmc_host
```

也就是说，这里说“它支持 runtime PM”，本质上是在说：

- SDIO host 控制器空闲时可以自动省电
- 后面再次有 MMC/SDIO 访问时，再自动恢复

这几个调用连起来的含义大致是：

- `pm_runtime_get_noresume()`
  - 先把设备使用计数加上，避免 `probe()` 过程中它立刻进入自动休眠
- `pm_runtime_enable()`
  - 打开这套 host 控制器的 runtime PM 支持
- `pm_runtime_put_autosuspend()`
  - 初始化完成后把使用计数放回去，并允许设备后续在空闲一段时间后自动 suspend

所以这段初始化的整体意思可以压成：

```text
probe 初期先别睡
-> 打开 runtime PM
-> 初始化完成后允许将来空闲自动休眠
```

如果后续遇到：

- 卡偶发休眠后不工作
- SDIO 功能唤醒异常
- resume 后 `CMD52/CMD53` 超时
- IRQ 来了但 host 没完全恢复

runtime PM 是必须排查的一层。

### 6.3 `devid` 被保留下来了

`dtsi_usage.txt` 明确写了：

- `devid = <0>` 表示 emmc
- `devid = <1>` 表示 sdio0
- `devid = <2>` 表示 sdio1

这对后续日志、调试输出、驱动内部槽位映射都有帮助。

## 7. 对理解整套架构最关键的一句话

HI3516CV610 这里的私有部分，主要在“控制器适配”和“DTS 选型”；而卡枚举、function 创建设备、总线匹配、I/O API、中断分发，仍然大体沿用 Linux 5.10 的标准 SDIO 主线。

## 8. 这对调试意味着什么

问题可以先粗分成两类：

### 8.1 板级/host 问题

例如：

- DTS 选错节点
- 时钟/复位/iocfg 有问题
- `nebula sdhci` 初始化失败
- 电源管理异常

### 8.2 core/function 问题

例如：

- 卡能识别，但 function 不出
- `probe()` 没进
- `sdio_enable_func()` 失败
- 中断收不到
- 数据读写异常

这个分层一旦建立起来，定位速度会快很多。

## 9. 全流程总结：从 DTS 到 SDIO function probe

这一章跨过了设备树、platform driver、SDHCI host、MMC core、SDIO core、function child node 和具体 function driver。完整闭环可以这样复盘：

```text
&sdio0 { status = "okay"; }          // 板级 DTS 选中 sdio0 这套控制器
-> compatible = "nebula,sdhci"       // OF/platform bus 用 compatible 匹配 host driver
-> platform driver probe             // `sdhci_nebula.c` 开始消费 reg/irq/clock/reset 等资源
-> sdhci_pltfm_init()                // 创建 sdhci_host，接入 Linux SDHCI 平台化框架
-> sdhci_nebula_add_host()           // 做 HI3516CV610 私有 host 初始化
-> sdhci_add_host()                  // SDHCI core 建立标准 host 能力和中断处理
-> mmc_add_host()                    // MMC core 接管 mmc_host 并安排 rescan
-> mmc_rescan()                      // 扫描 host 上是否存在 SDIO/SD/MMC 卡
-> mmc_attach_sdio()                 // SDIO attach 路径用 CMD5/OCR 识别 SDIO 卡
-> mmc_sdio_init_card()              // 初始化整卡，读取 CCCR/CIS，确认 function 数量
-> sdio_init_func()                  // 为每个 function 创建 struct sdio_func
-> mmc_of_find_child_device()        // 如果 DTS 有 function child node，就挂到 func->dev.of_node
-> sdio_add_func()                   // 把 function 注册成 sdio bus device
-> sdio_bus_match()                  // 用 func 身份字段匹配 driver 的 sdio_device_id
-> sdio_bus_probe()                  // bus 层做公共准备后调用具体 driver probe
-> sdio_driver.probe(func, id)       // function driver 开始芯片私有初始化、I/O 和 IRQ 配置
```

如果存在外部 out-of-band IRQ，还要额外接上这一条资源消费线：

```text
wifi@1 { interrupts = <...>; }       // function child node 描述模组额外 IRQ 线
-> func->dev.of_node                 // SDIO core 把 child node 绑定到对应 sdio_func
-> of_irq_get() / platform_get_irq() // function driver 把 DT 中断属性解析成 Linux IRQ
-> devm_request_threaded_irq()       // function driver 注册普通平台 IRQ handler
-> handler 内访问 SDIO 前 claim host  // 只要要发 CMD52/CMD53，就回到 SDIO I/O 规则
```

所以 HI3516CV610 的板级落地不是只看一行 `compatible`，而是要能把每个 DTS 属性说清楚：它描述的是哪层资源、被哪个内核对象消费、最终如何影响 `mmc_host`、`sdio_func`、I/O 或 IRQ。
