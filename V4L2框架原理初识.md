```C
权限与互斥 (`EBUSY`)：图像传感器硬件通常是独占资源。虽然 Linux 允许多个进程打开同一个普通文件，但对于底层的 ISP (图像信号处理器) 或 sensor，并发控制极其复杂。因此，V4L2 驱动通常在 `open` 回调中实现严格的排他性锁机制。一旦被占用，后续的 `open` 尝试直接返回 `EBUSY`，拒绝服务。这是硬件物理特性的限制在软件层面的直接映射。
```

```c
-VFS 映射机制底层的“指针劫持”逻辑-
动作一：用户态的错觉与 VFS 的拦截
当应用层代码执行 open("/dev/video0", O_RDWR) 时，用户态进程认为自己打开了一个普通文件。
实质： 系统调用触发后，执行流陷入内核态，交由 VFS 处理。VFS 不关心这是硬盘文件还是摄像头，它只负责沿着目录树（/ -> dev -> video0）去查找对应的物理节点，这个节点在内核中体现为 struct inode（索引节点）。
动作二：Inode 揭露真实身份 (设备号)
对于 /dev/video0，它所在的文件系统通常是存在于内存中的 devtmpfs。
实质： VFS 找到它的 inode 后，检查其属性标志。发现这不是一个常规文件（Regular File），而是一个字符设备文件（Character Device）。
此时，inode 内部并没有指向磁盘扇区的数据块指针，它的核心数据是一个无符号整数：i_rdev（设备号）。对于 V4L2 视频设备，主设备号（Major）硬编码为 81。次设备号（Minor）如果是 video0 通常是 0。
动作三：查表（cdev 注册树）
既然是字符设备，VFS 知道不能用读写磁盘的方式来操作它。内核需要知道主设备号 81 到底对应哪段驱动代码。
实质： Linux 内核维护了一个全局的字符设备注册表（类似一个大型哈希表或数组 cdev_map）。当 V4L2 核心层（或具体驱动）在系统启动时，它已经提前将主设备号 81 和一个名为 v4l2_fops（包含具体读、写、控制函数的结构体指针）打包成一个 struct cdev 对象，注册到了这个表中。
VFS 拿着提取出的设备号 (81, 0) 去查表，成功匹配到了之前注册的 struct cdev。
动作四：核心动作 —— 狸猫换太子（指针替换）
这是整个“映射”机制最关键的一步，通常发生在内核的 chrdev_open() 函数中。
VFS 会为当前进程的这次打开操作在内存中分配一个动态结构体 struct file。
struct file 中有一个至关重要的成员 f_op（这是一个指向 struct file_operations 的指针）。在默认情况下，它指向一些通用的、会报错的空函数。
劫持发生： VFS 从刚才查到的 struct cdev 中，提取出 V4L2 驱动提前备好的 v4l2_fops 指针，直接覆盖掉当前 struct file 中的 f_op 指针。
最终结果：控制权移交
一旦上述替换完成，open 系统调用返回给用户态一个文件描述符（fd）。
从这一刻起，当用户层对这个 fd 调用 ioctl(fd, VIDIOC_QUERYCAP, &cap) 时，内核不再经过任何文件系统的中间层逻辑。
它会直接解引用 struct file 中的 f_op 指针，精准命中并跳转执行 v4l2_fops->unlocked_ioctl()。
```

# V4L2 驱动框架深度整理

## 1. V4L2 是什么

V4L2（Video4Linux2）是 Linux 媒体子系统里最核心的一组框架之一，负责把视频采集、输出、编解码、图像处理、射频/电视等设备，以统一的内核对象模型和用户态 ioctl 接口暴露出来。

它不是“单一驱动模板”，而是一组可组合的基础设施：

- **设备节点框架**：把能力暴露成 `/dev/videoX`、`/dev/v4l-subdevX`、`/dev/radioX` 等节点。
- **子设备框架（subdev）**：描述 sensor、CSI bridge、ISP、decoder、mux、scaler 这类管线内部模块。
- **缓冲队列框架（vb2）**：统一 buffer 的申请、排队、DMA、完成回传。
- **控制框架（controls）**：统一亮度、曝光、增益、白平衡等控制项。
- **异步注册框架（async）**：处理“主设备先起来，sensor 晚点出现”的现实硬件拓扑。
- **Media Controller**：用 graph 描述复杂媒体拓扑，适合多实体、多 pad、多链路系统。
- **fwnode/DT 解析**：把 Device Tree / ACPI 里的 endpoint 拓扑映射为 media graph。

------

## 2. 先建立整体心智模型

一个典型相机链路：

```
sensor -> MIPI CSI-2 RX -> ISP -> video node
```

在 V4L2/Media 框架中，通常会拆成两层：

### 2.1 管线内部模块

由 **`struct v4l2_subdev`** 表示：

- sensor
- CSI bridge / receiver
- ISP 子模块
- scaler / mux / demosaic 等

这些模块多数不直接给用户打开 `/dev/videoX`，而是作为**子设备**存在。

### 2.2 最终暴露给用户空间的节点

由 **`struct video_device`** 表示：

- capture node
- output node
- mem2mem 节点
- metadata 节点

用户最常见交互对象是 `/dev/videoX`。它背后往往并不是“一个硬件块”，而是整条 pipeline 的一个出口或入口。

### 2.3 顶层聚合对象

由 **`struct v4l2_device`** 表示：

它像一个“设备实例级容器”，把：

- 主设备
- 各 subdev
- 控制处理器
- 通知机制

### 2.4数据缓冲

由 **`vb2_queue`** 表示：



聚合到一起。

一句话概括：

- **`v4l2_device`**：一台 V4L2 设备实例的总管
- **`v4l2_subdev`**：媒体管线内部模块
- **`video_device`**：真正创建设备节点的对象
- **`vb2_queue`**：数据缓冲流转核心
- **`media_device` / entity / pad / link`**：复杂拓扑图模型

------

## 3. V4L2 驱动分层

### 3.1 用户态接口层

#### 用户态通过：

- `open()` / `close()`
- `ioctl()`
- `mmap()`
- `poll()`
- `read()` / `write()`（少数场景）

访问设备。

#### 主 ioctl 包括：

- 能力查询：`VIDIOC_QUERYCAP`
- 枚举格式：`VIDIOC_ENUM_FMT`
- 设定格式：`VIDIOC_S_FMT` / `VIDIOC_G_FMT`
- 申请缓冲：`VIDIOC_REQBUFS`
- 查询缓冲：`VIDIOC_QUERYBUF`
- 入队/出队：`VIDIOC_QBUF` / `VIDIOC_DQBUF`
- 开始/停止流：`VIDIOC_STREAMON` / `VIDIOC_STREAMOFF`
- 控制项：`VIDIOC_S_CTRL` / `VIDIOC_G_CTRL` / `EXT_CTRLS`

这些 ioctl 最终会落到驱动填写的 **`struct v4l2_ioctl_ops`**。

------

### 3.2 V4L2 core 层

V4L2 core 负责共性工作：

- file handle 生命周期
- ioctl 分发
- 控制框架共性处理
- buffer core 逻辑（与 vb2 配合）
- 事件订阅
- subdev 基础管理
- 与 media controller 协同

驱动只实现“真正硬件相关”的最小集合。

这正是 V4L2 框架的设计精髓：**把规则、状态机、边界检查尽量收敛到框架层，把寄存器操作留给驱动。**

------

## 4. 关键内核对象

### 4.1 `struct video_device`

它代表一个最终出现在 `/dev` 下的设备节点。

典型字段关注点：

- `fops`：文件操作，通常使用 V4L2 提供的标准 helper
- `ioctl_ops`：各类 V4L2 ioctl 回调
- `v4l2_dev`：关联到顶层 `v4l2_device`
- `queue`：可选，关联 vb2 queue
- `ctrl_handler`：控制处理器
- `device_caps`：该节点能力
- `lock`：串行化 ioctl / 状态切换

常见注册流程：

1. 初始化 `video_device`
2. 填充 `fops/ioctl_ops/release/v4l2_dev/device_caps`
3. 关联 `vb2_queue`
4. `video_register_device()`

它本质上回答一个问题：**“这个 `/dev/videoX` 节点是谁，它支持什么操作？”**

------

### 4.2 `struct v4l2_device`

它是一个设备实例级对象，常嵌入主驱动私有结构体中。

典型职责：

- 作为 subdev 的宿主
- 挂接控制、通知、日志上下文
- 持有 parent `struct device`
- 协助管理设备级资源

probe 早期通常会：

```c
v4l2_device_register(dev, &foo->v4l2_dev);
```

之后各 subdev 会挂到它下面。

------

### 4.3 `struct v4l2_subdev`

它表示媒体管线中的内部模块。

典型例子：

- 摄像头 sensor
- HDMI decoder
- CSI 接收器
- ISP block
- scaler / mux

subdev 的特点：

- 不一定直接对应 `/dev/videoX`
- 更强调 pad、format、route、link
- 通过 `subdev_ops` 划分功能域

常见 ops 分类：

- `core_ops`
- `video_ops`
- `pad_ops`
- `sensor_ops`
- `audio_ops`
- `tuner_ops`

现代复杂图像链路里，**subdev + media controller** 才是主体，`video_device` 更像最终出口。

------

### 4.4 `struct v4l2_fh`

它表示一次打开设备后的 file handle 上下文。

用途：

- 事件订阅
- 优先级管理
- 文件级状态
- 与 V4L2 core 的 filehandle 框架对接

如果驱动支持事件，通常需要在 `open()` 中初始化 `v4l2_fh`。

------

### 4.5 `struct media_device` / `media_entity` / `media_pad` / `media_link`

这是 **Media Controller** 图模型的核心。

### `media_device`

整张媒体图。

### `media_entity`

图中的实体，比如 sensor、CSI、ISP、video node。

### `media_pad`

实体的输入/输出端口。

### `media_link`

pad 与 pad 之间的连接关系。

适合处理：

- 多 sensor
- 多输入源
- 动态路由
- 多输出
- 带 ISP / scaler / mux 的 SoC 相机系统

------

## 5. 文件与 ioctl 路径

一个 `video_device` 节点打开后，常见执行路径可以理解成：

```
sys_open -> v4l2_fops.open -> 驱动 open
sys_ioctl -> video_ioctl2() -> v4l2_ioctl_ops -> 驱动具体回调
```

这里常见模式是：

- `file_operations` 尽量复用 V4L2 提供的 helper
- 真正的业务逻辑塞进 `v4l2_ioctl_ops`

比如：

- `vidioc_querycap`
- `vidioc_enum_fmt_vid_cap`
- `vidioc_g_fmt_vid_cap`
- `vidioc_s_fmt_vid_cap`
- `vidioc_reqbufs`
- `vidioc_qbuf`
- `vidioc_streamon`

这样框架统一完成参数检查、命令分发和一部分通用状态机处理。

------

## 6. vb2：V4L2 数据通路核心

如果说 `video_device` 是“控制面入口”，那 **videobuf2（vb2）** 就是“数据面核心”。

几乎所有现代 V4L2 流式驱动都基于 vb2。

### 6.1 vb2 解决什么问题

它统一处理：

- buffer 生命周期
- `MMAP / USERPTR / DMABUF` 三种内存模型
- queue / dequeue
- plane 管理
- cache / sync 配合
- 驱动启动/停止流的时序

驱动不再手写一套 buffer 队列状态机。

------

### 6.2 核心对象：`struct vb2_queue`

这是每个流队列的中心对象。

关键内容包括：

- `type`：capture / output / mplane 等队列类型
- `io_modes`：支持 MMAP/USERPTR/DMABUF 哪些模式
- `mem_ops`：底层内存分配/映射实现
- `ops`：驱动自定义的 `vb2_ops`
- `buf_struct_size`：驱动扩展 buffer 结构大小
- `timestamp_flags`
- `lock`
- `drv_priv`

驱动通常在 probe 时初始化它，并把它挂到 `video_device->queue`。

------

### 6.3 `vb2_ops`：驱动真正要实现的流控回调

最重要的一组回调：

- `queue_setup`
- `buf_prepare`
- `buf_queue`
- `start_streaming`
- `stop_streaming`

理解这 5 个回调，基本就抓住了 V4L2 capture 驱动的命门。

#### `queue_setup`

决定：

- buffer 个数约束
- plane 个数
- 每个 plane 的大小
- 分配设备上下文

#### `buf_prepare`

在 buffer 入队前做最终检查：

- size 是否够
- payload 是否合理
- DMA 相关准备

#### `buf_queue`

把一个已经准备好的 buffer 交给驱动自己的活动队列，等待硬件消费。

#### `start_streaming`

当队列满足启动条件后调用：

- 配置 DMA
- 配置 pipeline
- 真正开中断/开流

#### `stop_streaming`

停止硬件，回收尚未完成的 buffer，并保证所有交给驱动的 buffer 最终都回到 vb2。

> 这是非常关键的正确性要求：**凡是驱动从 vb2 收走的 buffer，最终都必须通过 `vb2_buffer_done()` 还回去。**

------

### 6.4 一次采集的典型 vb2 流程

用户态：

1. `REQBUFS`
2. `QUERYBUF`
3. `mmap()`
4. 多次 `QBUF`
5. `STREAMON`
6. 循环 `DQBUF/QBUF`
7. `STREAMOFF`

驱动/框架侧大致对应：

1. vb2 创建队列和 buffer
2. `QBUF` 时触发 `buf_prepare`
3. 如果已 streaming，则触发 `buf_queue`
4. `STREAMON` 时触发 `start_streaming`
5. 硬件中断到来，驱动填充 buffer 完成
6. 驱动调用 `vb2_buffer_done(vb, VB2_BUF_STATE_DONE)`
7. `DQBUF` 把完成 buffer 返还用户态
8. `STREAMOFF` 触发 `stop_streaming`

------

### 6.5 三种内存模式

#### MMAP

最常见。buffer 由内核分配，用户态映射。

优点：

- 简单稳定
- 框架支持最好

#### USERPTR

buffer 由用户提供指针。

优点：

- 某些旧场景灵活

缺点：

- 较复杂
- 新驱动里使用不如 MMAP / DMABUF 普遍

#### DMABUF

buffer 通过 dma-buf 在子系统间共享。

适合：

- camera -> ISP -> codec -> display 零拷贝链路
- V4L2 与 DRM / GPU / NPU 协同

现代 SoC 多媒体系统里非常重要。

------

## 7. Controls 框架

V4L2 control 看似只是“亮度/对比度/曝光”的 ioctl，实际上其规则非常繁琐，所以框架单独提供了 control 子系统。

### 7.1 核心对象

- `struct v4l2_ctrl`
- `struct v4l2_ctrl_handler`
- `struct v4l2_ctrl_ops`

### `v4l2_ctrl`

描述一个控制项本身。

### `v4l2_ctrl_handler`

管理一组 controls。

### `v4l2_ctrl_ops`

驱动实现 `s_ctrl` / `g_volatile_ctrl` / `try_ctrl` 等。

------

### 7.2 为什么必须用 control framework

因为它统一处理：

- control ID 管理
- 类型/范围校验
- cluster
- auto/manual 依赖
- volatile control
- 菜单类 control
- 事件通知

驱动只关注：

- 控件如何创建
- 写寄存器时怎么落地

------

### 7.3 控制框架的典型接入步骤

1. `v4l2_ctrl_handler_init()`
2. 添加标准 controls / 自定义 controls
3. 挂到 `video_device` 或 `v4l2_subdev`
4. 如有必要执行 `v4l2_ctrl_handler_setup()` 做默认值同步

sensor 驱动通常 heavily 依赖 controls，例如：

- exposure
- analogue gain
- digital gain
- hblank / vblank
- test pattern
- link frequency
- pixel rate

------

## 8. subdev 框架：复杂摄像头系统的主体

### 8.1 为什么 subdev 很关键

早期简单 USB 摄像头，一个 `/dev/videoX` 就足够。

但在 SoC 摄像头系统里，一条链路往往横跨多个硬件块：

- sensor
- CSI receiver
- DPHY
- ISP
- scaler
- resizer
- stats block
- capture node

这些模块不能都粗暴塞进一个 video 节点里，于是需要 **subdev** 来描述内部结构。

------

### 8.2 pad-level 配置

subdev 的重要特征是 **pad 配置**：

- sink pad / source pad
- media bus format
- frame size
- crop / compose / selection
- route / stream

也就是说，subdev 管的不是“最终帧缓冲”，而是**链路中间格式和路由**。

------

### 8.3 active state / try state

现代 subdev 框架支持中心化状态管理：

- **try state**：每个 file handle 的试配状态
- **active state**：当前生效配置

这让驱动更容易正确处理：

- 尝试格式
- 多 pad 状态一致性
- 路由切换
- stream-aware 配置

这也是新式 subdev 驱动推荐的写法。

------

### 8.4 streams / multiplexed pads / internal routing

在更复杂硬件中，一个 pad 上可能不只是一条流。

此时 subdev 需要管理：

- stream id
- internal routing
- 一个输入 pad 到多个输出 pad 的映射
- 多流并存

这对于现代 ISP、桥接芯片、多路 CSI 汇聚器非常重要。

如果你只用“单输入单输出、单格式”去理解 V4L2，就会在这些驱动上迅速失真。

------

## 9. Media Controller：把媒体系统建模成图

### 9.1 为什么需要 MC

对复杂平台，单个 `/dev/videoX` 无法完整表达真实硬件关系。

例如：

- 哪个 sensor 连接到哪个 CSI？
- 哪个 ISP 输出连到哪个 video node？
- 哪个 scaler 当前启用？
- 哪条 link 可以动态 enable/disable？

Media Controller 用 **entity-pad-link graph** 解决这个问题。

------

### 9.2 图模型直观理解

- **entity**：模块
- **pad**：端口
- **link**：连接
- **pipeline**：当前激活路径

用户态常用 `media-ctl` 查看拓扑、设置 link、配置 pad format。

------

### 9.3 驱动侧通常要做什么

- 初始化 `media_device`
- 给每个 subdev / video node 初始化 `media_entity`
- 定义 pads
- 创建设备间 links
- 在复杂拓扑里维护 pipeline 启停

很多现代摄像头驱动其实是 **V4L2 + subdev + MC** 的组合，不是单独的“V4L2 video node 驱动”。

------

## 10. async 框架：解决探测顺序问题

现实里，sensor、bridge、host controller 的 probe 顺序经常不可控。

例如：

- host CSI 先 probe
- sensor 还没在 I2C 总线上注册
- 但最终必须把它们绑定成一条链路

这时需要 **V4L2 async notifier**。

### 10.1 它解决的核心问题

不是“延迟初始化”这么简单，而是：

- 描述预期要绑定哪些远端 subdev
- 当远端出现时完成匹配
- 全部依赖到齐后再完成拓扑绑定和后续注册

------

### 10.2 常见匹配方式

- I2C match
- fwnode match

现代 DT/ACPI 平台更多使用 **fwnode match**。

------

### 10.3 典型使用场景

host 驱动：

1. 解析 firmware endpoint
2. 为每个远端子设备创建 async connection
3. 初始化 notifier
4. 注册 notifier
5. 在 `bound()` 回调里拿到 sensor subdev
6. 建立 media links
7. 在 `complete()` 里完成整体注册

所以 async 更像一个**跨设备装配框架**。

------

## 11. fwnode / Device Tree 在 V4L2 中的作用

现代平台大量依赖 firmware 描述拓扑。

V4L2 fwnode API 负责解析：

- endpoint
- bus type
- lane 配置
- link frequency
- rotation / orientation 等属性

典型用途：

- 从 DT 读出 sensor 与 CSI 连接关系
- 解析 CSI lane 数量和极性
- 把 firmware endpoint 转成 async match / media link

在嵌入式相机驱动里，这一层几乎是必修课。

------

## 12. 一个 capture 驱动的典型结构

以“主控采集节点 + 若干 subdev”模式为例，probe 大致会这样展开：

### 第一步：分配主设备私有结构

通常你的顶层结构会内嵌：

- `struct v4l2_device`
- `struct video_device`
- `struct vb2_queue`
- `struct media_device`（若启用 MC）
- 锁、队列、时钟、电源、寄存器映射

### 第二步：注册 `v4l2_device`

把这个设备实例接到 V4L2 核心。

### 第三步：初始化 media device（可选但现代平台很常见）

填写 model / bus info / hw revision 等。

### 第四步：初始化 vb2 queue

指定：

- buffer 类型
- 支持的内存模式
- `vb2_ops`
- `mem_ops`
- buffer 私有结构大小

### 第五步：初始化 video_device

填写：

- `name`
- `v4l2_dev`
- `fops`
- `ioctl_ops`
- `queue`
- `release`
- `device_caps`

### 第六步：注册 subdev / async notifier

- 解析 DT endpoint
- 等待 sensor/bridge 绑定
- 绑定完成后建 links

### 第七步：`video_register_device()`

生成 `/dev/videoX`

### 第八步：运行态

- open/close
- format negotiation
- request buffers
- start/stop streaming
- interrupt 完成 buffer 回收

------

## 13. 一次典型视频采集的完整调用链

下面用“用户打开摄像头并采帧”的视角，把框架串起来。

### 13.1 打开设备

用户：

```text
open(/dev/video0)
```

框架/驱动：

- `v4l2_fops.open`
- 初始化 `v4l2_fh`
- 上电 / runtime PM get（可选）
- 标记 owner / 检查 busy 状态

------

### 13.2 查询并设置格式

用户：

- `VIDIOC_ENUM_FMT`
- `VIDIOC_G_FMT`
- `VIDIOC_S_FMT`

驱动：

- 检查是否允许改格式（通常要求未 streaming）
- 对齐 width/height
- 选定 pixel format
- 计算 bytesperline / sizeimage
- 把结果同步给 pipeline 中的 subdev format

复杂平台中，`S_FMT` 不只是改一个寄存器，而是**沿着 pipeline 做格式协商**。

------

### 13.3 申请缓冲

用户：

- `VIDIOC_REQBUFS`
- `VIDIOC_QUERYBUF`
- `mmap()`

框架：

- 由 vb2 创建 buffers
- 建立内核/用户映射

------

### 13.4 缓冲入队

用户：

- 多次 `VIDIOC_QBUF`

框架/驱动：

- `buf_prepare`
- `buf_queue`
- 驱动把 buffer 放入活跃队列

------

### 13.5 开流

用户：

- `VIDIOC_STREAMON`

框架/驱动：

- vb2 检查队列状态
- `start_streaming`
- 启动 pipeline：
  - sensor start
  - CSI enable
  - DMA enable
  - interrupt enable

现代 pipeline 往往需要按顺序启动各 subdev。

------

### 13.6 帧完成

硬件：

- DMA 完成 / 中断到来

驱动：

- 找到当前完成的 buffer
- 填写 sequence / timestamp / payload
- `vb2_buffer_done(..., DONE)`

框架：

- buffer 进入 done 队列
- `DQBUF` 时返给用户

------

### 13.7 停流

用户：

- `VIDIOC_STREAMOFF`

驱动：

- 停 sensor / CSI / DMA
- 处理尚未完成 buffer
- 把所有被驱动持有的 buffer 返还 vb2

这是驱动稳定性的重灾区，很多 bug 都发生在 stop_streaming 和错误恢复路径。

------

## 14. 与 mem2mem 驱动的关系

V4L2 不只做 capture/output，也支持 **mem2mem** 驱动，例如：

- 视频编解码器
- 图像后处理
- 格式转换
- 缩放器

这类驱动往往有：

- output queue：输入原始数据
- capture queue：输出处理结果

它们不一定需要复杂 media graph，但仍 heavily 使用：

- `video_device`
- `v4l2_ioctl_ops`
- `vb2`
- controls
- m2m helper

所以学 V4L2 时，capture 驱动和 mem2mem 驱动要区分开看。

------

## 15. 常见驱动形态分类

### 15.1 简单 USB UVC 摄像头

特点：

- 一个主 video node 为主
- subdev / MC 参与较少
- 重点在 USB 数据搬运和格式支持

### 15.2 SoC camera host 驱动

特点：

- subdev + async + MC 很关键
- 需要处理 DT endpoint / pipeline / PM
- 常与 sensor 驱动协同

### 15.3 纯 sensor 驱动

特点：

- 通常实现为 `v4l2_subdev`
- 强依赖 controls
- 强依赖 pad format / frame interval / power management

### 15.4 编解码 / 图像处理 mem2mem 驱动

特点：

- 双 vb2 队列
- m2m 调度
- 偏重状态机和 buffer 流转

## 16. V4L2 新手最容易混淆的几个点

### 16.1 `video_device` 不等于一个硬件 IP

它只是一个用户可见节点。复杂平台上，它可能只是整条 pipeline 的出口。

### 16.2 `subdev` 不一定对用户直接可见

subdev 的核心是内部格式、路由、链路协商，不是最终采帧接口。

### 16.3 真正的数据面核心通常是 vb2

很多 bug 表面是 ioctl 问题，实际是 queue/buffer 生命周期问题。

### 16.4 简单相机能不显式使用 MC，但复杂平台几乎离不开

尤其是多实体、可路由、多路输入输出平台。

### 16.5 `S_FMT` 往往不是单点动作

复杂驱动里，设置格式意味着沿媒体图做传播、裁剪、约束收敛。

### 16.6 stop/error path 往往比 start path 更难写对

特别是：

- 中断并发
- runtime PM
- 流关停时 buffer 回收
- async 解绑

------

## 17. 一个推荐的学习顺序

如果你要真正读懂 V4L2 驱动，建议顺序如下：

### 第 1 步：先只看简单 `video_device + vb2`

目标：搞清 `/dev/videoX` 如何工作。

重点：

- `v4l2_ioctl_ops`
- `vb2_queue`
- `QBUF/DQBUF/STREAMON/OFF`

### 第 2 步：再看 controls

目标：理解 control handler 如何减少样板代码。

### 第 3 步：再看 `v4l2_subdev`

目标：理解 sensor / bridge 如何抽象。

### 第 4 步：再看 Media Controller

目标：把复杂相机链路看成 graph。

### 第 5 步：最后看 async + fwnode

目标：理解系统如何从 DT 拓扑自动装配成一条 pipeline。

这个顺序比一上来就啃复杂 SoC camera driver 更稳。

## 18. 你写一个 V4L2 驱动时，脑子里要始终问的 10 个问题

1. 我暴露的是 `video_device` 还是 `subdev`，还是两者都有？
2. 我的数据路径是 capture、output，还是 mem2mem？
3. buffer 生命周期由谁持有、何时归还？
4. `start_streaming` 和 `stop_streaming` 的对称性是否完整？
5. 格式协商只影响本节点，还是要传播到 subdev pipeline？
6. 控制项属于主节点还是 sensor subdev？
7. 我是否需要 media graph 来表达真实拓扑？
8. 子设备 probe 顺序不可控时，是否应使用 async notifier？
9. DT/fwnode 里的 endpoint 是否已正确解析并映射到 link？
10. 错误路径下是否保证所有 buffer、时钟、电源、锁都能正确回收？

## 20. 最后的总结

V4L2 驱动框架不是单一 API，而是一套分层协作体系：

- **`video_device`** 负责用户可见节点
- **`v4l2_device`** 负责设备实例聚合
- **`v4l2_subdev`** 负责内部模块抽象
- **`vb2`** 负责 buffer 与流式状态机
- **controls** 负责控制项规则化管理
- **Media Controller** 负责复杂拓扑建模
- **async + fwnode** 负责跨设备按拓扑自动装配

真正理解它的关键，不是背结构体名字，而是建立下面这条主线：

> **“控制面通过 video/ioctl 暴露，数据面通过 vb2 流动，拓扑面通过 subdev + media graph 建模，系统装配通过 async + fwnode 完成。”**

只要这条主线清楚，读大多数 V4L2 驱动都会有抓手。

------

## 21. 继续深挖时最值得读的方向

1. `v4l2_ioctl_ops` 与 `vb2_ops` 的精确调用时机
2. sensor subdev 的 pad format / frame interval / controls
3. media graph 的 entity / pad / link 生命周期
4. async notifier 的 bound/complete/unbind 过程
5. stream-aware subdev 与 routing API
6. mem2mem helper 框架
7. request API（更高级的原子化配置模型）

## 22. 给你一个实战建议

如果你的目标是“会改 V4L2 驱动”，最有效的方式不是只看文档，而是选一个真实驱动，按下面顺序剖：

1. 看顶层私有结构体，识别 `v4l2_device/video_device/vb2_queue/subdev/media_device`
2. 看 `probe()`，画出注册顺序
3. 看 `v4l2_ioctl_ops`
4. 看 `vb2_ops`
5. 看中断处理和 `vb2_buffer_done()`
6. 看 `stop_streaming()` 错误恢复
7. 看 subdev / async / media link 装配过程

这样你会比纯看 API 更快进入真正的驱动思维。
