### wpa_supplicant -Dwext -iwlan0 -c/etc/wpa.conf &命令

这是一个启动无线网络认证守护进程的系统级调用命令。在嵌入式 Linux 网络协议栈中，它负责在数据链路层（Layer 2）完成 802.11 的加密认证与密钥协商（如 WPA2 的四次握手）。

具体参数的工程含义拆解如下：

### 1. `wpa_supplicant` （核心主程序）

这是一个运行在用户态的守护进程（Daemon），作为 IEEE 802.1X/WPA 的客户端组件。它的核心职责是维护 Wi-Fi 连接的状态机。没有它，网卡即使物理上接收到了射频信号，也无法进行数据解密和合法性验证。

### 2. `-D wext` （指定驱动后端接口）

`-D` 代表 Driver backend。`wext` 是 **Wireless Extensions** 的缩写，定义了用户空间程序与内核驱动之间交互的 API。

- **架构层面的客观事实**：`wext` 是一个已经被 Linux 主线内核标记为过时的老旧接口，现代 Linux 的标准网络栈采用的是基于 Netlink 的 `nl80211` 架构。但由于你当前板子上使用的是 Realtek 原厂提供的私有驱动（`8188eu.ko`），这类外挂驱动往往没有完全适配 mac80211 框架，因此必须强制指定使用老旧的 `wext` 接口来下发底层指令。这正是该命令运行时会产生 `ioctl[SIOCSIWAP]: Operation not permitted` 等兼容性错误打印的根本原因。

### 3. `-i wlan0` （指定网络接口）

`-i` 代表 Interface。明确指示 `wpa_supplicant` 接管系统中名为 `wlan0` 的网络设备节点。

### 4. `-c /etc/wpa.conf` （指定配置文件路径）

`-c` 代表 Configuration。向守护进程传递配置文件的绝对路径，程序启动时会解析该文件内部的 `ssid`、`key_mgmt` 和 `psk` 字段，作为寻找和比对目标物理热点的唯一依据。

### 5. `&` （Shell 异步执行控制符）

这是基础的 Linux Shell 语法。它指示系统将该进程挂起至**后台（Background）**异步运行。因为网络认证是一个需要长期维持和监听的常驻任务，如果不加 `&`，进程会一直霸占（阻塞）当前终端控制台，导致你无法继续输入后续的网络层命令（如 `udhcpc`）。









## NFS文件挂载

```C
~ # mount -t nfs -o nolock,tcp,v3 192.168.1.10:/home/ipc/Hi3516/nfs_share /mnt

mount: mounting 192.168.1.10:/home/ipc/Hi3516/nfs_share on /mnt failed: Permission denied
    

mount: mounting 192.168.1.10:/home/ipc/Hi3516/nfs_share on /mnt failed: Resource busy
```

- 配置文件的输出内容直接暴露了问题的根本原因：**目标目录根本没有被 NFS 服务端导出**。
- 目标文件已经存在挂载文件了

你之前在板端执行的挂载命令是：

```
mount ... 192.168.1.10:/home/ipc/Hi3516/nfs_share /mnt
```

但在你的 `/etc/exports` 文件中，实际对外共享的目录只有：

- `/home/ipc/sdk`
- `/home/ipc/sdk/Hi3519DV500_SDK...` (另外一个海思平台的路径)

由于 `/home/ipc/Hi3516/nfs_share` 不在白名单列表中，NFS 协议栈在第一道防线就直接拒绝了访问请求（抛出 `Permission denied`）。

### 修复步骤

**1. 修改导出表**

使用文本编辑器（以 `sudo` 权限）打开 `/etc/exports` 文件：

Bash

```
sudo nano /etc/exports
```

在文件末尾追加你当前正在使用的共享目录：

Plaintext

```
/home/ipc/Hi3516/nfs_share *(rw,sync,no_root_squash,no_subtree_check)
```

*(保存并退出)*

**2. 确保目录真实存在并赋权**

防止因为宿主机物理目录不存在导致的隐性拒绝：

Bash

```
mkdir -p /home/ipc/Hi3516/nfs_share
chmod 777 -R /home/ipc/Hi3516/nfs_share
```

**3. 重载 NFS 内核配置**

使刚刚修改的 `/etc/exports` 立即生效，无需重启服务：

Bash

```
sudo exportfs -arv
```

输出日志中必须明确出现 `exporting *:/home/ipc/Hi3516/nfs_share` 这一行。

## hi3516视频格式

图像中显示的终端输出是海思（HiSilicon）MPP（Media Processing Platform）视频编码模块（VENC）的经典测试程序（通常是 `sample_venc`）的交互界面。

这里的选型分为两大类：**GOP Mode（图像组模式）** 和 **RC Mode（码率控制模式）**。它们决定了视频压缩的底层策略，直接影响视频的画质、网络带宽占用以及存储消耗。

以下是基于底层编码机制的严肃分析：

### 一、 GOP Mode (Group of Pictures - 图像组模式)

GOP 决定了视频帧序列中 I 帧（关键帧，全量数据）和 P 帧（预测帧，增量数据）的排列结构与参考关系。

- **0) normal p. (Normal P-frame)**
  - **机制**：标准的 GOP 结构。一个 I 帧后接多个普通的 P 帧（例如 IPPPPP...），每个 P 帧只参考它前面的前一帧。
  - **特点**：兼容性最好，是所有解码器都支持的基础模式。适用于常规的视频录制与传输。
- **1) dual p. (Dual P-frame)**
  - **机制**：双参考帧模式。在此模式下，P 帧不仅可以参考前一帧，还可以参考更早的一帧。
  - **特点**：能够在不显著增加码率的情况下提升预测准确度，从而提高画质，或在同等画质下稍微降低码率。代价是增加了编码侧的内存占用和计算负荷。
- **2) smart p. (Smart P-frame / 智能 P 帧)**
  - **机制**：海思针对安防监控场景（IPC）的专有优化技术。它通常利用一个“长期参考帧”（通常是背景帧）作为虚拟 I 帧，后续的 P 帧可以跨越时间参考这个长期帧。
  - **特点**：在**静止或微小运动的场景**下，能成倍降低码率（甚至可降至常规的 1/3 到 1/5），因为静止背景不需要重复编码。一旦画面发生剧烈运动，它会动态调整参考策略。非常适合长时间定点监控场景。

------

### 二、 RC Mode (Rate Control - 码率控制模式)

码率控制决定了编码器如何在不同的画面复杂度下分配数据量（Bit/s）。这在受限的网络传输环境（如 5G/Wi-Fi）中是核心考量点。

- **c) cbr. (Constant Bit Rate - 固定码率)**
  - **机制**：编码器强制输出恒定大小的数据流。无论画面是静止还是剧烈运动，每秒的数据量（带宽占用）都卡在设定的目标值上。
  - **特点**：网络传输极其稳定，**不会产生带宽毛刺**。代价是画质不稳定：静止画面浪费带宽，剧烈运动画面（如大面积树叶摇晃）会因为码率不够而产生严重的马赛克（块效应）。
- **v) vbr. (Variable Bit Rate - 可变码率)**
  - **机制**：以目标画质为基准，根据画面复杂度自动调整码率。画面复杂时码率飙升，画面静止时码率下降。
  - **特点**：整体画质最优，存储空间利用率高。但**网络极度不友好**，剧烈运动时的码率突发（Peak Bitrate）极易导致无线网络（如 Wi-Fi/5G）拥塞丢包，引发视频卡顿。
- **b) abr. (Average Bit Rate - 平均码率)**
  - **机制**：CBR 和 VBR 的折中。在短时间内码率可以波动（类似 VBR），但在一段较长的时间窗口内，平均码率会向设定的目标值收敛。
  - **特点**：兼顾了一定的画质和整体带宽预期，常用于流媒体点播（如本地存储回放）。
- **a) avbr. (Adaptive VBR - 自适应可变码率)**
  - **机制**：海思 IPC 场景下的主力推荐模式。在 VBR 的基础上增加了对运动场景的智能自适应：静止时极低码率，运动时提高码率但也受限于算法的平滑控制。
  - **特点**：比普通 VBR 节省存储，比 CBR 画质好，是目前网络摄像机（IPC）主流的通用选择。
- **x) cvbr. (Constrained VBR - 约束型可变码率)**
  - **机制**：给 VBR 加上了一个**硬性的最大码率上限（MaxBitrate）**。
  - **特点**：保证平时享受 VBR 的低带宽优势，同时确保在极端复杂场景下，码率不会冲破网络带宽的物理上限。**在无线网络传输方案中，CVBR 通常是最佳选择，能兼顾画质与链路防拥塞。**
- **q) qvbr. (Quality VBR - 质量优先可变码率)**
  - **机制**：以主观视觉质量为核心目标的 VBR 变种，根据人眼视觉模型动态分配码率，重点保证人眼敏感区域的清晰度。
- **f) fix_qp (Fixed Quantization Parameter - 固定量化参数)**
  - **机制**：**不控制码率**，直接锁定底层编码的压缩力度（QP 值，值越小压缩越少，画质越好）。
  - **特点**：每一帧都使用固定的压缩比例，导致最终输出的码率完全不可控，呈抛物线状剧烈波动。此模式**绝对不能用于实际产品**，仅供研发人员在前期调试时评估传感器（Sensor）的基础画质极限或进行算法比对。

- normal p：参考关系最简单，一般只参考前一帧或少量参考帧，不会超过 SPS 声明。
- smart p：虽然会做背景帧/虚拟 I 帧一类优化，但当前这套参数下 FFmpeg 看到的参考帧数量没有超过 SPS 限制。
- dual p：会多出 SP/分层参考帧，FFmpeg 看到的 DPB/reference 数量可能超过 SPS 的 max_num_ref_frames，于是报警。

# Linux Dropbear SSH 操作 SOP


## 从源码编译（适合定制或嵌入式系统）

1. 下载源码：

```
wget https://matt.ucc.asn.au/dropbear/releases/dropbear-2026.82.tar.bz2tar xjf dropbear-2026.82.tar.bz2cd dropbear-2026.82
```

2. 编译：

```
./configuremakesudo make install
```

3. 编译完成后，会生成：

- `dropbear`
- `dropbearkey`
- `dbclient`


## 1️⃣ 目录准备

```bash
# 进入存放 Dropbear 的目录
cd /home/r8188eu/ssh
```

目录应包含：

- `dropbear` → SSH 服务程序
    
- `dropbearkey` → 用于生成主机密钥
    
- `dbclient` → SSH 客户端
    

---

## 2️⃣ 生成主机密钥

```bash
# 生成 RSA 主机密钥
./dropbearkey -t rsa -f dropbear_rsa_host_key
```

说明：

- `-t rsa` → 指定密钥类型为 RSA
    
- `-f dropbear_rsa_host_key` → 输出密钥文件名，可自定义
    

✅ 完成后目录中会生成 `dropbear_rsa_host_key` 文件，这会在当前目录生成一个名为 `dropbear_rsa_host_key` 的私钥文件。嵌入式系统通常默认去 `/etc/dropbear/` 寻找密钥，为了避免底层 rootfs 目录缺失或只读问题，直接在当前工作目录生成并显式调用最为稳妥。

---

## 3️⃣ 启动 Dropbear SSH 服务

```bash
# 启动服务，监听 22 端口
./dropbear -B -r dropbear_rsa_host_key -p 22
./dropbear -B -r /etc/dropbear/dropbear_rsa_host_key -p 22
```

参数说明：

- `-r` → 指定主机密钥文件
    
- `-p` → 指定监听端口（默认 22）
    
- **临时绕过：** 启动时追加 `-B` 参数（允许空密码登录）：
---

## 4️⃣ 验证 SSH 服务运行

```bash
# 查看 dropbear 进程
ps aux | grep dropbear
```

- 如果看到类似 `dropbear -r dropbear_rsa_host_key -p 22` 的进程，说明服务已启动
    

---

## 5️⃣ 测试远程连接

假设服务器 IP 为 `192.168.1.100`，本地执行：

```bash
ssh 用户名@192.168.1.100
```

- `用户名` = Linux 上的账户名
    
- 默认端口 = 22（可自定义）
    

成功后可输入密码登录。

---

## 6️⃣ 停止 SSH 服务

```bash
killall dropbear
```

---

## 7️⃣ （可选）开机自启

在 `/etc/rc.local` 中添加：

```bash
/home/r8188eu/ssh/dropbear -r /home/r8188eu/ssh/dropbear_rsa_host_key -p 22 &
```

- `&` → 后台运行
    
- 保存文件后，系统启动时会自动启动 Dropbear SSH
    

---

💡 **注意事项**：

- Dropbear 不依赖 systemd，所有启动/停止命令都是直接运行二进制。
    
- 建议把 `dropbear_rsa_host_key` 放到固定路径，不要随意移动。
    
- 如果想修改端口，只需修改 `-p` 参数即可。
