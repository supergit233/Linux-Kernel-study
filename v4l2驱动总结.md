# V4L2 驱动总结

## 核心队列（链表）的**总结日志**

这是一份针对 V4L2 及其底层架构中所有核心队列（链表）的**全局硬核总结日志**。

为了便于你在实际编写驱动或排查 Kernel Crash 时作为速查手册，我将其按子系统划分为四个维度，并严格界定了它们的生命周期、所属域以及在内核调试（如 `dmesg` 或 `debugfs`）时的监控重点。

------

### 一、 异步匹配阶段 (V4L2 Async Framework)

**核心职责：** 解决离散硬件加载顺序导致的“找不到设备”问题。
**调试切入点：** 当系统启动后没有生成 `/dev/videoX` 时，最应该查阅这些队列。
#链表

| 队列名称                 | 所属域 / 所在结构体                               | 存储节点类型                | 入队触发时机 (En-queue)                                           | 出队/转移触发时机 (De-queue)                                    | 致命排查点 (Crash/Bug 根源)               |
| :------------------- | :---------------------------------------- | :-------------------- | :---------------------------------------------------------- | :------------------------------------------------------ | :--------------------------------- |
| **`notifier_list`**  | **全局 (Global)**<br>`v4l2-async.c`         | `v4l2_async_notifier` | Host 驱动 probe 结束，调用 `v4l2_async_notifier_register()` 时。     | Host 驱动 remove，调用 `v4l2_async_notifier_unregister()` 时。 | 如果卸载模块时忘记注销，会导致野指针 Panic。          |
| **`subdevice_list`** | **全局 (Global)**<br>`v4l2-async.c`         | `v4l2_subdev`         | Sensor/CSI 驱动 probe 结束，调用 `v4l2_async_register_subdev()` 时。 | 发生匹配并被 Notifier 认领时，或驱动被 `rmmod` 卸载时。                   | 查 `dmesg` 看是否有 i2c 探活失败，导致根本没进此队列。 |
| **`waiting`**        | **私有 (Private)**<br>`v4l2_async_notifier` | `v4l2_async_subdev`   | Host 解析设备树 (DT) 后，调用 `_add_fwnode_subdev()` 挂入预期清单。         | 在 `subdevice_list` 中匹配到真实硬件，**转移至 `done` 队列**。          | 如果图迟迟不闭合，说明这里面有“老赖”没匹配上。           |
| **`done`**           | **私有 (Private)**<br>`v4l2_async_notifier` | `v4l2_async_subdev`   | 从 `waiting` 链表中匹配成功并转移过来。触发 `.bound` 回调。                    | 绑定的子设备被动态卸载（热插拔/rmmod），**退回 `waiting` 队列**。             | `.unbind` 回调处理不当，会在此处产生内存泄漏。       |

------

### 二、 拓扑骨架阶段 (Media Controller)

**核心职责：** 在内存中建立不可篡改的硬件连线关系。
**调试切入点：** 应用程序通过 `media-ctl -p` 打印出的树状图，其底层数据全部来源于这三个链表。

| 队列名称       | 所属域 / 所在结构体        | 存储节点类型   | 入队触发时机                                                 | 状态变化/遍历时机                                            | 致命排查点 (Crash/Bug 根源)                                  |
| :------------- | :------------------------- | :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`entities`** | **私有**<br>`media_device` | `media_entity` | 子设备或 Video 节点调用 `media_device_register_entity()` 时。 | 用户层发起图遍历，或内核检查 Pipeline 连通性时遍历此表。     | Entity ID 冲突，或注册前未初始化 pad 数量。                  |
| **`links`**    | **私有**<br>`media_entity` | `media_link`   | Host 驱动在 `.bound` 中调用 `media_create_pad_link()` 时建立。 | 用户空间下发 `MEDIA_IOC_SETUP_LINK` 改变连通状态 (ENABLE/DISABLE)。 | 强制修改被标记为 `IMMUTABLE`（不可变）的硬件链路会导致内核警告。 |
| **`subdevs`**  | **私有**<br>`v4l2_device`  | `v4l2_subdev`  | 异步匹配 `.bound` 阶段，调用 `v4l2_device_register_subdev()` 时。 | 遍历下发全局命令，如 `v4l2_device_call_all(s_stream, 1)`。   | 这是 Host 实际管辖的“下属花名册”，区别于全局的 `subdevice_list`。 |

------

### 三、 数据流转阶段 (Videobuf2 & Driver)

**核心职责：** 管理 DMA 与应用层之间的所有权交接。**（发生频率最高、最易死锁的地方）**
**调试切入点：** 画面撕裂、掉帧、或 `DQBUF` 永久阻塞卡死。

| 队列名称                           | 所属域 / 所在结构体              | 存储节点类型           | 入队触发时机                                                 | 出队/转移触发时机                                            | 致命排查点 (Crash/Bug 根源)                                  |
| :--------------------------------- | :------------------------------- | :--------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`queued_list`**                  | **框架私有**<br>`vb2_queue`      | `vb2_buffer`           | 用户层通过 `ioctl(VIDIOC_QBUF)` 交出控制权时。               | `start_streaming` 后，或运行时，被框架取走交给底层驱动。     | 应用层 `QBUF` 速度慢于硬件出图速度，导致此列表空虚（饥饿）。 |
| **`pending_list` / `active_list`** | **驱动私有**<br>（各平台自定义） | `驱动私有 buffer 结构` | vb2 框架调用驱动的 `buf_queue` 回调时，驱动将其物理地址记录于此。 | **硬件中断 (ISR) 发生时**，DMA 已写满当前 Buffer，驱动将其摘下。 | **锁争用极高发区。** `buf_queue` 和 ISR 之间必须使用 Spinlock (`irqsave`) 严格保护。 |
| **`done_list`**                    | **框架私有**<br>`vb2_queue`      | `vb2_buffer`           | 驱动在 ISR 中摘下 Buffer 后，调用 `vb2_buffer_done()` 扔进此队列。 | 用户层通过 `ioctl(VIDIOC_DQBUF)` 取回控制权时。              | 驱动层在 `stop_streaming` 时，**必须**将 pending_list 中的残余 Buffer 强制转入此处并标为 ERROR，否则用户层 DQBUF 将永久休眠。 |

------

### 四、 旁路通信阶段 (V4L2 Events)

**核心职责：** 处理非视频流的异步事件（如垂直同步信号、分辨率动态改变）。
**调试切入点：** 应用程序 `select/poll` 返回 Exception fd 时。

| 队列名称         | 所属域 / 所在结构体       | 存储节点类型            | 入队触发时机                                               | 出队触发时机                                               | 致命排查点 (Crash/Bug 根源)                                  |
| :--------------- | :------------------------ | :---------------------- | :--------------------------------------------------------- | :--------------------------------------------------------- | :----------------------------------------------------------- |
| **`subscribed`** | **文件句柄**<br>`v4l2_fh` | `v4l2_subscribed_event` | 用户空间调用 `VIDIOC_SUBSCRIBE_EVENT` 明确订阅某类事件时。 | 用户调用 `VIDIOC_UNSUBSCRIBE_EVENT` 时。                   | 驱动发送了某事件，但用户态并未在此列表里注册订阅，事件将被丢弃。 |
| **`available`**  | **文件句柄**<br>`v4l2_fh` | `v4l2_kevent`           | 底层驱动侦测到硬件状态改变，调用 `v4l2_event_queue()` 时。 | 用户空间被 `poll` 唤醒后，调用 `VIDIOC_DQEVENT` 取出事件。 | 队列深度是有限的（通常较小），如果用户层不及时读走，新事件会覆盖老事件（或者被丢弃，取决于 flag 设置）。 |

### 韦东山的用户态代码流程

没写查询能力VIDIOC_QUERYCAP部分，但有枚举格式与分辨率部分

```C

#include <sys/types.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <sys/ioctl.h>
#include <unistd.h>
#include <stdio.h>
#include <string.h>
#include <linux/types.h>          /* for videodev2.h */
#include <linux/videodev2.h>
#include <poll.h>
#include <sys/mman.h>
#include <pthread.h>

/* ./video_test </dev/video0> */

static void *thread_brightness_control (void *args)
{
    int fd = (int)args;

    unsigned char c;
    int brightness;
    int delta;
    
    struct v4l2_queryctrl   qctrl;
    memset(&qctrl, 0, sizeof(qctrl));
    qctrl.id = V4L2_CID_BRIGHTNESS; // V4L2_CID_BASE+0;
    if (0 != ioctl(fd, VIDIOC_QUERYCTRL, &qctrl))
    {
        printf("can not query brightness\n");
        return NULL;
    }

    printf("brightness min = %d, max = %d\n", qctrl.minimum, qctrl.maximum);
    delta = (qctrl.maximum - qctrl.minimum) / 10;
        
    struct v4l2_control ctl;
    ctl.id = V4L2_CID_BRIGHTNESS; // V4L2_CID_BASE+0;
    ioctl(fd, VIDIOC_G_CTRL, &ctl);

    while (1)
    {
        c = getchar();
        if (c == 'u' || c == 'U')
        {
            ctl.value += delta;
        }
        else if (c == 'd' || c == 'D')
        {
            ctl.value -= delta;
        }
        if (ctl.value > qctrl.maximum)
            ctl.value = qctrl.maximum;
        if (ctl.value < qctrl.minimum)
            ctl.value = qctrl.minimum;

        ioctl(fd, VIDIOC_S_CTRL, &ctl);
    }
    return NULL;
}

int main(int argc, char **argv)
{
    int fd;
    struct v4l2_fmtdesc fmtdesc;
    struct v4l2_frmsizeenum fsenum;
    int fmt_index = 0;
    int frame_index = 0;
    int i;
    void *bufs[32];
    int buf_cnt;
    int type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    struct pollfd fds[1];
    char filename[32];
    int file_cnt = 0;

    if (argc != 2)
    {
        printf("Usage: %s </dev/videoX>, print format detail for video device\n", argv[0]);
        return -1;
    }

    /* open */
    fd = open(argv[1], O_RDWR);
    if (fd < 0)
    {
        printf("can not open %s\n", argv[1]);
        return -1;
    }

    /* 查询能力 */
    struct v4l2_capability cap;
    memset(&cap, 0, sizeof(struct v4l2_capability));
    
    if (0 == ioctl(fd, VIDIOC_QUERYCAP, &cap))
    {        
        if((cap.capabilities & V4L2_CAP_VIDEO_CAPTURE) == 0) {
            fprintf(stderr, "Error opening device %s: video capture not supported.\n",
                    argv[1]);
            return -1;
        }
        
        if(!(cap.capabilities & V4L2_CAP_STREAMING)) {
            fprintf(stderr, "%s does not support streaming i/o\n", argv[1]);
            return -1;
        }
    }
    else
    {
        printf("can not get capability\n");
        return -1;
    }

    while (1)
    {
        /* 枚举格式 */
        fmtdesc.index = fmt_index;  // 比如从0开始
        fmtdesc.type  = V4L2_BUF_TYPE_VIDEO_CAPTURE;  // 指定type为"捕获"
        if (0 != ioctl(fd, VIDIOC_ENUM_FMT, &fmtdesc))
            break;

        frame_index = 0;
        while (1)
        {
            /* 枚举这种格式所支持的帧大小 */
            memset(&fsenum, 0, sizeof(struct v4l2_frmsizeenum));
            fsenum.pixel_format = fmtdesc.pixelformat;
            fsenum.index = frame_index;

            if (ioctl(fd, VIDIOC_ENUM_FRAMESIZES, &fsenum) == 0)
            {
                printf("format %s,%d, framesize %d: %d x %d\n", fmtdesc.description, fmtdesc.pixelformat, frame_index, fsenum.discrete.width, fsenum.discrete.height);
            }
            else
            {
                break;
            }

            frame_index++;
        }

        fmt_index++;
    }


    /* 设置格式 */
    struct v4l2_format fmt;
    memset(&fmt, 0, sizeof(struct v4l2_format));
    fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    fmt.fmt.pix.width = 1024;
    fmt.fmt.pix.height = 768;
    fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_MJPEG;
    fmt.fmt.pix.field = V4L2_FIELD_ANY;
    if (0 == ioctl(fd, VIDIOC_S_FMT, &fmt))
    {
        printf("set format ok: %d x %d\n", fmt.fmt.pix.width, fmt.fmt.pix.height);
    }
    else
    {
        printf("can not set format\n");
        return -1;
    }

    /*
     * 申请buffer
     */
    struct v4l2_requestbuffers rb;
    memset(&rb, 0, sizeof(struct v4l2_requestbuffers));
    rb.count = 32;
    rb.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    rb.memory = V4L2_MEMORY_MMAP;

    if (0 == ioctl(fd, VIDIOC_REQBUFS, &rb))
    {
        /* 申请成功后, mmap这些buffer */
        buf_cnt = rb.count;
        for(i = 0; i < rb.count; i++) {
            struct v4l2_buffer buf;
            memset(&buf, 0, sizeof(struct v4l2_buffer));
            buf.index = i;
            buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
            buf.memory = V4L2_MEMORY_MMAP;
            if (0 == ioctl(fd, VIDIOC_QUERYBUF, &buf))
            {
                /* mmap */
                bufs[i] = mmap(0 /* start anywhere */ ,
                                  buf.length, PROT_READ | PROT_WRITE, MAP_SHARED, fd,
                                  buf.m.offset);
                if(bufs[i] == MAP_FAILED) {
                    perror("Unable to map buffer");
                    return -1;
                }
            }
            else
            {
                printf("can not query buffer\n");
                return -1;
            }            
        }

        printf("map %d buffers ok\n", buf_cnt);
        
    }
    else
    {
        printf("can not request buffers\n");
        return -1;
    }

    /* 把所有buffer放入"空闲链表" */
    for(i = 0; i < buf_cnt; ++i) {
        struct v4l2_buffer buf;
        memset(&buf, 0, sizeof(struct v4l2_buffer));
        buf.index = i;
        buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
        buf.memory = V4L2_MEMORY_MMAP;
        if (0 != ioctl(fd, VIDIOC_QBUF, &buf))
        {
            perror("Unable to queue buffer");
            return -1;
        }
    }
    printf("queue buffers ok\n");

    /* 启动摄像头 */
    if (0 != ioctl(fd, VIDIOC_STREAMON, &type))
    {
        perror("Unable to start capture");
        return -1;
    }
    printf("start capture ok\n");


    /* 创建线程用来控制亮度 */
    pthread_t thread;
    pthread_create(&thread, NULL, thread_brightness_control, (void *)fd);
   

    while (1)
    {
        /* poll */
        memset(fds, 0, sizeof(fds));
        fds[0].fd = fd;
        fds[0].events = POLLIN;
        if (1 == poll(fds, 1, -1))
        {
            /* 把buffer取出队列 */
            struct v4l2_buffer buf;
            memset(&buf, 0, sizeof(struct v4l2_buffer));
            buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
            buf.memory = V4L2_MEMORY_MMAP;
            
            if (0 != ioctl(fd, VIDIOC_DQBUF, &buf))
            {
                perror("Unable to dequeue buffer");
                return -1;
            }
            
            /* 把buffer的数据存为文件 */
            sprintf(filename, "video_raw_data_%04d.jpg", file_cnt++);
            int fd_file = open(filename, O_RDWR | O_CREAT, 0666);
            if (fd_file < 0)
            {
                printf("can not create file : %s\n", filename);
            }
            printf("capture to %s\n", filename);
            write(fd_file, bufs[buf.index], buf.bytesused);
            close(fd_file);

            /* 把buffer放入队列 */
            if (0 != ioctl(fd, VIDIOC_QBUF, &buf))
            {
                perror("Unable to queue buffer");
                return -1;
            }
        }
    }

    if (0 != ioctl(fd, VIDIOC_STREAMOFF, &type))
    {
        perror("Unable to stop capture");
        return -1;
    }
    printf("stop capture ok\n");
    close(fd);

    return 0;
}
```

### 标准用户态代码流程
#标准流程
没有执行枚举格式与分辨率命令，这两个是用来给用户参考以便后续设置格式的

```C
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <fcntl.h>
#include <unistd.h>
#include <sys/ioctl.h>
#include <sys/mman.h>
#include <linux/videodev2.h>

// 用于记录映射到用户空间后的内存起始地址和长度
struct buffer {
    void   *start;
    size_t  length;
};

int main() {
    int fd;
    struct v4l2_capability cap;
    struct v4l2_format fmt;
    struct v4l2_requestbuffers req;
    struct v4l2_buffer buf;
    struct buffer *buffers;
    unsigned int n_buffers;

    // ==========================================================
    // 步骤 1：打开设备节点
    // ==========================================================
    // O_RDWR: 读写模式，O_NONBLOCK: 非阻塞模式（本例中为简化代码未使用非阻塞特性）
    fd = open("/dev/video0", O_RDWR | O_NONBLOCK, 0);
    if (fd < 0) {
        perror("Cannot open device");
        return -1;
    }

    // ==========================================================
    // 步骤 2：查询设备能力 (Query Capability)
    // ==========================================================
    // 作用：确认底层驱动是否支持视频捕获功能（有些 /dev/videoX 可能是纯输出设备）
    ioctl(fd, VIDIOC_QUERYCAP, &cap);
    if (!(cap.capabilities & V4L2_CAP_VIDEO_CAPTURE)) {
        fprintf(stderr, "Not a video capture device\n");
        return -1;
    }

    // ==========================================================
    // 步骤 3：设置图像格式 (Set Format)
    // ==========================================================
    // 作用：与底层硬件协商所需的分辨率和色彩空间格式
    memset(&fmt, 0, sizeof(fmt));
    fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    fmt.fmt.pix.width       = 1920;             // 期望宽度 1920
    fmt.fmt.pix.height      = 1080;             // 期望高度 1080
    fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_YUYV; // 像素格式 YUYV (4:2:2)
    fmt.fmt.pix.field       = V4L2_FIELD_INTERLACED; // 交错模式（现代传感器多用 V4L2_FIELD_NONE 逐行扫描）
    ioctl(fd, VIDIOC_S_FMT, &fmt);

    // ==========================================================
    // 步骤 4：请求分配内核缓冲区 (Request Buffers)
    // ==========================================================
    // 作用：告诉内核我们需要使用 MMAP 内存映射模式，并申请 4 个帧缓冲区以供轮转
    memset(&req, 0, sizeof(req));
    req.count  = 4;                             // 申请 4 个 Buffer
    req.type   = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    req.memory = V4L2_MEMORY_MMAP;              // 内存模式：内存映射 (Memory Mapping)
    ioctl(fd, VIDIOC_REQBUFS, &req);

    // 分配用户态管理这些缓冲区的结构体数组
    // req.count以内核分配为准，不一定是设定的4个，就跟前面fmt.pix.分辨率设置一样内核会根据实际情况给出值
    buffers = calloc(req.count, sizeof(*buffers));

    // ==========================================================
    // 步骤 5：内存映射 (mmap) 与 缓冲区入队 (QBUF)
    // ==========================================================
    // 作用：将内核态的物理内存映射到用户态虚拟地址空间，避免后期 read/write 的数据拷贝。
    //       映射完成后，立刻把它们塞入底层的输入队列，等待硬件填充图像数据。
    for (n_buffers = 0; n_buffers < req.count; ++n_buffers) {
        memset(&buf, 0, sizeof(buf));
        buf.type        = V4L2_BUF_TYPE_VIDEO_CAPTURE;
        buf.memory      = V4L2_MEMORY_MMAP;
        buf.index       = n_buffers;

        // 查询缓冲区的物理偏移地址和实际长度
        ioctl(fd, VIDIOC_QUERYBUF, &buf);

        buffers[n_buffers].length = buf.length;
        
        // 核心动作：将内核显存地址映射到本进程的 buffers[n_buffers].start
        buffers[n_buffers].start = mmap(NULL, buf.length,
                                        PROT_READ | PROT_WRITE, MAP_SHARED,
                                        fd, buf.m.offset);

        // 将空闲缓冲区入队 (Queue Buffer)，交还给驱动程序准备采集
        ioctl(fd, VIDIOC_QBUF, &buf);
    }

    // ==========================================================
    // 步骤 6：启动视频流采集 (Stream On)
    // ==========================================================
    // 作用：触发底层硬件管线开始工作。Sensor 开始曝光，MIPI 开始传输，ISP 开始写入前面申请的内核 Buffer。
    enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    ioctl(fd, VIDIOC_STREAMON, &type);

    // ==========================================================
    // 步骤 7：采集主循环 (DQBUF -> Process -> QBUF)
    // ==========================================================
    // 本例循环 100 次，代表抓取 100 帧图像
    for (int i = 0; i < 100; i++) {
        fd_set fds;
        FD_ZERO(&fds);
        FD_SET(fd, &fds);
        
        // 使用 select 阻塞等待，直到硬件写满一帧数据并触发中断唤醒
        select(fd + 1, &fds, NULL, NULL, NULL);

        memset(&buf, 0, sizeof(buf));
        buf.type   = V4L2_BUF_TYPE_VIDEO_CAPTURE;
        buf.memory = V4L2_MEMORY_MMAP;

        // 【出队 (Dequeue Buffer)】
        // 作用：从驱动输出队列中取出一个已经被硬件写满图像数据的 Buffer。
        // 调用完成后，buf.index 会指向当前包含最新数据的那个缓冲区索引。
        ioctl(fd, VIDIOC_DQBUF, &buf);

        // -------------------------------------------------------------
        // 【关键处理区：用户态数据消费】
        // 此时，本帧视频数据在内存中的绝对起始地址是：buffers[buf.index].start
        // 实际有效数据大小是：buf.bytesused
        // 
        // 警告：这里不能有任何耗时操作（如慢速磁盘 IO）！
        // 如果处理过慢，底层的 4 个 Buffer 会全部被硬件写满却无法归还，导致硬件被迫丢帧。
        printf("Frame %d captured, index: %d, bytes used: %d\n", i, buf.index, buf.bytesused);
        // -------------------------------------------------------------

        // 【重新入队 (Queue Buffer)】
        // 作用：数据消费完毕，该 Buffer 被标记为空闲。立即交还给底层驱动循环使用。
        ioctl(fd, VIDIOC_QBUF, &buf);
    }

    // ==========================================================
    // 步骤 8：停止视频流并清理资源
    // ==========================================================
    // 作用：关闭硬件管线，解除内存映射，关闭设备节点
    ioctl(fd, VIDIOC_STREAMOFF, &type);
    
    for (int i = 0; i < n_buffers; ++i) {
        munmap(buffers[i].start, buffers[i].length);
    }
    free(buffers);
    close(fd);

    return 0;
}
```

# V4L2 驱动全生命周期总结

------

## 一、先给总结论

V4L2 驱动不是“实现一组 ioctl”这么简单。
对 **5.10 平台型 camera 驱动** 来说，它的本质是 4 条线同时成立：

- **拓扑线**：设备树 / fwnode 把 pipeline 画出来
- **对象线**：probe 把 `v4l2_device`、`media_device`、`subdev`、`video_device`、`vb2_queue` 这些对象建出来
- **装配线**：async notifier 把本来分散 probe 的设备异步绑定成一张 media graph
- **数据线**：vb2 + DMA/IRQ 把 buffer 真正流起来，并在 `stop_streaming/remove` 时完整收尾

所以真正的驱动思维不是“会调 API”，而是会回答这 4 个问题：

- 图是怎么描述的
- 对象是谁拥有谁
- 设备是什么时候真正接上的
- 数据是怎么从硬件流到 `DQBUF` 的

------

## 二、完整生命周期的主流程

把全生命周期压成一句话就是：

**设备树描述拓扑**
→ **platform/i2c 各设备分别 probe**
→ **主设备初始化 v4l2/media/vb2 等核心对象**
→ **解析 endpoint，建立 async 等待项**
→ **sensor 等 subdev 注册并被匹配**
→ **`.bound()` 补 media link，`.complete()` 宣告异步图闭合**
→ **video node 暴露给用户态**
→ **用户态通过 ioctl + vb2 开始采流**
→ **中断/DMA 完成后 `vb2_buffer_done()`**
→ **`STREAMOFF/remove` 按依赖逆序回收**

这就是 5.10 平台 camera 驱动的全链路。

------

## 三、设备树的本质结论

设备树在这里最重要的作用，不是“填几个驱动参数”，而是：

> **描述媒体拓扑和链路属性**

它主要提供两类信息：

### 1. 谁和谁相连

通过 `ports/port/endpoint/remote-endpoint` 描述。

### 2. 这条边是什么总线属性

例如：

- CSI-2 / parallel / BT656
- lanes
- flags
- bus type

驱动在 probe 时并不会直接从 DT 拿到“远端设备指针”，而是：

- 先解析本地 endpoint
- 找 remote endpoint
- 再把这个 remote endpoint 变成 **async match 条件**

所以 DT 先变成“等待清单”，而不是立刻变成“连接完成的图”。

------

## 四、probe 的本质结论

probe 的目的不是“一次性把整条 camera pipeline 跑起来”，而是：

> **先把主设备自己的世界搭起来**

它一般分两段：

### 第一段：基础资源

- 寄存器
- 时钟
- reset
- IRQ
- runtime PM
- 硬件 ID 检查

这是让设备“活着”。

### 第二段：V4L2/media 主容器

- 私有大结构体
- `v4l2_device`
- `media_device`
- 内部 entities
- `video_device`
- `vb2_queue`

这是让设备“进入 V4L2/media 世界”。

这里一个非常重要的结论是：

> **内部 graph 往往先于外部 sensor 绑定建立**

也就是说，在像 `rkisp1` 这种驱动里：

- ISP / CSI / resizer / capture / stats / params 这些内部节点，probe 阶段就建好了
- 外部 sensor 到 ISP/CSI 的边，是后来 async `.bound()` 才补上的

------

## 五、对象关系的结论

很多人学 V4L2 容易把 `/dev/video0` 当主角，但源码告诉你不是这样。

真正的对象关系应该理解为：

```text
私有设备对象
 ├─ v4l2_device      // V4L2 家族容器
 ├─ media_device     // graph 容器
 ├─ 若干 v4l2_subdev // ISP/CSI/sensor 等实体
 ├─ video_device     // 暴露给用户态的入口
 └─ vb2_queue        // buffer 生命周期核心
```

关键结论：

- **`video_device` 只是对外入口，不是全局 owner**
- **`media_device` 管图**
- **`v4l2_device` 管设备家族**
- **`vb2_queue` 管 buffer 生命周期**
- **真正的业务状态都挂在驱动私有结构体里**

------

## 六、async notifier 的核心结论

async 的意义不只是“晚点注册也能工作”，而是：

> **把不可控的 probe 顺序，收敛成可控的装配时机**

因为：

- sensor 可能先 probe
- ISP 可能后 probe
- CSI 可能更晚
- 但最终 graph 必须一致

所以 async 框架做的事情是：

- 主设备把“我在等谁”放进 waiting list
- subdev 自己注册成可匹配对象
- 匹配成功后：
  - 先 `v4l2_device_register_subdev()`
  - 再调用 `.bound()`
- 所有等待项都满足后，调用 `.complete()`

这里最关键的结论有两个：

### 1. `.bound()` 不是“注册 subdev”

因为调用 `.bound()` 之前，subdev 已经挂到 `v4l2_device` 里了。

所以 `.bound()` 的职责是：

- 保存远端 `sd` 指针
- 找 pad
- 补 media link
- 补 pipeline 关系

### 2. `.complete()` 不是“第一次创建设备”

而是：

> **整棵异步图已经闭合，可以做最终暴露或收尾注册**

------

## 七、media graph 的核心结论

media graph 不是抽象概念，它在源码里就是三层对象：

- **entity**：节点
- **pad**：输入/输出端口
- **link**：pad 之间的边

对于平台 camera 驱动，graph 一般分两部分：

### 1. 内部静态图

例如：

- CSI → ISP
- ISP → Resizer
- Resizer → Capture
- Params → ISP
- ISP → Stats

这些往往在 probe 阶段就通过 `media_create_pad_link()` 建好了。

### 2. 外部异步边

例如：

- Sensor source pad → CSI sink pad
- 或 Sensor source pad → ISP sink pad

这些边通常在 `.bound()` 里建立。

所以你调试 graph 时，必须先分清：

- 是内部静态链没建好
- 还是外部 sensor 没有 bound 上

这两个问题的查法完全不同。

------

## 八、video node 的结论

`/dev/videoX` 的出现，不等于 camera 系统刚刚诞生。
恰恰相反：

> **用户看到 `/dev/videoX` 时，前面大部分生命周期通常已经完成了**

因为在这之前，驱动通常已经完成：

- V4L2/media 主容器建立
- 内部 entities 建立
- 大部分 link 建立
- vb2 queue 初始化

所以 `video_device` 只是把内核里的结果暴露给用户态，而不是整个系统的起点。

------

## 九、vb2 的核心结论

vb2 是数据面的核心。
对 capture 驱动来说，真正的运行态不是 ioctl 本身，而是这条 buffer 生命周期：

```text
REQBUFS -> queue_setup
QBUF -> buf_queue
STREAMON -> start_streaming
DMA/IRQ完成 -> vb2_buffer_done
DQBUF -> 用户取走完成帧
QBUF -> 用户归还空buffer
STREAMOFF -> stop_streaming -> 全部回收
```

这条线的重要程度高于单个 ioctl 名字。

核心结论：

- `REQBUFS` 是建立 buffer 池
- `QBUF` 是把 buffer 交给驱动内部 active queue
- `STREAMON` 才是真正启动硬件流
- `vb2_buffer_done()` 是用户态能 `DQBUF` 的前提
- `STREAMOFF` 必须把驱动持有的 buffer 全部收回

------

## 十、用户态样例对应的本质结论

你前面给的用户态程序：

```c
open
QUERYCAP
S_FMT
REQBUFS
QUERYBUF + mmap
QBUF
STREAMON
select + DQBUF + QBUF
STREAMOFF
```

从驱动角度翻译其实就是：

1. 给我一个 video 会话入口
2. 告诉我设备能力
3. 协商当前格式
4. 建立 vb2 buffer 池
5. 把这些 buffer 映射到用户态
6. 把空 buffer 全部交给驱动
7. 启动硬件/管线
8. 每一帧完成后 done 给我
9. 我取走、处理、再还回去
10. 最后彻底停流并收干净

这说明用户程序消费的是 **“已经建立好的对象和 graph”**，并不参与“建立 camera pipeline”本身。

------

## 十一、中断 / DMA / `vb2_buffer_done()` 的结论

真正决定 `DQBUF` 能不能返回的，不是 graph 有没有存在，而是：

> **硬件完成路径有没有把 buffer 放进 vb2 done 队列**

也就是说：

- graph 建得再漂亮
- `/dev/video0` 也正常出现
- ioctl 也都能调通

如果中断路径里没有正确：

- 拿到当前 buffer
- 写 `bytesused/sequence/timestamp`
- 调 `vb2_buffer_done()`

那用户态还是拿不到帧。

所以调试时要始终记住：

- **graph 解决“谁跟谁连”**
- **vb2 + ISR 解决“帧怎么回来”**

这是两条不同的问题线。

------

## 十二、`stop_streaming()` 的结论

这是最容易被低估、但最能体现驱动质量的部分。

一个驱动能不能只启动一次，不重要；
重要的是它能不能：

- 正常停流
- 异常停流
- start 失败后恢复
- remove 时收尾干净
- 下一次还能正常起流

`stop_streaming()` 的硬要求是：

1. 停 DMA / 停中断 / 停 sensor 或子模块
2. 停止接受新的帧完成
3. 把驱动内部 active queue 里的 buffer 全部归还给 vb2
4. 保证下次 `STREAMON` 还能从干净状态开始

一句话：

> **start path 让它跑起来，stop path 决定它稳不稳。**

很多“第一次采流正常，第二次死掉”的 bug，根都在 stop path。

------

## 十三、remove 的结论

remove 不是“close 的放大版”，它是整个生命周期的逆向拆解。

正确顺序应该是按依赖反着来：

1. 先禁止新的 userspace 访问
2. 注销 notifier
3. 解绑 subdev / 外部边
4. 注销 video node / entities
5. 注销 media_device
6. 注销 v4l2_device
7. 关时钟 / IRQ / PM / 寄存器资源

所以 remove 的本质是：

> **把运行态世界、图世界、对象世界按依赖反向折叠回去**

如果 remove 顺序错了，就容易出现：

- dangling link
- subdev 引用悬挂
- 下次 probe 异常
- 资源泄露
- PM 状态错乱

# V4L2 核心架构与异步流转梳理日志 (完整修正版)

## 第一部分：推翻单线谬误 —— 双轨并发的初始化时序模型

在 Linux 设备模型 (LDM) 中，设备树的解析和驱动的 `probe` 是高度并发的。Sensor 驱动 (I2C) 和 Host 驱动 (Platform) 的生命周期是平行的，必须以**双轨合并**的视角来理解系统的启动。

### 轨道 A：外设线 (如 I2C Sensor，执行时机随机)

1. **硬件探活**：总线匹配，触发 `probe()`。拉起底层时钟与电源，读取 Chip ID 确认硬件存活。
2. **逻辑实例化**：构造并初始化 `v4l2_subdev` 结构体。
3. **异步宣告入库**：调用 `v4l2_async_register_subdev()`，将自身肉身挂入全局 **`subdevice_list`**。随后休眠，等待被唤醒。

### 轨道 B：主控线 (如 Platform CSI/ISP，执行时机随机)

1. **顶层容器注册 (必须极其靠前)**：在 `probe()` 最前期，调用 `media_device_init()` 和 `v4l2_device_register()`，确立图骨架和行政管理权。
2. **查阅静态族谱**：解析设备树 (DT)。**注意：此时绝不关心对端硬件是否存活或驱动是否加载**。纯粹基于静态的文本连接映射。
3. **登记预期名单**：将解析出的实体包装为 `asd`，指定匹配规则，强行挂入 Notifier 的私有 **`waiting`** 链表。
4. **激活匹配引擎**：调用 `v4l2_async_notifier_register()` 挂入全局 **`notifier_list`**，向核心层宣告“我已就绪，开始帮我寻人”。

### 轨道 C：核心层撮合与图闭合 (V4L2 Async Framework 接管)

只要 A 和 B 轨道的最终挂载动作都已完成（不分先后），核心层瞬间触发以下状态机：

1. **匹配命中**：将实体从 Host 的 `waiting` 移至 `done` 链表。
2. **权属划归**：核心层调用 `v4l2_device_register_subdev()`，Sensor 认领 Host 为直接上级（进入 `subdevs` 链表）。
3. **连实线 (`.bound`)**：回调 Host 驱动，执行 `media_create_pad_link()`，在内存图纸中闭合物理链路。
4. **大局底定 (`.complete`)**：`waiting` 链表清空，回调 Host 驱动。在此**最终合法时机**，调用 `video_register_device()` 暴露出 `/dev/videoX` 节点，向用户层开放。

## 第二部分：异步匹配机制的多维性 (Match Types)

Host 将设备放入 `waiting` 链表时，并非只有“端点 (endpoint)”这一种死板的途径。内核底层定义了 4 种 `v4l2_async_match_type` 匹配规则：

1. **`V4L2_ASYNC_MATCH_FWNODE` (固件节点 - 绝对主流)**：对比双方绑定的设备树或 ACPI 节点指针是否一致。现代 ARM/SoC 必备。
2. **`V4L2_ASYNC_MATCH_I2C` (硬件地址)**：硬编码指定 I2C 适配器号和器件物理地址（如 I2C-2, 0x3c）。老旧系统或 board file 时代常用。
3. **`V4L2_ASYNC_MATCH_DEVNAME` (字符串)**：直接执行 `strcmp` 匹配设备名称。用于非标准总线或虚拟驱动（如 vivid）。
4. **`V4L2_ASYNC_MATCH_CUSTOM` (自定义)**：Host 提供自定义 `.match` 回调函数，拥有读取任何寄存器进行后门校验的终极权力。

## 第三部分：VB2 内存流转与所有权协议

V4L2 内存管理的本质是**所有权 (Ownership) 的严格轮转**。一旦发生越权读写，必导致数据撕裂或死锁。

1. **归属应用层**：状态 `DEQUEUED`。应用层通过 `mmap` 读写，或者通过 `REQBUFS` 刚申请完毕。
2. **归属 VB2 框架**：状态 `QUEUED` (处于 `queued_list`)。应用层调用 `QBUF` 交出控制权，数据进入排队池。
3. **归属 底层 DMA**：状态 `ACTIVE` (处于驱动私有 `pending_list`)。驱动将物理地址写入硬件寄存器，DMA 正在搬运数据。**（此阶段高频竞争，必须带 Spinlock `irqsave` 保护）**
4. **归属 VB2 框架**：状态 `DONE` (处于 `done_list`)。硬件发生**完成中断 (ISR)**，驱动调用 `vb2_buffer_done` 交还框架。应用层通过 `DQBUF` 取走，重回阶段 1。

> **致命陷阱**：在应用层下发 `STREAMOFF` 或驱动卸载时，驱动必须强制清空 `pending_list`，并将残余 Buffer 标记为 `VB2_BUF_STATE_ERROR` 返还系统。否则用户态进程将陷入永久的休眠阻塞 (D状态)。

# 提问与回答

## 不同传输模式优缺点

**USERPTR 的致命缺陷：** USERPTR 的本质是传递用户空间的虚拟地址（VA）。内核为了让 DMA 能够访问这些虚拟地址对应的物理内存，必须通过 `get_user_pages()` 或类似机制将这些物理页“锁”在内存中（Pinning），防止被 Swap 出去。这会带来严重的内存碎片化问题，且极其容易引发 OOM（Out of Memory）或被恶意利用导致内核态安全风险。更重要的是，用户空间的内存通常是 Cacheable 的，硬件 DMA 读写时如果不进行复杂的 Cache 刷新/失效（Flush/Invalidate）操作，必然导致数据损坏。

**DMABUF 的降维打击：** 现代内核摒弃 USERPTR 转向 DMABUF，核心在于 **文件描述符（FD）抽象与 Scatter-Gather (SG) List**。DMABUF 允许不同的硬件（如 ISP、VPU、GPU）通过一个跨进程、跨驱动的 FD 共享同一块物理内存。在一致性处理上，V4L2 驱动并不是袖手旁观的，底层会通过 dma-buf 框架的 `dma_buf_begin_cpu_access()` 和 `dma_buf_end_cpu_access()` 强制执行 Cache 同步，彻底对用户态屏蔽了复杂的 CPU/DMA 缓存一致性问题

##  **vb2 缓冲区的内核生命周期** 

### videobuf2 内核状态机全景解析

在内核源码（`include/media/videobuf2-core.h`）中，一个 `vb2_buffer` 主要会在以下几个核心状态之间流转：

1. **`VB2_BUF_STATE_DEQUEUED` (出队状态 / 用户态持有)**
   - **含义：** 这个 Buffer 目前由用户空间控制，内核和硬件绝对不会去触碰它。
   - **触发点：** `VIDIOC_REQBUFS` 刚分配完内存时，或者用户空间成功调用 `VIDIOC_DQBUF` 取走数据后。
2. **`VB2_BUF_STATE_PREPARING` (准备状态 - 短暂过渡)**
   - **含义：** 当用户空间下发 `QBUF` 操作，内核开始接管，并进行必要的 Cache 同步（Flush/Invalidate）或 DMA 映射检查时的短暂状态。
3. **`VB2_BUF_STATE_QUEUED` (入队状态 / VB2 核心层持有)**
   - **含义：** 用户态已经交出控制权，Buffer 进入了 VB2 框架维护的逻辑队列（通常是一个 FIFO 链表），等待被底层驱动拿走。
   - **触发点：** `VIDIOC_QBUF` 执行完毕，但底层硬件尚未开始处理它（例如还没执行 `STREAMON`）。
4. **`VB2_BUF_STATE_ACTIVE` (活跃状态 / 硬件驱动持有 - 最危险的阶段)**
   - **含义：** 底层驱动（例如你的 MIPI 或 VI 驱动）已经通过 `vb2_ops->buf_queue` 回调，把这个 Buffer 从 VB2 核心队列拿走了，并且 **把它的物理地址填入了硬件 DMA 寄存器**。此时硬件可能正在往里面疯狂写数据。
   - **关键点：** 这个状态下的 Buffer 受驱动内部的自旋锁（Spinlock）保护，任何对它的非法访问都会导致 Panic。
5. **`VB2_BUF_STATE_DONE` (完成状态)**
   - **含义：** 硬件 DMA 传输完成。
   - **触发点：** 硬件触发 Frame End 中断，驱动的 ISR（中断服务例程）中必须调用 `vb2_buffer_done(vb, VB2_BUF_STATE_DONE)`。
6. **`VB2_BUF_STATE_ERROR` (错误状态)**
   - **含义：** 硬件传输过程中发生错误（如 MIPI FIFO 溢出、CRC 校验失败）。ISR 同样调用 `vb2_buffer_done`，但传入 `VB2_BUF_STATE_ERROR`。虽然数据是坏的，但流程必须闭环，让用户层去丢弃它。

## 用户调用的到驱动执行的流程
