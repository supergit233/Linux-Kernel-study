# Kernel Driver Study

## 总览

本 Skill 用于建立和维护结构化的 Linux 内核驱动源码学习项目。
重点不是孤立记忆 API，而是围绕对象模型、生命周期、主流程、回调关系、资源模型、执行上下文、用户态到内核路径以及工程调试方法，形成可复用的驱动学习笔记规范。

## 使用场景

当用户提出下面任务时，使用本 Skill：

- 新建或维护 Linux 驱动源码学习项目。
- 生成结构化驱动学习笔记。
- 按对象、生命周期、回调、资源、执行上下文和主流程分析驱动源码。
- 将真实驱动问题整理成分层工程调试笔记。
- 更新已有驱动学习文档。

## 不使用场景

不要把每一个 Linux 驱动问题都当作文档生成任务。

对于普通学习疑问：

- 直接回答问题。
- 不创建文件。
- 不更新项目笔记。
- 不更新本 Skill。

只有当用户明确要求把内容写入项目文件时，才更新项目笔记。
只有当用户明确说明某条规则要沉淀为可复用 Skill 规则时，才更新本 Skill。

## 维护边界

需要区分三类内容：

1. 普通学习问答
   - 直接回答。
   - 除非用户明确要求，否则不写入文件。

2. 项目专属学习内容
   - 只有在用户明确要求时，才写入对应项目笔记。
   - 示例：V4L2 对象关系、SDIO probe 路径、USB 枚举细节、真实板级调试日志。

3. 可复用 Skill 规则
   - 只有在用户明确要求修改通用工作流时，才更新本 Skill。
   - 示例：新增固定章节结构、新增调试模板、新增输出规则、新增嵌入式环境约束。

不要把具体驱动知识写进本 Skill，除非它已经被抽象成可复用的分析规则。

## 语言规则

默认生成的项目笔记使用中文。

代码符号、结构体名、函数名、文件路径和调用路径保持原文形式。
只有在用户明确要求英文，或者代码注释必须匹配已有英文代码风格时，才使用英文。

## 源码准确性规则

不要编造源码路径、函数名、结构体字段或调用链。

如果当前工作区没有源码，或者相关源码尚未被检查，必须明确说明“源码尚未实际检查”。

分析真实代码时：

- 先检查相关源码文件。
- 给出准确的源码路径。
- 区分“已由源码确认的事实”和“基于框架经验的推断”。

## 核心工作流

### 1. 明确项目边界

- 判断任务属于总线项目、子系统项目、板级 bring-up 项目，还是混合型项目。
- 在展开函数细节前，先确定源码入口。
- 如果项目已存在，保持原有学习目录和章节编号一致。

### 2. 建立章节骨架

- 新项目默认创建 [project-skeleton.md](references/project-skeleton.md) 中定义的 `00-09` 章节集。
- 复杂项目可根据需要增加生命周期、对象关系、回调、资源、上下文和用户态路径章节。
- 章节开头保持统一结构：`本章定位`、`核心对象`、`关键函数`、`主流程`。
- 章节正文开始前，要先说明本章按什么逻辑展开，例如按生命周期、按主流程阶段、按对象分层、按上下游边界或按工程问题路径展开，并说明为什么这样拆分。

### 3. 先建立对象模型和主流程模型

- 深入源码走读前，先阅读 [object-flow-model.md](references/object-flow-model.md)。
- 核心结构体必须写到字段层，但只展开理解当前章节所必需的字段。
- 如果某一章引入了 `01` 核心对象章里没有讲过的新私有对象、桥接对象或示例驱动私有结构，必须在该章就地补字段层说明。
- 如果对象前面章节已经讲过，不重复整套解释，但要补回链到前文对应章节。
- 进入长调用链前，先建立对象关系图。
- 生命周期、回调、资源、执行上下文、用户态到内核路径，都是驱动理解的一等内容。

### 4. 按层分析源码

- 说明每个函数在主流程中的位置，不要只罗列调用关系。
- 区分直接调用和框架回调。
- 将代码绑定到具体对象、资源和执行上下文。
- 如果任务以示例驱动为主，需要把通用框架规则映射到真实驱动文件和真实回调表。

### 5. 将源码分析整理成工程笔记

- 生成项目文件时，只使用学习笔记口吻。
- 优先记录结构体、函数、调用路径、对象关系图和故障点，避免大段粘贴代码。
- 笔记要适合后续复习和真实调试，而不是只服务于一次性解释。
- 图和代码样例不能孤立出现，必须补解释层，并和前文对象、关键函数、主流程或上一章边界明确连起来。
- 每章开头不仅要给出对象、函数和主流程，还要交代“本章为什么这样展开”，避免后续分点只有罗列而没有逻辑连续性。

### 6. 出现现象问题时使用工程调试结构

- 使用 [engineering-debug.md](references/engineering-debug.md) 中的分层调试和工程问答规则。
- 将每个现象映射到：层级、关键对象、主流程阶段、资源检查点或回调检查点。
- 对嵌入式目标板，始终考虑 BusyBox 或最小 rootfs 下的替代命令。

## 输出规则

- 生成项目文件时，使用学习笔记口吻。
- 默认生成中文笔记，除非用户要求其他语言。
- 笔记内不使用“你/我/我们”等对话式、人称化表达。
- 优先使用源码路径、函数名、结构体名、调用路径和对象关系图，少贴长代码块。
- 如果使用流程图、关系图或代码样例，必须说明图里的阶段含义、代码所在层级以及它和前文知识的对应关系。
- 如果章节正文按若干分点展开，必须先说明分点依据，例如生命周期阶段、主流程阶段、上下层边界或对象职责边界。
- 任务属于问题排查时，始终按 `现象 -> 层级 -> 对象 -> 主流程阶段 -> 资源/回调检查点` 组织。
- 除非用户明确要求创建或维护文件，否则不要写文件或修改文件。
- 除非用户明确要求更新可复用 Skill 规则，否则不要更新本 Skill。
