### 主流程

```text
probe()                                  // 驱动被匹配后开始建立框架对象
-> v4l2_device_register()                // 建立 V4L2 顶层容器
-> video_register_device() / subdev 注册  // 暴露 video node，或先把内部模块接入管线
-> open("/dev/videoX")                    // 用户态进入设备节点
-> video_ioctl2()                         // 控制类 UAPI 进入 V4L2 core
-> vb2 ioctl / vb2_ops                    // buffer 与 streaming 进入队列框架
-> media_pipeline_start() / s_stream()   // 管线型设备拉起上游模块
-> IRQ / vb2_buffer_done()                // 硬件完成并归还 buffer
-> DQBUF                                  // 用户态拿回一帧
```

本章按“能力 -> 必备知识 -> 源码达标 -> 独立问答 -> 复盘方式”展开。这样拆分，是为了避免只记住函数名，却无法判断自己是否已经能把真实现象放回正确对象层和主流程阶段。

## 1. 学完后应该达到的效果

### 1.1 对象分层能力

- 能区分 `v4l2_device`、`video_device`、`v4l2_subdev`、`vb2_queue`、`media_device` 各自解决的问题
- 能说明 `video_device`、`cdev`、`struct device` 三者在 `/dev/videoX` 暴露链路中的分工
- 能区分真实 device 字段、框架对象字段和回调表字段，不把 `vdev->fops`、`vdev->ioctl_ops`、`q->ops` 混为一类

### 1.2 注册与可见性能力

- 能从 `probe()` 画出 `/dev/videoX` 出现前的关键链路
- 能说明 sensor `subdev` probe 成功，为什么不等于 capture 节点已经出现
- 能说清 `/dev/videoX` 和 `/dev/mediaX` 分别是谁注册出来的

### 1.3 用户态到内核路径能力

- 能把 `open("/dev/videoX")` 画到 `chrdev_open()`、`v4l2_open()`、驱动 `fops->open()`
- 能把 `VIDIOC_QUERYCAP` 画到 `video_ioctl2()`、`__video_do_ioctl()`、`v4l2_ioctl_ops`
- 能把 `VIDIOC_S_CTRL` 画到 control core、sensor `s_ctrl` 和最终寄存器写入
- 能把 `REQBUFS/QBUF/STREAMON/DQBUF` 归位到 `vb2_ioctl_*`、`vb2_core_*`、`vb2_ops`

### 1.4 buffer 与完成路径能力

- 能说明 `vb2_queue` 负责什么，驱动还必须自己负责什么
- 能把 `QBUF -> 驱动私有队列 -> DMA/IRQ -> vb2_buffer_done() -> DQBUF` 串成闭环
- 能解释 `DQBUF` 卡住时，为什么要先查完成路径，而不是只看 `DQBUF` 本身

### 1.5 管线与异步绑定能力

- 能区分 `subdev`、`entity`、`pad`、`link`、`pipeline`
- 能说明 `v4l2_async_subdev` 是等待模板，不是真实 sensor 对象
- 能说明 `bound()`、`complete()`、`media_create_pad_link()`、`media_pipeline_start()` 分别解决什么问题
- 能解释 link 已 enable，为什么 pipeline 仍可能因为 `link_validate` 失败而不能起流

### 1.6 工程反推能力

- 能根据 `/dev/videoX` 缺失、`STREAMON` 失败、`DQBUF` 卡住、sensor 已 probe 但 host 不绑定、`MEDIA_IOC_SETUP_LINK` 返回 `-EBUSY` 等现象，反推出优先检查层级
- 能把问题落到 `video_device`、`v4l2_ioctl_ops`、`vb2_queue`、async notifier、media graph 或硬件完成路径，而不是一句“驱动有问题”

## 2. 必须知道的内容清单

### 2.1 核心对象清单

- `v4l2_device` 是一套 V4L2 实例的上层容器
- `video_device` 是面向用户态的设备节点封装对象
- `v4l2_subdev` 是 sensor、bridge、CSI、ISP 等内部模块抽象
- `vb2_queue` 是 buffer 生命周期和 streaming 框架对象
- `media_device / media_entity / media_pad / media_link` 是 Media Controller 图模型对象
- `v4l2_async_notifier / v4l2_async_subdev` 是异步绑定阶段的管理器和等待模板

### 2.2 注册与节点暴露清单

- `v4l2_device_register()` 只建立 V4L2 容器，不创建 `/dev/videoX`
- `video_register_device()` / `__video_register_device()` 才真正完成字符设备和设备模型注册
- `video_device->cdev` 负责字符设备分发，`video_device->dev` 负责设备模型注册
- `/dev/videoX` 对应 `video_device`，不是 sensor `subdev`
- `/dev/v4l-subdevX` 还依赖 `V4L2_SUBDEV_FL_HAS_DEVNODE` 和 `v4l2_device_register_subdev_nodes()`
- `/dev/mediaX` 由 Media Controller 侧的 `media_device_register()` 暴露

### 2.3 `open` 与 `ioctl` 清单

- `open("/dev/videoX")` 先走普通字符设备主线，再进入 V4L2 core
- `def_chr_fops`、`v4l2_fops`、`vdev->fops` 是三层不同对象
- `video_ioctl2()` 是标准 ioctl 入口，不是最终业务回调
- `v4l2_ioctl_ops` 才承载 `VIDIOC_*` 的具体业务语义
- `v4l2_ctrl_handler` 是一组 controls 的容器，`v4l2_ctrl` 是单个参数对象，`v4l2_ctrl_ops` 决定参数变化后回调谁
- `VIDIOC_S_CTRL` / `VIDIOC_S_EXT_CTRLS` 会进入 control core，再由 sensor `s_ctrl` 把曝光、增益等参数落到寄存器
- 对 `/dev/videoX` 设置 control 是否能命中 sensor，取决于 host 是否把 `sd->ctrl_handler` 合并或暴露出去
- `valid_ioctls`、锁、用户拷贝和 compat 处理都在 V4L2 core 层完成

### 2.4 `vb2` 与 streaming 清单

- `REQBUFS`、`QBUF`、`DQBUF`、`STREAMON` 常由 `vb2_ioctl_*` helper 接管
- `vb2_queue` 负责通用状态机，驱动通过 `vb2_ops` 接入硬件语义
- `queue_setup()`、`buf_prepare()`、`buf_queue()`、`start_streaming()`、`stop_streaming()` 分别处于不同阶段
- `mem_ops` 决定 buffer 怎么分配、映射和同步
- `vb2_buffer_done()` 是硬件完成路径交回框架的关键节点
- `DQBUF` 成功说明 buffer 生命周期至少闭合了一轮，不等于图像内容一定正确

### 2.5 `subdev`、Media Controller 与 host 清单

- `subdev` 是内部模块，不等于用户态 video node
- `entity/pad/link/pipeline` 是标准媒体图表达，不等于单纯的对象包含关系
- `v4l2_async_subdev` 是 host 声明“等待谁”的模板，真实设备是 `v4l2_subdev`
- `bound()` 表示真实 subdev 和等待模板匹配成功，`complete()` 表示 host 等待的外部模块到齐
- `media_create_pad_link()` 建立拓扑边，`media_pipeline_start()` 做启动前图遍历和校验
- `s_stream(1)` 负责把起流请求真正传到上游模块

### 2.6 典型源码例子清单

- `drivers/media/platform/sh_vou.c`
  - 适合理解单节点 `video_device + vb2`
- `drivers/media/i2c/imx219.c`
  - 适合理解 sensor `subdev`
- `drivers/media/platform/qcom/camss/`
  - 适合理解 host、async、media graph 和 pipeline

### 2.7 工程排查清单

- `/dev/videoX` 没出现，先查 `video_device` 注册链
- `QUERYCAP` 失败，先查 `video_ioctl2()` 到 `v4l2_ioctl_ops`
- `STREAMON` 失败，先查 buffer 数量、`start_streaming()`、pipeline 和上游 `s_stream(1)`
- `DQBUF` 卡住，先查 DMA/IRQ 和 `vb2_buffer_done()`
- sensor 已 probe 但 host 不绑定，先查 endpoint、fwnode、notifier 和 async match
- `/dev/mediaX` 有但拓扑不全，先查 entity、pad、link 和 notifier `complete()`
- `MEDIA_IOC_SETUP_LINK` 返回 `-EBUSY`，先查 active pipeline 和 `stream_count`

## 8. 参考答案

### 8.1 对 2.1 核心对象清单的参考答案

- `v4l2_device` 是上层容器，管理一套 V4L2 设备实例及其 subdev
- `video_device` 是用户态可见节点的 V4L2 封装对象
- `v4l2_subdev` 是媒体管线内部模块对象
- `vb2_queue` 是 buffer 生命周期和 streaming 框架对象
- `media_device / entity / pad / link` 是标准图模型对象
- `v4l2_async_notifier / async_subdev` 分别表示异步绑定管理器和等待模板

### 8.2 对 2.2 注册与节点暴露清单的参考答案

- `v4l2_device_register()` 只把 V4L2 容器立起来
- 真正把 video node 注册进字符设备和设备模型的是 `video_register_device()` / `__video_register_device()`
- `cdev` 解决字符设备分发，`dev` 解决设备模型注册
- `/dev/videoX` 来自 `video_device`，不是 sensor `subdev`
- `/dev/v4l-subdevX` 需要 subdev 声明可暴露，并由 host 统一注册 nodes
- `/dev/mediaX` 来自 `media_device_register()`

### 8.3 对 2.3 `open` 与 `ioctl` 清单的参考答案

- `open()` 先经过通用字符设备主线，再进入 `v4l2_open()`，最后才转发到驱动 `vdev->fops->open()`
- `def_chr_fops` 是普通字符设备默认入口，`v4l2_fops` 是 V4L2 core 的 Linux 原生 `file_operations`，`vdev->fops` 是驱动提供的 `v4l2_file_operations`
- `video_ioctl2()` 负责标准派发，真正的业务语义在 `v4l2_ioctl_ops`
- `v4l2_ctrl_handler` 管一组 controls，`v4l2_ctrl` 表示单个参数，`v4l2_ctrl_ops` 提供 `s_ctrl` 等回调
- control ioctl 进入 core 后，会先查找、校验 control，再由 sensor `s_ctrl` 把参数写到寄存器；video node 是否能命中 sensor controls，取决于 host 是否完成 handler 合并或转发
- `valid_ioctls`、用户拷贝和 compat 处理属于 V4L2 core，而不是驱动私有逻辑

### 8.4 对 2.4 `vb2` 与 streaming 清单的参考答案

- `REQBUFS/QBUF/DQBUF/STREAMON` 常先进入 `vb2_ioctl_*`
- `vb2_queue` 管状态机，驱动通过 `vb2_ops` 接住硬件语义
- `queue_setup()` 定规格，`buf_prepare()` 做入硬件前校验，`buf_queue()` 把 buffer 交给驱动，`start_streaming()` 真正拉起硬件，`stop_streaming()` 收尾
- `mem_ops` 决定 buffer 内存模型和映射方式
- `vb2_buffer_done()` 是完成路径交回 vb2 的关键点
- `DQBUF` 成功只说明 buffer 回来了，图像内容正确与否还要继续看格式、stride、DMA 和时序

### 8.5 对 2.5 `subdev`、Media Controller 与 host 清单的参考答案

- `subdev` 表示内部模块，`video_device` 才是常见用户态入口
- `entity/pad/link/pipeline` 用于表达图拓扑和起流关系
- `async_subdev` 是 host 的等待模板，真实 sensor 是 `v4l2_subdev`
- `bound()` 说明匹配成功，`complete()` 说明等待的外部模块到齐
- `media_create_pad_link()` 建边，`media_pipeline_start()` 做启动前遍历和校验
- `s_stream(1)` 把起流请求真正传到上游模块

### 8.6 对 2.6 典型源码例子清单的参考答案

- `sh_vou` 适合看单节点模型，因为 `video_device`、`vb2_queue`、open 和 streaming 都比较集中
- `imx219` 适合看 sensor `subdev`，因为它能展示 `v4l2_i2c_subdev_init()` 和 async 注册
- `camss` 适合看 host 侧，因为它同时包含 notifier、media graph、pipeline 和 video node 暴露

### 8.7 对 2.7 工程排查清单的参考答案

- 节点缺失看注册链
- 控制类 ioctl 失败看 `video_ioctl2()` 和 `v4l2_ioctl_ops`
- streaming 失败看 buffer 数量、`start_streaming()`、pipeline 和 `s_stream(1)`
- `DQBUF` 卡住看 IRQ、active buffer 和 `vb2_buffer_done()`
- host 不绑定看 endpoint、fwnode、notifier、async match
- media graph 不完整看 entity、pad、link、`complete()`
- `-EBUSY` 看活动 pipeline 和 `stream_count`

### 8.8 对第 4 节独立问题的参考答案

#### 8.8.1 `v4l2_device`、`video_device`、`v4l2_subdev`、`vb2_queue` 分别解决什么问题

`v4l2_device` 管一整套实例，`video_device` 暴露用户态节点，`v4l2_subdev` 表示内部模块，`vb2_queue` 管 buffer 生命周期和 streaming。

#### 8.8.2 `/dev/videoX` 是怎样从 `probe()` 一路注册出来的

通常先在 `probe()` 中建立私有结构体和 `v4l2_device`，再填好 `video_device`，最后调用 `video_register_device()`；其内部会分配 minor、创建 `cdev`、注册 `vdev->dev`，最终让 `/dev/videoX` 对用户态可见。

#### 8.8.3 `open("/dev/videoX")` 为什么不是直接进入驱动自己的 `.open`

因为普通字符设备首先经过 `chrdev_open()`，再把 `filp->f_op` 切到 `v4l2_fops`；之后才进入 `v4l2_open()`，最后由 V4L2 core 转发到 `vdev->fops->open()`。

#### 8.8.4 `fops`、`v4l2_fops`、`v4l2_file_operations`、`v4l2_ioctl_ops` 分别是什么关系

`v4l2_fops` 是挂在 `cdev->ops` 上的 Linux 原生 `file_operations`；`v4l2_file_operations` 是驱动注册给 `video_device` 的 V4L2 文件操作表；`v4l2_ioctl_ops` 是具体 `VIDIOC_*` 回调表。前两层负责文件入口，后一层负责业务命令。

#### 8.8.5 `video_ioctl2()` 和驱动自己的 `vidioc_*` 回调之间隔着哪些层

常见路径是 `video_ioctl2() -> video_usercopy() -> __video_do_ioctl() -> vdev->ioctl_ops->vidioc_*`。中间由 V4L2 core 处理参数拷贝、合法性和分发。

#### 8.8.6 用户态设置曝光时，从 `VIDIOC_S_CTRL` 到 sensor 寄存器会经过哪些层

常见直接 subdev 路径是 `VIDIOC_S_CTRL -> subdev_ioctl -> v4l2_s_ctrl() -> control core 校验 -> sensor ctrl_ops.s_ctrl -> sensor write_reg -> I2C`。如果从 `/dev/videoX` 进入，还要先满足 host 已把 sensor controls 合并或暴露到 video node。

#### 8.8.7 `REQBUFS/QBUF/STREAMON/DQBUF` 分别落到 `vb2` 的哪一层

通常先到 `vb2_ioctl_*` helper，再进入 `vb2_core_*`，最后在需要驱动参与的阶段回调 `q->ops` 中的 `queue_setup()`、`buf_prepare()`、`buf_queue()`、`start_streaming()` 或 `stop_streaming()`。

#### 8.8.8 `DQBUF` 卡住时，为什么要优先看 IRQ 和 `vb2_buffer_done()`

因为 `DQBUF` 等的是已经完成的 buffer。没有硬件完成事件、没有 active buffer、或者驱动没有调用 `vb2_buffer_done()`，`done_list` 就不会有可返回的帧。

#### 8.8.9 sensor 已经 probe，为什么 `/dev/videoX` 仍可能没有出现

sensor probe 成功只说明内部 `subdev` 存在。capture 节点通常由 host 驱动注册；如果 host 没 probe、async 没绑定、或者 host 尚未完成 video node 注册，`/dev/videoX` 仍不会出现。

#### 8.8.10 `v4l2_async_subdev` 和 `v4l2_subdev` 为什么不能混为一个对象

`v4l2_async_subdev` 是 host 声明等待对象时使用的匹配模板，`v4l2_subdev` 才是真实 sensor/bridge 驱动注册出来的运行时对象。二者只有匹配成功后才建立关系。

#### 8.8.11 `entity`、`pad`、`link`、`pipeline` 各自解决什么问题

`entity` 表示图中的节点，`pad` 表示节点端口，`link` 表示端口之间的连接边，`pipeline` 表示一次实际起流时被激活的一组路径和状态。

#### 8.8.12 link 已经 enable，为什么 `STREAMON` 仍可能失败

enable 只说明拓扑开关打开；起流前还要经过 `media_pipeline_start()` 和 `link_validate`，若两端 pad format、mbus code、尺寸或必连关系不满足，仍会失败。

#### 8.8.13 `STREAMOFF` 后再次 `STREAMON` 失败，优先怀疑哪些收尾动作没有完成

优先怀疑 DMA 没停、上游 `s_stream(0)` 没下发、active buffer 没归还、驱动私有队列没清、`media_pipeline_stop()` 没走、硬件 pending 状态没清或 runtime PM 引用不平衡。
