- 把 `video_device`、`vb2`、`subdev`、Media Controller、host pipeline 这几层问题拆开定位
- 建立一套稳定的驱动阅读顺序
- 把常见故障现象映射回正确的对象层和函数层

## 1. 先说最实用的阅读顺序

遇到一个新的 V4L2 驱动时，不必一上来就从头读到尾。  
较稳的顺序如下：

1. 先找 `probe()`
2. 看它是注册 `video_device` 还是 `subdev`
3. 找 `v4l2_file_operations`
4. 找 `v4l2_ioctl_ops` 或 `v4l2_subdev_ops`
5. 找 `vb2_queue` 初始化
6. 找 `start_streaming/stop_streaming`
7. 找中断里有没有 `vb2_buffer_done()`

## 2. 判断驱动属于哪一类

### 2.1 直接视频节点型

出现下面这些特征时：

- `video_register_device()`
- `vb2_queue_init()`

那它多半是直接面向用户态的 `/dev/videoX` 驱动。

### 2.2 管线组件型

出现下面这些特征时：

- `v4l2_i2c_subdev_init()`
- `v4l2_async_register_subdev()`

那它多半是 sensor/bridge/ISP 子模块。

## 3. 调试 `video_device` 驱动时重点打哪些点

推荐优先观察这些点：

1. `probe()` 是否跑完
2. `video_register_device()` 是否成功
3. `open()` 是否进到了驱动回调
4. `VIDIOC_QUERYCAP` / `VIDIOC_S_FMT` 是否进了对应 `vidioc_*`
5. `REQBUFS/QBUF/STREAMON` 是否进了 `vb2_ops`
6. 中断里是否调用了 `vb2_buffer_done()`
7. `STREAMOFF` 时是否清空了驱动私有队列

## 4. 调试 `subdev` 驱动时重点看什么

推荐优先看：

1. `v4l2_i2c_subdev_init()` 是否执行
2. fwnode / DT endpoint 是否解析成功
3. `v4l2_async_register_subdev()` 是否成功
4. host notifier 的 `bound()` / `complete()` 是否触发
5. `s_stream(1)` 是否真的进到 sensor

## 5. 很多问题其实是边界没分清

### 5.1 `/dev/videoX` 没出来

优先怀疑：

- host/capture 驱动没注册 `video_device`
- 不是先怀疑 sensor 驱动

### 5.2 `DQBUF` 一直卡住

优先怀疑：

- 驱动没有 `vb2_buffer_done()`
- `start_streaming()` 没真正启动硬件
- 中断没来

### 5.3 `STREAMON` 失败

优先怀疑：

- `min_buffers_needed` 不满足
- `start_streaming()` 返回错误
- 下游/上游 `s_stream(1)` 调用失败

### 5.4 sensor 已 probe 但管线没起来

优先怀疑：

- DT endpoint 不匹配
- async notifier 没匹配上
- media pads / entity 配置不完整

## 6. V4L2 常用调试命令思路

如果系统里有 `v4l2-ctl`，最常用的是：

```bash
v4l2-ctl --all -d /dev/video0
v4l2-ctl --list-formats-ext -d /dev/video0
v4l2-ctl --stream-mmap --stream-count=4 -d /dev/video0
```

如果是 subdev 节点，有时会配合：

```bash
media-ctl -p
```

来先看媒体拓扑。

## 7. 看日志时要把“框架层”和“驱动层”分开

很多新手容易把所有报错都算到驱动头上。  
实际上 V4L2 问题常常分三层：

### 7.1 V4L2 core 层

例如：

- ioctl 不合法
- 设备未注册
- 锁冲突

### 7.2 vb2 层

例如：

- buffer 大小不匹配
- 队列状态机错误
- streaming 状态错误

### 7.3 硬件驱动层

例如：

- DMA 没启动
- sensor 没出流
- 中断没到

分层定位会快很多。

## 8. 实际读源码时的固定问题模板

每看一个函数，都问自己这四个问题：

1. 这个函数是框架层还是驱动层
2. 它操作的是 `video_device`、`subdev` 还是 `vb2_queue`
3. 它是“注册路径”、`ioctl` 路径，还是 “streaming 路径”
4. 它失败以后会把状态回滚到哪里

## 9. 这套笔记的收束方式

前面的主线章节读完后，下一步可做两件事：

1. 拿一个真正关心的 capture/codec 驱动，按这套框架再手拆一遍
2. 对照启动日志或 `v4l2-ctl` 行为，把 `probe -> open -> ioctl -> streamon -> irq -> dqbuf` 串起来
3. 对照 [[13-V4L2实际工程问题问答]]，把具体故障现象放回对应对象层和函数层

## 10. 最后一句

V4L2 看起来复杂，主要是因为它同时覆盖：

- 设备节点层
- 媒体拓扑层
- 缓冲队列层

但只要把这三层分开读，再把 `video_device` 和 `subdev` 这两类驱动模型分开看，源码就会一下清楚很多。

## 11. 回答的问题

- 遇到一个新驱动时，阅读顺序应该怎样安排
- `/dev/videoX`、`vb2`、`subdev`、`/dev/mediaX` 相关问题分别该先看哪一层
- `DQBUF` 卡住、`STREAMON` 失败、sensor 已 probe 但管线没起来，这几类问题最常见的根因是什么
- 把对象层、函数层和日志层串起来时，最稳的调试抓手是什么
