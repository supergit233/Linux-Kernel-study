# V4L2 实际工程问题问答

## 学习目标

- 把真实工程里的 V4L2 故障现象映射到对应对象层和函数层
- 建立从用户态现象反推驱动内部路径的排查方法
- 区分 `video_device`、`vb2`、`subdev`、Media Controller、硬件中断完成路径各自负责的问题
- 形成一套能直接用于驱动调试的问答清单

## 导读

### 本章定位

这一章是整套 V4L2 笔记的工程问答章。前面章节已经分别建立了 `open/ioctl/vb2/subdev/media graph/host pipeline` 的理论主线，本章把这些主线重新组合成实际问题的排查入口。

### 核心对象

- `struct video_device`
  - `/dev/videoX` 是否出现、open 是否能进入 V4L2 框架
- `struct v4l2_ioctl_ops`
  - `VIDIOC_*` 是否被正确派发到驱动业务回调
- `struct vb2_queue`
  - buffer 申请、入队、出队、streaming 状态是否正确
- `struct vb2_ops`
  - 驱动如何接入 `queue_setup / buf_prepare / buf_queue / start_streaming / stop_streaming`
- `struct v4l2_subdev`
  - sensor/bridge/CSI/ISP 模块是否被 host 识别并参与起流
- `struct v4l2_async_notifier / v4l2_async_subdev`
  - 外部 subdev 是否和 host 匹配成功
- `struct media_entity / media_pad / media_link`
  - 拓扑是否完整、link 是否 enable、pipeline 是否能启动

### 关键函数

- `video_register_device()`
- `v4l2_open()`
- `video_ioctl2()`
- `__video_do_ioctl()`
- `vb2_ioctl_reqbufs() / vb2_ioctl_qbuf() / vb2_ioctl_dqbuf() / vb2_ioctl_streamon()`
- `vb2_core_reqbufs() / vb2_core_qbuf() / vb2_core_streamon() / vb2_core_dqbuf()`
- `v4l2_async_register_subdev()`
- `v4l2_async_notifier_register()`
- notifier `bound() / complete()`
- `media_create_pad_link()`
- `media_pipeline_start()`
- `link_validate`
- `v4l2_subdev_call(..., s_stream, 1)`
- `vb2_buffer_done()`

### 主流程

从现象出发 -> 定位对象层 -> 找到对应主链 -> 检查驱动回调是否进入 -> 检查状态是否回滚 -> 检查硬件是否真正完成。

### 检查项写法

本章的检查点统一按下面的方式记录：

```text
检查项：说明为什么这个点会导致当前现象；给出相关调用链。
```

这样每个问题都能从用户态现象落回驱动内部路径，而不是只保留一个孤立清单。

## 1. 工程排查的总原则

V4L2 问题不要直接从“硬件坏了”开始判断。更稳定的顺序是按层拆：

1. 节点是否注册出来
2. open 是否进入 V4L2 core 和驱动 `fops`
3. ioctl 是否进入 `v4l2_ioctl_ops`
4. buffer 类 ioctl 是否进入 `vb2`
5. `vb2` 是否回调到驱动 `vb2_ops`
6. `STREAMON` 是否拉起 media pipeline
7. 上游 subdev 是否收到 `s_stream(1)`
8. 硬件是否产生中断或完成回调
9. 驱动是否调用 `vb2_buffer_done()`
10. 用户态是否能 `DQBUF` 拿回完成 buffer

完整排查主链可以压成：

```text
/dev/videoX
-> open
-> ioctl
-> v4l2_ioctl_ops
-> vb2_ioctl_*
-> vb2_core_*
-> vb2_ops
-> media_pipeline_start
-> subdev s_stream
-> DMA / IRQ
-> vb2_buffer_done
-> DQBUF
```

## 2. `/dev/videoX` 没有出现

### 现象

- 系统里没有 `/dev/video0`
- `v4l2-ctl --list-devices` 看不到目标设备
- sensor 日志存在，但 capture 节点不存在

### 定位层

优先定位 `video_device` 注册路径，而不是直接定位 sensor。

### 驱动检查点

- `probe()` 是否执行到结尾：`video_device` 注册通常在 probe 后半段完成，probe 中途失败会直接导致 `/dev/videoX` 不出现；相关链路：`driver probe -> v4l2_device_register -> video_register_device`。
- `v4l2_device_register()` 是否成功：`video_device` 需要挂到一个有效的 `v4l2_device` 管理对象下，前置容器没建立时后续注册没有可靠归属；相关链路：`probe -> v4l2_device_register -> vdev->v4l2_dev`。
- `vdev->v4l2_dev` 是否填好：`video_register_device()` 需要通过它找到 V4L2 管理对象，缺失时节点即使创建也无法形成正确 V4L2 归属；相关链路：`probe -> vdev->v4l2_dev -> video_register_device`。
- `vdev->fops` 是否填好：open 之后 `v4l2_open()` 会转发到 `vdev->fops->open`，缺失会导致节点可见但无法进入驱动文件操作；相关链路：`open -> v4l2_open -> vdev->fops->open`。
- `vdev->ioctl_ops` 是否填好：`video_ioctl2()` 最终要通过它找到 `vidioc_*`，缺失会导致能力查询和格式设置类 ioctl 无法分发；相关链路：`ioctl -> video_ioctl2 -> __video_do_ioctl -> vdev->ioctl_ops`。
- `vdev->release` 是否填好：`video_device` 引用释放依赖该回调，生命周期错误会造成节点注册后释放异常或悬空访问；相关链路：`video_register_device -> video_device refcount -> vdev->release`。
- `vdev->device_caps` 是否填好：用户态依赖它判断节点能力，能力缺失会让应用认为该节点不可用；相关链路：`VIDIOC_QUERYCAP -> vidioc_querycap -> device_caps`。
- `video_register_device()` 是否返回错误：它是真正创建 V4L2 字符设备节点的入口，失败会直接阻断 `/dev/videoX`；相关链路：`probe -> video_register_device -> __video_register_device`。
- `__video_register_device()` 是否完成 minor 分配：minor 决定 `/dev/videoX` 的编号，分配失败时节点无法落到字符设备编号空间；相关链路：`__video_register_device -> minor allocation`。
- `__video_register_device()` 是否完成 `cdev_add()`：`cdev_add()` 决定 `inode->i_rdev` 能否找到 V4L2 的 `cdev`，缺失会导致字符设备 open 分发断掉；相关链路：`open -> chrdev_open -> cdev lookup -> v4l2_fops`。
- `__video_register_device()` 是否完成 `device_register(&vdev->dev)`：设备模型注册负责让 devtmpfs/sysfs 看到该设备，缺失会导致用户态看不到 `/dev/videoX`；相关链路：`__video_register_device -> device_register -> /dev/videoX`。

### 常见根因

- capture/host 驱动没有 probe
- `video_register_device()` 前字段没填完整
- `v4l2_device_register()` 失败后仍继续执行
- `video_device` 生命周期错误，过早释放
- 误以为 sensor 驱动会自己注册 `/dev/videoX`

### 核心判断

`/dev/videoX` 是 `video_device` 注册出来的。sensor `subdev` probe 成功，只说明上游模块存在，不代表 capture 节点一定已经暴露。

## 3. `/dev/videoX` 存在，但 open 后没有进入驱动 open

### 现象

- `/dev/videoX` 存在
- 用户态 `open()` 成功或失败
- 驱动自己的 `.open` 日志没有出现

### 定位层

重点区分三层 `fops`：

```text
def_chr_fops
-> v4l2_fops
-> vdev->fops
```

### 驱动检查点

- `vdev->cdev->ops` 是否是 `v4l2_fops`：字符设备 open 后会把 `filp->f_op` 替换成 `cdev->ops`，这里不对就进不了 V4L2 core；相关链路：`def_chr_fops -> chrdev_open -> replace_fops -> v4l2_fops`。
- `v4l2_open()` 是否执行：它是 V4L2 层 open 总入口，没执行说明问题还停在字符设备分发层；相关链路：`open -> chrdev_open -> v4l2_fops.open -> v4l2_open`。
- `video_devdata(file)` 是否能拿到正确 `vdev`：后续转发依赖 `vdev`，拿错实例会导致日志出现在别的节点或直接失败；相关链路：`v4l2_open -> video_devdata(file) -> vdev`。
- `vdev->fops` 是否为空：`v4l2_open()` 不直接进入驱动私有函数，而是通过 `vdev->fops` 转发；相关链路：`v4l2_open -> vdev->fops`。
- `vdev->fops->open` 是否为空：该回调为空时，open 只完成 V4L2 公共层，不会进入驱动私有 open；相关链路：`v4l2_open -> vdev->fops->open`。
- `V4L2_FL_REGISTERED` 状态是否正常：节点注销或注册未完成时，V4L2 core 会拒绝继续访问，表现为 open 没进驱动；相关链路：`v4l2_open -> test_bit(V4L2_FL_REGISTERED)`。

### 常见根因

- 只检查了驱动私有 `.open`，没有确认是否先进入 `v4l2_open()`
- `vdev->fops` 没填
- 节点已注销但文件仍被访问
- 多实例驱动中 minor 对应到了另一个 `video_device`

### 核心判断

字符设备层只把 `def_chr_fops` 覆盖为 `v4l2_fops`。`v4l2_open()` 不会再覆盖成驱动私有 `fops`，只会转发到 `vdev->fops->open`。

## 4. `VIDIOC_QUERYCAP` 失败

### 现象

- `v4l2-ctl --all -d /dev/videoX` 报错
- 应用第一步查询能力失败
- 驱动 probe 和 open 都正常

### 定位层

重点看 `ioctl` 派发是否进入 `v4l2_ioctl_ops`。

### 驱动检查点

调用链应当是：

```text
v4l2_fops.unlocked_ioctl
-> v4l2_ioctl()
-> vdev->fops->unlocked_ioctl
-> video_ioctl2()
-> video_usercopy()
-> __video_do_ioctl()
-> vdev->ioctl_ops->vidioc_querycap
```

检查点：

- `vdev->fops->unlocked_ioctl` 是否指向 `video_ioctl2`：V4L2 标准 ioctl 派发依赖这个入口，未设置时不会进入 `video_usercopy()` 和 `__video_do_ioctl()`；相关链路：`v4l2_ioctl -> vdev->fops->unlocked_ioctl -> video_ioctl2`。
- `vdev->ioctl_ops` 是否为空：`__video_do_ioctl()` 最终靠它找到具体 `vidioc_*` 回调，缺失会导致业务命令无处派发；相关链路：`video_ioctl2 -> __video_do_ioctl -> vdev->ioctl_ops`。
- `.vidioc_querycap` 是否实现：`QUERYCAP` 的最终处理函数就是它，缺失时能力查询无法返回驱动能力；相关链路：`VIDIOC_QUERYCAP -> v4l2_ioctl_ops->vidioc_querycap`。
- `device_caps` 是否设置合理：应用会用 capability 判断节点类型和能力，设置错误会表现为应用拒绝继续使用；相关链路：`vidioc_querycap -> vdev->device_caps -> userspace capability check`。
- `valid_ioctls` 中该命令是否有效：V4L2 core 会先根据有效 ioctl 位图过滤命令，命令无效时不会进入驱动回调；相关链路：`__video_do_ioctl -> valid_ioctls -> vidioc_querycap`。

### 常见根因

- 忘记填 `vdev->ioctl_ops`
- 驱动没有实现 `.vidioc_querycap`
- `vdev->device_caps` 和返回能力不一致
- 把 `video_ioctl2()` 当成最终业务函数，忽略了后面的 `v4l2_ioctl_ops`

### 核心判断

`QUERYCAP` 是最适合验证 ioctl 主链是否打通的命令。它不涉及 `vb2`，失败时优先看 `v4l2_ioctl_ops` 和 `video_ioctl2()`。

## 5. `S_FMT` 成功，但实际采集格式不对

### 现象

- `VIDIOC_S_FMT` 返回成功
- `G_FMT` 看起来也正确
- 实际采集出来的图像宽高、颜色、stride 不符合预期

### 定位层

分清 video node format 和 subdev pad format。

### 驱动检查点

video node 侧：

- `.vidioc_try_fmt_*` 是否修正格式：用户态传入的格式可能不被硬件支持，未修正会让后续 buffer 大小和硬件实际格式不一致；相关链路：`VIDIOC_S_FMT -> try_fmt -> s_fmt`。
- `.vidioc_s_fmt_*` 是否保存到驱动私有格式结构：硬件配置和 `queue_setup()` 通常读取驱动私有格式，未保存会导致后续仍按旧格式运行；相关链路：`S_FMT -> driver current_fmt -> queue_setup/start_streaming`。
- `bytesperline` 和 `sizeimage` 是否重新计算：vb2 分配 buffer 依赖 `sizeimage`，DMA 写入依赖 stride，旧值会造成花屏或越界；相关链路：`S_FMT -> sizeimage -> REQBUFS -> DMA`。
- 已经分配 buffer 后是否禁止修改格式：buffer 已按旧格式分配，再改格式会造成 `QBUF/DQBUF` 阶段尺寸不匹配；相关链路：`REQBUFS -> buffers allocated -> S_FMT`。

subdev 侧：

- `set_fmt()` 是否设置的是 `ACTIVE`：真正起流使用 `ACTIVE` format，只有 `TRY` 正确不会改变硬件运行格式；相关链路：`set_fmt(TRY/ACTIVE) -> media_pipeline_start -> s_stream`。
- 上游 source pad 和下游 sink pad 的 `mbus code` 是否一致：link 两端 code 不一致时，接收端会按错误格式解析数据；相关链路：`sensor source pad -> CSI sink pad -> link_validate`。
- 宽高是否逐级匹配：尺寸不一致会让 CSI/ISP/DMA 对一帧边界判断错误，表现为黑屏、裁切或花屏；相关链路：`subdev set_fmt -> pad formats -> link_validate -> DMA`。
- crop/compose 是否改变了实际输出尺寸：内部裁剪或缩放会改变后级看到的有效尺寸，只查 video node format 容易漏掉；相关链路：`selection -> subdev source pad fmt -> video node fmt`。

### 常见根因

- video node 格式改了，但 subdev pad format 没同步
- `TRY` format 正确，`ACTIVE` format 没更新
- `sizeimage` 没按新格式重新计算
- DMA stride 使用旧值
- sensor mode 切换后，host 侧仍使用旧时序

### 核心判断

`VIDIOC_S_FMT` 只证明 video node 接受了用户态格式，不等于整条媒体管线的 pad format 已经一致。

## 6. sensor 已经 probe，但 host 的 `bound()` 没触发

### 现象

- I2C sensor probe 日志正常
- `v4l2_async_register_subdev()` 返回成功
- host notifier 的 `bound()` 没有日志
- `/dev/mediaX` 拓扑缺少 sensor

### 定位层

重点看 async 匹配层：

```text
真实 subdev: sd
匹配模板: asd
管理器: notifier
```

### 驱动检查点

sensor 侧：

- `sd->dev` 是否正确：async 框架常从 `sd->dev` 补 `sd->fwnode`，底层设备错误会导致匹配键缺失；相关链路：`v4l2_i2c_subdev_init -> sd->dev -> dev_fwnode`。
- `sd->fwnode` 是否能从 `sd->dev` 得到：fwnode 是 host 等待项最常见的匹配键，缺失时 `find_match()` 无法命中；相关链路：`v4l2_async_register_subdev -> sd->fwnode -> v4l2_async_find_match`。
- 是否调用了 `v4l2_async_register_subdev()` 或 sensor common helper：没有进入 async 框架时，host notifier 没有机会看到这个 sensor；相关链路：`sensor probe -> v4l2_async_register_subdev -> subdev_list/notifier match`。

host 侧：

- `v4l2_async_notifier_init()` 是否执行：notifier 未初始化时等待集合和回调入口都不可用；相关链路：`host probe -> notifier_init -> add async_subdev`。
- 是否通过 fwnode/endpoint 添加了 `v4l2_async_subdev`：`asd` 是 host 声明“正在等谁”的模板，没有它就没有匹配目标；相关链路：`parse endpoint -> add_fwnode_subdev -> notifier waiting list`。
- `v4l2_async_notifier_register()` 是否执行：notifier 只有注册后才会进入全局匹配流程；相关链路：`host probe -> notifier_register -> v4l2_async_find_match`。
- host 的 DT endpoint 是否能找到 remote sensor：host 通常根据 remote endpoint 生成 `asd`，解析失败会等错对象或不等待；相关链路：`of_graph endpoint -> remote fwnode -> async_subdev match key`。

匹配点：

- `v4l2_async_find_match(notifier, sd)` 是否能返回 `asd`：这是 `sd` 和 host 等待模板真正对上的位置，返回空就不会进入 `bound()`；相关链路：`sd + notifier waiting asd -> find_match -> asd`。
- `v4l2_async_match_notify()` 是否执行：该函数会先注册 subdev 到 host，再触发 `bound()`，没执行说明匹配后续链没走起来；相关链路：`find_match -> match_notify -> v4l2_device_register_subdev -> bound`。

### 常见根因

- DT endpoint 的 remote-endpoint 写错
- sensor 节点和 host port 没有互相引用
- host 解析的是本地 endpoint，但等待项使用了错误 fwnode
- sensor 没有 `sd->fwnode`
- host notifier 注册早，但没有添加等待项

### 核心判断

sensor probe 成功只说明真实模块存在。`bound()` 不触发，说明 host 的 `async_subdev` 匹配模板没有和这个 `sd` 对上。

## 7. `/dev/mediaX` 存在，但拓扑不完整

### 现象

- `/dev/mediaX` 存在
- `media-ctl -p` 能看到部分 entity
- sensor、CSI 或 video node 缺失
- link 没有出现

### 定位层

重点看 Media Controller 建图阶段。

### 驱动检查点

- `media_device_init()` 是否执行：`media_device` 是整张媒体图容器，未初始化会导致后续 entity/link 没有全局归属；相关链路：`host probe -> media_device_init -> media graph container`。
- `v4l2_dev.mdev` 是否指向 `media_device`：V4L2 对象需要通过该指针接入 MC 图，缺失会导致 V4L2 entity 无法进入同一张图；相关链路：`v4l2_device -> mdev -> media_device`。
- `media_entity_pads_init()` 是否为每个 entity 初始化 pads：link 连接的是 pad 不是 entity 本体，pad 缺失时无法建立拓扑边；相关链路：`entity -> pads -> media_create_pad_link`。
- `media_device_register_entity()` 是否成功：entity 注册失败时，即使模块存在，也不会出现在 `/dev/mediaX` 拓扑里；相关链路：`subdev/video entity -> media_device_register_entity -> media-ctl -p`。
- `media_create_pad_link()` 是否被调用：外部 sensor 和 host 内部模块之间靠它建立 pad-to-pad 连接，缺失会造成图断开；相关链路：`complete -> media_create_pad_link -> sensor pad -> csiphy pad`。
- notifier `complete()` 是否执行到建图代码：很多 host 在所有外部 subdev 到齐后才建外部 link，`complete()` 不走就没有完整拓扑；相关链路：`bound all -> try_complete -> complete -> create links`。
- `media_device_register()` 是否在建图后调用：用户态看到的 `/dev/mediaX` 应该在拓扑准备好后发布，过早注册会暴露不完整图；相关链路：`create entities/links -> media_device_register -> /dev/mediaX`。

### 常见根因

- entity 有了，但 pad 没初始化
- `media_create_pad_link()` 参数里的 pad index 错
- host 的 `complete()` 没触发，因此外部 sensor link 没建
- 内部模块注册失败后仍继续注册 media device
- `v4l2_dev.mdev` 没挂，导致 V4L2 对象没有进入 MC 图

### 核心判断

`media_device` 注册出来只说明 `/dev/mediaX` 存在，不代表 entity/pad/link 全部完整。完整拓扑依赖每个模块的 entity 初始化和 link 建立。

## 8. `MEDIA_IOC_SETUP_LINK` 返回 `-EBUSY`

### 现象

- `media-ctl` 修改 link 时失败
- 返回 `Device or resource busy`
- 停止应用后同样命令又可能成功

### 定位层

重点看 `__media_entity_setup_link()` 和 `stream_count`。

### 驱动检查点

- 当前 link 是否带 `MEDIA_LNK_FL_DYNAMIC`：非 dynamic link 在活动 pipeline 中不能修改，缺失该标志会直接触发 busy 保护；相关链路：`MEDIA_IOC_SETUP_LINK -> __media_entity_setup_link -> MEDIA_LNK_FL_DYNAMIC`。
- link 两端 entity 的 `stream_count` 是否大于 0：`stream_count` 表示 entity 已被活动 pipeline 占用，占用期间修改非 dynamic link 会返回 `-EBUSY`；相关链路：`media_pipeline_start -> stream_count++ -> setup_link`。
- 是否有应用正在 streaming：正在采集的 fd 会维持 pipeline 活动，link 改动会影响运行中的数据路径；相关链路：`STREAMON -> media_pipeline_start -> MEDIA_IOC_SETUP_LINK`。
- 是否有文件句柄没有关闭：未关闭的 fd 可能仍持有队列或 pipeline 状态，导致资源看起来已经停用但状态未释放；相关链路：`open -> STREAMON -> fd lifetime -> release`。
- pipeline 是否没有正确 `media_pipeline_stop()`：`stream_count` 需要在 stop 时回退，漏掉 stop 会让后续 link 修改持续看到 busy；相关链路：`STREAMOFF -> stop_streaming -> media_pipeline_stop -> stream_count--`。

### 常见根因

- 非 dynamic link 不能在运行中的 pipeline 里修改
- 上一次 `STREAMOFF` 没有完整走到 `media_pipeline_stop()`
- 驱动错误地提前 `stream_count++`，失败路径没有回滚
- 多路 pipeline 共享同一 entity

### 核心判断

`-EBUSY` 不是 link 建立失败，而是当前 entity 正在活动 pipeline 中，非动态 link 不能运行时切换。

## 9. `STREAMON` 失败

### 现象

- `REQBUFS/QBUF` 成功
- `VIDIOC_STREAMON` 返回错误
- 没有进入持续采集状态

### 定位层

重点看 `vb2_core_streamon()`、驱动 `start_streaming()`、`media_pipeline_start()`。

### 驱动检查点

调用链通常是：

```text
VIDIOC_STREAMON
-> vb2_ioctl_streamon()
-> vb2_core_streamon()
-> q->ops->start_streaming()
-> media_pipeline_start()
-> v4l2_subdev_call(..., s_stream, 1)
```

检查点：

- queued buffer 数是否满足 `min_buffers_needed`：vb2 会在启动前检查最小 buffer 数，不满足时不会进入真正硬件启动；相关链路：`VIDIOC_STREAMON -> vb2_core_streamon -> min_buffers_needed`。
- `start_streaming()` 是否被调用：这是 vb2 转入驱动启动硬件的入口，没进入说明问题还在 vb2 状态机或 buffer 条件；相关链路：`vb2_core_streamon -> q->ops->start_streaming`。
- `media_pipeline_start()` 是否失败：host 驱动通常在 `start_streaming()` 里先启动 pipeline，失败会阻止后续 subdev 起流；相关链路：`start_streaming -> media_pipeline_start -> graph walk`。
- `link_validate` 是否失败：format、尺寸或连接关系不合法时，pipeline 启动前会被拦住；相关链路：`media_pipeline_start -> link_validate -> pad formats`。
- 上游 `s_stream(1)` 哪一级返回错误：sensor/CSI/ISP 任一级起流失败都会让整条采集链启动失败；相关链路：`media_pipeline_start ok -> v4l2_subdev_call(..., s_stream, 1)`。
- `start_streaming()` 失败后是否把已交给驱动的 buffer 退回 `vb2`：启动失败时 buffer 已经可能进入驱动队列，不退回会导致后续 `DQBUF/STREAMOFF` 状态异常；相关链路：`start_streaming error -> vb2_buffer_done(..., ERROR) -> queue rollback`。

### 常见根因

- buffer 数不够
- pad format 不匹配
- link 没 enable
- sensor `s_stream(1)` 失败
- DMA 初始化失败
- 时钟、复位、电源没有准备好
- `start_streaming()` 失败路径没有 `vb2_buffer_done(..., ERROR)`

### 核心判断

`STREAMON` 是 video node、vb2、MC pipeline、subdev、硬件启动第一次真正合流的位置。

## 10. `REQBUFS` 失败或申请到的 buffer 数不对

### 现象

- `VIDIOC_REQBUFS` 返回错误
- 用户申请 4 个 buffer，实际只得到更少数量
- 后续 `QUERYBUF` 或 `mmap` 失败

### 定位层

重点看 `vb2_core_reqbufs()` 和 `queue_setup()`。

### 驱动检查点

调用链通常是：

```text
VIDIOC_REQBUFS
-> vb2_ioctl_reqbufs()
-> vb2_core_reqbufs()
-> call_qop(q, queue_setup, ...)
-> q->mem_ops->alloc()
```

检查点：

- `vb2_queue_init()` 是否成功：队列初始化会校验 `q->ops / q->mem_ops / type / io_modes` 等基础字段，失败后所有 buffer ioctl 都没有可靠基础；相关链路：`probe -> vb2_queue_init -> VIDIOC_REQBUFS`。
- `q->type` 是否和用户请求匹配：用户传入的 buffer type 必须和队列类型一致，不一致会被 vb2 拒绝；相关链路：`VIDIOC_REQBUFS -> vb2_ioctl_reqbufs -> q->type check`。
- `q->io_modes` 是否包含当前 memory 类型：MMAP/USERPTR/DMABUF 必须由队列声明支持，否则 REQBUFS 会失败；相关链路：`VIDIOC_REQBUFS(memory) -> q->io_modes`。
- `q->ops` 是否设置：`REQBUFS` 需要通过 `queue_setup()` 向驱动询问 buffer 规格，缺失会导致 vb2 无法进入驱动策略；相关链路：`vb2_core_reqbufs -> call_qop(queue_setup)`。
- `q->mem_ops` 是否设置：buffer 背后的内存分配依赖 memory backend，缺失会导致无法分配或映射 buffer；相关链路：`vb2_core_reqbufs -> q->mem_ops->alloc`。
- `queue_setup()` 是否正确设置 `num_planes` 和 `sizes[]`：vb2 按这些值创建 plane 和分配内存，错误会导致申请数量、大小或后续 mmap 异常；相关链路：`REQBUFS -> queue_setup -> __vb2_queue_alloc`。
- CMA 或 DMA 内存是否不足：使用连续 DMA 内存时，分配失败会直接表现为 REQBUFS 返回错误或 buffer 数减少；相关链路：`q->mem_ops->alloc -> dma_alloc_attrs/CMA`。

### 常见根因

- `sizeimage` 为 0 或过小
- `queue_setup()` 没按当前格式计算 buffer 大小
- `mem_ops` 选错，例如硬件需要 DMA 却使用了不合适的内存后端
- CMA 不足导致 `vb2_dma_contig_memops` 分配失败
- streaming 状态下重复 `REQBUFS`

### 核心判断

`REQBUFS` 阶段主要是在谈 buffer 规格和内存后端，还没有真正启动硬件。

## 11. `QBUF` 失败

### 现象

- `REQBUFS/QUERYBUF/mmap` 成功
- `VIDIOC_QBUF` 返回错误
- buffer 无法进入采集队列

### 定位层

重点看 `vb2_core_qbuf()`、`buf_prepare()`、`buf_queue()`。

### 驱动检查点

调用链通常是：

```text
VIDIOC_QBUF
-> vb2_ioctl_qbuf()
-> vb2_core_qbuf()
-> q->ops->buf_prepare()
-> q->ops->buf_queue()
```

检查点：

- buffer index 是否有效：vb2 只能操作 `REQBUFS` 已创建的 buffer，index 越界会在进入驱动前被拒绝；相关链路：`VIDIOC_QBUF -> vb2_core_qbuf -> buffer index check`。
- memory 类型是否一致：队列创建时确定了 memory 类型，QBUF 使用不同类型会破坏 buffer 管理模型；相关链路：`REQBUFS(memory) -> q->memory -> QBUF(memory)`。
- plane length 是否大于等于 `sizeimage`：硬件写入一帧需要足够空间，长度不足会导致越界风险或被 vb2/驱动拒绝；相关链路：`S_FMT -> sizeimage -> QBUF plane length`。
- `buf_prepare()` 是否检查格式和长度：该回调是驱动进入硬件队列前的最后软件校验，漏检会把错误 buffer 送入硬件；相关链路：`vb2_core_qbuf -> q->ops->buf_prepare`。
- `buf_queue()` 是否把 buffer 放进驱动私有队列：vb2 状态变为 queued 后，还需要驱动接管 buffer，否则硬件没有可用帧地址；相关链路：`vb2_core_qbuf -> q->ops->buf_queue -> driver queue`。
- streaming 已经开始时，`buf_queue()` 是否会立即喂给硬件：运行中新增 buffer 如果只入软件队列不提交，可能导致硬件缺 buffer 后停住；相关链路：`QBUF while streaming -> buf_queue -> DMA submit`。

### 常见根因

- 用户态传入的 buffer 类型和队列类型不一致
- plane 数量不一致
- buffer 长度不足
- 驱动私有 buffer 队列没有初始化
- `buf_queue()` 只入队，不在 streaming 状态下提交给硬件

### 核心判断

`QBUF` 是 buffer 从用户态控制进入 vb2，再进入驱动私有队列的关键点。

## 12. `DQBUF` 一直卡住

### 现象

- `STREAMON` 成功
- 应用阻塞在 `VIDIOC_DQBUF`
- `poll()` 超时
- 没有帧返回用户态

### 定位层

重点看完成路径，而不是只看入队路径。

### 驱动检查点

正常完成链应当是：

```text
硬件完成
-> IRQ / tasklet / worker
-> 找到 active buffer
-> 写 sequence / timestamp / payload
-> vb2_buffer_done()
-> DQBUF 返回
```

检查点：

- `start_streaming()` 是否真正启动硬件：没有写寄存器、开 DMA、开中断时，vb2 已经进入 streaming 但硬件不会产生完成事件；相关链路：`STREAMON -> start_streaming -> DMA/IRQ enable`。
- 中断是否到达：`DQBUF` 等的是完成 buffer，中断不到通常说明硬件没有完成一帧或中断配置错误；相关链路：`DMA frame done -> IRQ -> vb2_buffer_done`。
- 中断里是否能找到当前 active buffer：完成事件必须对应一个正在硬件中的 buffer，找不到 active buffer 就无法把任何帧归还给 vb2；相关链路：`buf_queue -> active buffer -> IRQ handler`。
- 是否调用了 `vb2_buffer_done()`：这是驱动把完成 buffer 交回 vb2 的关键动作，不调用时 `done_list` 为空，`DQBUF` 会继续等；相关链路：`IRQ -> vb2_buffer_done -> done_list -> DQBUF`。
- buffer 状态是否为 `VB2_BUF_STATE_DONE`：状态决定用户态拿到的是成功帧还是错误帧，状态错误会表现为应用拿不到有效数据；相关链路：`vb2_buffer_done(state) -> vb2_core_dqbuf`。
- `wait_prepare / wait_finish` 是否正确释放和恢复锁：阻塞等待期间锁处理错误会造成唤醒路径或其他 ioctl 被锁住；相关链路：`vb2_core_dqbuf -> wait_prepare -> sleep -> wait_finish`。
- 是否错误地把 buffer 留在驱动私有 active 队列：buffer 已完成却不从 active 队列移除，会导致重复完成、队列泄漏或后续无 buffer；相关链路：`IRQ -> active list remove -> vb2_buffer_done`。

### 常见根因

- 硬件没有出中断
- DMA 没有真正启动
- sensor 没有 `s_stream(1)`
- 中断来了但没有 `vb2_buffer_done()`
- `vb2_buffer_done()` 状态错误
- 驱动锁没释放导致等待路径无法唤醒

### 核心判断

`DQBUF` 卡住通常不是 `DQBUF` 自己的问题，而是上游没有把完成 buffer 交回 vb2。

## 13. `STREAMON` 成功，但画面黑屏或第一帧异常

### 现象

- 应用能持续 `DQBUF`
- 帧数据全黑、花屏、颜色错乱
- 第一帧异常，后续可能正常

### 定位层

重点看硬件时序、buffer 地址、格式一致性。

### 驱动检查点

- sensor 是否已经真正出流：host 能返回 buffer 不代表上游有有效像素，sensor 未出流时 DMA 可能拿到空数据或旧数据；相关链路：`STREAMON -> s_stream(1) -> sensor output`。
- CSI/ISP 是否配置了正确 lane、mbus code、宽高：接收端解析规则不一致会把有效像素解释成错误颜色或错误帧边界；相关链路：`subdev pad fmt -> CSI/ISP config -> DMA input`。
- DMA 地址是否写入正确：地址错误会让硬件写到错误内存，表现为黑屏、花屏或内存破坏；相关链路：`buf_queue -> DMA address -> hardware descriptor`。
- `bytesperline` 是否和硬件 stride 一致：stride 决定每行跳转距离，不一致会造成图像错位、斜纹或颜色错乱；相关链路：`S_FMT -> bytesperline -> DMA stride`。
- `sizeimage` 是否覆盖完整一帧：buffer 太小会截断帧数据，太大可能掩盖格式计算错误；相关链路：`S_FMT -> sizeimage -> REQBUFS -> DQBUF payload`。
- cache 是否需要同步：CPU 和 DMA 对同一块内存的可见性不同步时，用户态可能看到旧数据；相关链路：`DMA write -> cache sync -> userspace read`。
- 第一帧是否需要丢弃：sensor mode 切换和 ISP 收敛可能让第一帧无效，直接上报会表现为首帧异常；相关链路：`s_stream(1) -> first frame -> vb2_buffer_done`。
- `sequence` 是否从正确时机开始递增：sequence 可用来判断丢帧和首帧时序，递增时机错误会干扰问题定位；相关链路：`IRQ done -> sequence++ -> DQBUF`。

### 常见根因

- sensor 输出格式和 host 接收格式不一致
- RGB/YUV/Bayer code 配错
- stride 使用了用户态宽度，而硬件需要对齐值
- DMA 地址或 plane 偏移错误
- `vb2_buffer_done()` 在硬件尚未真正完成时提前调用
- sensor 刚切 mode 后第一帧不稳定

### 核心判断

能 `DQBUF` 只说明 buffer 生命周期闭合，不代表图像内容一定正确。画面问题要继续查格式、stride、DMA 和硬件时序。

## 14. `STREAMOFF` 后再次 `STREAMON` 失败

### 现象

- 第一次采集正常
- `STREAMOFF` 后再次启动失败
- 第二次启动无中断或 `DQBUF` 卡住

### 定位层

重点看 `stop_streaming()` 和状态清理。

### 驱动检查点

- `stop_streaming()` 是否停止硬件 DMA：DMA 未停会继续写旧 buffer 或干扰下一次启动；相关链路：`STREAMOFF -> vb2_core_streamoff -> q->ops->stop_streaming -> DMA stop`。
- 是否调用上游 `s_stream(0)`：sensor/CSI/ISP 未停流会让管线状态和 host 状态不一致；相关链路：`stop_streaming -> v4l2_subdev_call(..., s_stream, 0)`。
- 是否归还所有还在驱动手里的 active buffer：STREAMOFF 时 vb2 需要收回所有 buffer，未归还会导致下次启动队列状态异常；相关链路：`stop_streaming -> active buffers -> vb2_buffer_done(..., ERROR)`。
- 是否清空驱动私有队列：残留 buffer 会在下一次 STREAMON 时被误用或重复完成；相关链路：`STREAMOFF -> driver queued/active list cleanup`。
- 是否调用 `media_pipeline_stop()`：pipeline 的 `stream_count` 需要回退，漏掉会影响后续 link 切换和再次启动；相关链路：`stop_streaming -> media_pipeline_stop -> stream_count--`。
- 硬件寄存器状态是否恢复到可重新启动状态：某些硬件需要清 pending、reset FIFO 或重新配置 DMA 描述符，否则第二次启动无响应；相关链路：`STREAMOFF -> hardware reset/clear -> next STREAMON`。
- runtime PM 引用是否平衡：电源引用不平衡会造成设备未上电或无法休眠，表现为二次启动失败；相关链路：`pm_runtime_get -> streaming -> pm_runtime_put`。

### 常见根因

- `stop_streaming()` 没有把所有 buffer 用 `VB2_BUF_STATE_ERROR` 退回
- active buffer 指针残留
- 中断未关闭或状态未清
- `media_pipeline_stop()` 漏掉，导致 `stream_count` 没减
- sensor 停流顺序和 host 停 DMA 顺序不匹配

### 核心判断

`STREAMOFF` 不是简单停止硬件，还必须把 vb2、media pipeline、subdev、驱动私有队列全部收尾。

## 15. subdev 节点 `/dev/v4l-subdevX` 没有出现

### 现象

- sensor 已经 probe
- media graph 里能看到 sensor entity
- 但没有 `/dev/v4l-subdevX`

### 定位层

重点看 subdev devnode 生成条件。

### 驱动检查点

- `sd.flags` 是否设置 `V4L2_SUBDEV_FL_HAS_DEVNODE`：该标志声明 subdev 允许生成用户态节点，缺失时 host 不会为它创建 `/dev/v4l-subdevX`；相关链路：`subdev init -> sd.flags -> register_subdev_nodes`。
- host 是否调用 `v4l2_device_register_subdev_nodes()`：真正创建 subdev devnode 的动作在 host 侧统一执行，只设置 flag 不会自动出现节点；相关链路：`complete -> v4l2_device_register_subdev_nodes -> /dev/v4l-subdevX`。
- 内核配置是否启用 subdev devnode 支持：配置关闭时，即使驱动设置 flag，也不会暴露 subdev 节点；相关链路：`CONFIG_VIDEO_V4L2_SUBDEV_API -> subdev devnode creation`。
- `v4l2_device_register_subdev()` 是否已把 subdev 挂进 host：只有归属到 `v4l2_device` 后，host 才能统一注册它的 devnode；相关链路：`async match -> v4l2_device_register_subdev -> register_subdev_nodes`。

### 常见根因

- 只设置了 `V4L2_SUBDEV_FL_HAS_DEVNODE`，但 host 没注册 subdev nodes
- subdev 没有成功匹配到 host
- 期望 sensor 自己创建 `/dev/v4l-subdevX`

### 核心判断

`V4L2_SUBDEV_FL_HAS_DEVNODE` 只是声明这个 subdev 可以有节点。真正创建节点，还要由 host 侧统一注册。

## 16. `media_pipeline_start()` 或 `link_validate` 失败

### 现象

- `STREAMON` 返回错误
- 日志里出现 link validate 相关错误
- `media-ctl -p` 看起来 link 已经 enable

### 定位层

重点看 enabled link 两端 pad format。

### 驱动检查点

逐跳检查：

```text
上游 source pad format
<-> 下游 sink pad format
```

重点字段：

- `mbus code`：决定像素总线编码，source/sink 不一致时接收端会按错误格式解释数据；相关链路：`set_fmt -> source pad code -> sink pad code -> link_validate`。
- `width / height`：决定一帧的边界和行数，逐级不一致会导致 pipeline 验收失败或 DMA 尺寸错误；相关链路：`subdev pad fmt -> link_validate -> video node size`。
- `field`：隔行/逐行属性不一致会影响帧组织方式，某些链路会在 validate 阶段拒绝；相关链路：`pad field -> link_validate`。
- crop / compose / selection：裁剪和缩放会改变模块内部前后 pad 的有效尺寸，只看 sink/source 原始宽高不够；相关链路：`set_selection -> source pad fmt -> downstream sink pad`。
- sink/source pad 是否属于直通或转换关系：直通模块通常要求两端一致，转换模块允许不同但必须符合驱动规则；相关链路：`entity pad ops -> link_validate -> pipeline_start`。

### 常见根因

- link enable 了，但 format 没协商
- `TRY` format 正确，`ACTIVE` format 不正确
- sensor source pad 和 CSI sink pad 的 mbus code 不一致
- scaler/ISP 前后 pad 的转换规则没维护好
- `MUST_CONNECT` pad 没有有效连接

### 核心判断

link enable 只说明拓扑开关打开。pipeline 能不能跑，要等 `media_pipeline_start()` 和 `link_validate` 做启动前验收。

## 17. DMA 内存分配失败

### 现象

- `REQBUFS` 返回 `ENOMEM`
- 分辨率降低后正常
- buffer 数量减少后正常

### 定位层

重点看 `vb2_mem_ops` 和系统 DMA 内存。

### 驱动检查点

- `q->mem_ops` 使用的是哪种后端：不同后端决定 buffer 如何分配和映射，硬件需要 DMA 时选错后端会导致分配或访问失败；相关链路：`vb2_core_reqbufs -> q->mem_ops->alloc/map`。
- `vb2_dma_contig_memops` 是否适合当前硬件：连续 DMA 后端依赖 CMA，适合需要物理连续内存的硬件；相关链路：`REQBUFS -> dma-contig alloc -> CMA`。
- `vb2_dma_sg_memops` 是否适合当前硬件：scatter-gather 后端要求硬件或 IOMMU 能处理离散页，硬件不支持时会出数据路径问题；相关链路：`REQBUFS/QBUF -> dma-sg map -> hardware SG DMA`。
- `vb2_vmalloc_memops` 是否适合当前硬件：vmalloc 后端通常不适合直接硬件 DMA，更多用于纯软件处理场景；相关链路：`vb2_vmalloc_memops -> CPU buffer -> no direct DMA`。
- `q->dev` 是否设置成正确 DMA device：DMA API 需要根据 device 做地址映射和限制判断，指错 device 会导致分配、映射或 cache 属性错误；相关链路：`vb2_queue_init -> q->dev -> mem_ops alloc`。
- `alloc_devs[]` 是否正确：多 plane 或特殊硬件可能按 plane 指定分配设备，错误会导致某些 plane 分配失败；相关链路：`queue_setup -> alloc_devs -> mem_ops alloc`。
- `sizeimage * buffer_count` 是否超出 CMA 能力：总申请量超过连续内存池时，REQBUFS 会失败或只能分配更少 buffer；相关链路：`S_FMT -> sizeimage -> REQBUFS count -> CMA`。
- DMA mask 是否配置正确：DMA mask 决定设备可访问地址范围，配置过窄会让分配到的内存无法被设备访问；相关链路：`device dma_mask -> dma_alloc -> DMA address`。

### 常见根因

- CMA 太小
- 驱动把 `q->dev` 指错到不具备 DMA 能力的 device
- 格式计算导致 `sizeimage` 过大
- 多路视频同时申请连续内存
- 硬件需要连续内存，但驱动误用非预期内存模型

### 核心判断

buffer 分配失败不一定是 vb2 状态机问题，经常是 DMA device、CMA、sizeimage 和 memory backend 的组合问题。

## 18. 多进程访问时返回 `-EBUSY`

### 现象

- 一个应用打开后，另一个应用访问失败
- 某些 ioctl 返回 `EBUSY`
- 关闭应用后恢复正常

### 定位层

重点看文件句柄、队列 owner、优先级和 streaming 状态。

### 驱动检查点

- 是否使用 `v4l2_fh`：文件句柄用于记录每个 fd 的 V4L2 状态，多进程场景缺少它会让 owner、优先级和事件管理混乱；相关链路：`open -> v4l2_fh_init/add -> ioctl/release`。
- `vb2_queue` owner 是否被某个 file handle 占用：vb2 会限制队列被一个 owner 进行 streaming 操作，其他 fd 重配队列会返回 busy；相关链路：`REQBUFS/STREAMON -> vb2 queue owner -> second fd ioctl`。
- streaming 是否已经开始：运行中的队列通常禁止修改格式和重新申请 buffer，另一个进程操作会被拒绝；相关链路：`STREAMON -> q->streaming -> S_FMT/REQBUFS`。
- 是否有多个进程同时 `REQBUFS/QBUF/STREAMON`：buffer 队列不是任意多进程共享配置对象，并发操作会破坏队列状态机；相关链路：`fd A queue setup -> fd B queue setup/streamon`。
- 驱动私有锁是否允许多开：驱动如果只支持单 open，需要在 open 或关键 ioctl 上明确互斥，否则会出现状态互相覆盖；相关链路：`open -> driver lock -> shared hardware state`。

### 常见根因

- 一个 fd 已经拥有 streaming 队列
- 另一个 fd 尝试重配格式或 buffer
- 驱动没有明确处理多 open 场景
- release 路径没有关闭 streaming

### 核心判断

V4L2 允许多 fd 访问设备节点，但 buffer 队列和 streaming 通常有明确 owner，不能被多个应用随意重配。

## 19. 工程排查固定清单

实际调试时可以按下面顺序逐层打点：

1. `probe()`
   - 原因：所有 V4L2 对象都从驱动 probe 开始建立，probe 没跑或中途失败时，后续节点、队列、拓扑都不会完整出现。
   - 相关链路：`module/device match -> probe -> V4L2 object init`
2. `v4l2_device_register()`
   - 原因：它建立 host 的 V4L2 管理容器，后面的 `video_device` 和 `subdev` 归属都依赖这层对象。
   - 相关链路：`probe -> v4l2_device_register -> v4l2_dev`
3. `video_register_device()`
   - 原因：它是真正把 video node 注册成用户态可见节点的入口，失败时 `/dev/videoX` 不会出现。
   - 相关链路：`video_register_device -> cdev_add/device_register -> /dev/videoX`
4. `v4l2_open()`
   - 原因：它证明字符设备层已经从通用 `def_chr_fops` 切到 V4L2 core，open 问题可以继续往 V4L2 层排查。
   - 相关链路：`open -> chrdev_open -> v4l2_fops.open -> v4l2_open`
5. `vdev->fops->open`
   - 原因：`v4l2_open()` 只是公共入口，驱动私有打开逻辑必须靠这一层转发进入。
   - 相关链路：`v4l2_open -> vdev->fops->open`
6. `video_ioctl2()`
   - 原因：大多数标准驱动把 `vdev->fops->unlocked_ioctl` 设为它，进入这里说明 ioctl 已到 V4L2 标准派发层。
   - 相关链路：`v4l2_ioctl -> vdev->fops->unlocked_ioctl -> video_ioctl2`
7. `v4l2_ioctl_ops->vidioc_*`
   - 原因：这是控制类 ioctl 的最终业务入口，未进入说明命令没有从 V4L2 core 派发到驱动。
   - 相关链路：`video_ioctl2 -> __video_do_ioctl -> v4l2_ioctl_ops->vidioc_*`
8. `vb2_ioctl_*`
   - 原因：`REQBUFS/QBUF/DQBUF/STREAMON` 这类 buffer 命令通常由 vb2 helper 接管，未进入说明 `v4l2_ioctl_ops` 绑定有问题。
   - 相关链路：`v4l2_ioctl_ops->vidioc_* -> vb2_ioctl_*`
9. `vb2_ops`
   - 原因：vb2 只负责通用状态机，真正的 buffer 规格、入队和启停要回调驱动的 `vb2_ops`。
   - 相关链路：`vb2_ioctl_* -> vb2_core_* -> q->ops->*`
10. `media_pipeline_start()`
   - 原因：它会沿 enabled link 做 graph walk 和启动前验收，失败时说明问题进入 MC 拓扑或 format 层。
   - 相关链路：`start_streaming -> media_pipeline_start -> link_validate`
11. `v4l2_subdev_call(..., s_stream, 1)`
   - 原因：video node 启动不等于 sensor/CSI/ISP 都出流，上游模块必须逐级收到 `s_stream(1)`。
   - 相关链路：`media_pipeline_start ok -> subdev s_stream(1)`
12. IRQ / DMA done
   - 原因：只有硬件完成事件到达，驱动才有机会把 active buffer 变成 done buffer。
   - 相关链路：`sensor/CSI/DMA running -> IRQ/DMA done`
13. `vb2_buffer_done()`
   - 原因：这是完成路径的关键交接点，不调用时用户态 `DQBUF` 没有 completed buffer 可取。
   - 相关链路：`IRQ handler -> vb2_buffer_done -> vb2 done_list`
14. `DQBUF`
   - 原因：它是整条传输链对用户态的最终反馈，成功返回说明 buffer 生命周期至少完成了一轮闭环。
   - 相关链路：`vb2 done_list -> VIDIOC_DQBUF -> userspace`

## 20. 一句话总结

V4L2 工程问题的关键不是记住单个错误码，而是把现象放回正确层次：

- 节点问题看 `video_device`
- ioctl 问题看 `v4l2_fops / vdev->fops / v4l2_ioctl_ops`
- buffer 问题看 `vb2_queue / vb2_ops / mem_ops`
- 拓扑问题看 `media_entity / pad / link`
- 异步问题看 `subdev / async_subdev / notifier`
- 传输问题看 `start_streaming / s_stream / DMA / IRQ / vb2_buffer_done`

## 21. 回答的问题

- `/dev/videoX` 没出来时应该先看哪条注册链
- `QUERYCAP / S_FMT / REQBUFS / QBUF / STREAMON / DQBUF` 分别落到哪一层
- sensor probe 成功但 host 不绑定时应该怎么查
- media graph 完整但不能起流时应该怎么查
- `DQBUF` 卡住时为什么要优先看完成路径
- `STREAMOFF` 后再次启动失败时为什么要看状态清理
- 工程调试时怎样从用户态现象一路反推到驱动函数
