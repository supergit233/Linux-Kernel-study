# 陀螺仪防抖核心对象

## 导读

### 本章定位

这一章先建立对象模型。陀螺仪防抖跨越传感器驱动、视频驱动和算法层，如果只记函数名，很容易分不清“谁产生数据、谁保存时间、谁负责坐标转换、谁生成补偿”。

### 核心对象

- IMU sample
- gyro sample
- video frame
- timestamp
- calibration model
- stabilization state
- warp/crop result

### 关键函数

当前先记录对象职责，不绑定具体源码函数。后续接入真实代码后，再补 `struct iio_dev`、`struct video_device`、`struct vb2_buffer`、算法私有结构等字段。

### 主流程

```text
IMU sample                   // 表示一次传感器采样包，可能同时包含 accel/gyro/temp
-> gyro sample                // 从 IMU sample 中取出角速度，作为防抖主输入
-> timestamp                 // 把采样放到统一时间轴
-> calibration model         // 修正零偏、比例、轴向和相机外参
-> video frame               // 等待被稳定的一帧图像
-> stabilization state       // 保存积分姿态、平滑轨迹和历史状态
-> warp/crop result          // 输出当前帧补偿参数
```

## 这一章按什么逻辑展开

本章按“数据从哪里来 -> 如何被校准 -> 如何参与每帧防抖”的顺序展开。这样拆可以避免把驱动对象、算法对象和图像对象混在一起。

## 1. IMU sample：accel / gyro / temp

职责：

- 保存一次 IMU 传感器采样
- 同时描述加速度、角速度和芯片温度
- 给防抖算法、标定和调试提供原始输入

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `ax/ay/az` | 三个方向的加速度。静止时主要看到的是重力落在哪些轴上，例如某个轴接近 `1g` 或 `-1g` | 它不是防抖的主补偿量，但能帮忙判断板子姿态、安装方向、加速度 scale 是否合理 | 静止时检查三轴合成值是否接近 `1g`，也可辅助确认芯片轴和相机轴的对应关系 | 把加速度当成画面位移会走偏；scale 或轴向错时，会误判安装方向 |
| `gx/gy/gz` | 摄像头绕 X/Y/Z 三个方向“转得有多快”。例如手轻轻旋转板子，某一个轴的数值会明显变化 | 画面抖动很多时候就是摄像头发生了小幅旋转。gyro 告诉算法“刚才往哪个方向转、转得多快” | 后面会用“转得多快 × 持续多久”算出“转了多少角度”，再让画面往反方向补偿 | 轴方向错会越补越抖；单位错会让补偿量过大或过小 |
| `temperature` | 采样时芯片温度，例如 `29.24C` | 很多 gyro 温度变了，静止时输出的假速度也会变 | 先作为日志观察温度是否稳定；后续可以建立“温度变化 -> bias 变化”的修正关系 | 温漂没处理时，热起来后静止画面也可能慢慢漂 |
| `timestamp` | 这一次 IMU 数据对应的时间点 | 视频是一帧一帧来的，IMU 是连续采样的；必须知道这笔数据对应哪一帧或哪一行画面 | 用来从 gyro 数据流里找出“当前视频帧曝光时，相机到底怎么动了” | 时间对不齐会慢半拍，快速运动时可能越防越晃 |
| `seq` | 第几笔 IMU 样本，或者 FIFO 里的样本序号 | 用来确认采样有没有断、有没有重复、间隔是不是稳定 | 调试时看 `seq` 是否连续，结合 timestamp 判断是否丢样 | 丢样会导致积分中间缺一段运动，补偿可能突然跳一下 |

三类数据在防抖里的分工不同：

| 数据类型 | 直接含义 | 在防抖里的角色 | 新人容易混淆的点 |
| --- | --- | --- | --- |
| `accel` 加速度 | 设备受到的加速度，静止时主要是重力 | 更适合做姿态、安装方向、scale 和异常检查 | 它不是“速度”，也不是“位移”；手移动时会混入线性加速度，不能简单积分成画面位移 |
| `gyro` 角速度 | 设备正在绕某个轴转得多快 | 视频防抖的主输入，用来积分相机旋转 | 它不是角度，必须乘时间差后才变成一小段角度变化 |
| `temp` 温度 | 芯片内部温度或接近芯片温度 | 用来观察零偏随温度变化的趋势 | 它通常不直接进第一版防抖算法，但会影响长时间稳定性 |

这里第一次出现“积分”这个词，可以先不要按复杂数学理解。防抖里的积分可以先理解成：

```text
转得有多快                         // gyro 给出的角速度
× 转了多久                         // 两个 timestamp 之间的时间差
= 这段时间大概转了多少角度          // 后面用这个角度去反向补偿画面
```

常见误区：

- 只有 `gx/gy/gz` 不够，没有 timestamp 就无法和视频帧对齐
- 看到 MPU6050 同时输出 `accel/gyro/temp` 时，不要把三类数据都当成同一种“防抖输入”
- 角速度单位必须确认，`deg/s` 和 `rad/s` 混用会让补偿量完全错误
- 坐标轴方向不能凭直觉猜，要通过标定或实验确认

## 2. video frame

职责：

- 保存一帧待稳定图像
- 提供图像尺寸、像素格式、buffer 状态和时间戳
- 是 warp/crop 的目标对象

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `width/height` | 这一帧图像有多宽、多高 | 防抖补偿会移动或旋转画面，必须知道图像边界在哪里 | 计算图像中心、裁剪范围和补偿后是否露黑边 | 尺寸理解错，warp/crop 会裁错位置 |
| `format` | 这一帧像素怎么存，例如 YUV、RGB、RAW | 算法算出来的是“怎么变换画面”，真正执行 warp 时还要知道像素怎么读写 | 决定 CPU/GPU/ISP 处理图像时按什么格式访问 buffer | 格式错会花屏、颜色错，或者 warp 模块无法处理 |
| `timestamp` | 这一帧画面对应的时间点 | 要找出这一帧曝光时相机怎么抖，就必须拿它去对齐 gyro 数据 | 用它去 gyro 样本里查“这一帧应该用哪段运动补偿” | 如果它其实是 buffer 完成时间而不是曝光时间，补偿会有延迟 |
| `exposure_time` | 这一帧曝光持续了多久 | 曝光不是瞬间完成的，曝光期间相机也可能在动 | 用来估算曝光中心；rolling shutter 时还要估算每一行的时间 | 曝光时间理解错，运动模糊和逐行补偿都会判断错 |
| `frame_id` | 第几帧视频 | gyro 是连续流，video 是一帧一帧的；帧序号能帮忙对日志和查丢帧 | 调试时把 `frame_id`、frame timestamp、gyro timestamp 对齐看 | 丢帧没发现时，算法可能把上一帧的运动用到下一帧 |

常见误区：

- buffer done timestamp 不一定等于曝光开始时间
- 如果是 rolling shutter，一帧内每一行的曝光时间不同
- 防抖算法通常需要为画面预留裁剪边界，否则补偿后会露黑边

## 3. timestamp

职责：

- 把 gyro sample 和 video frame 放到同一时间轴
- 是防抖系统里最关键的工程字段之一

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `gyro_ts` | gyro 样本发生的时间 | 同样一组 `gx/gy/gz`，早 10ms 或晚 10ms，对应的画面帧可能完全不同 | 用它把 gyro 样本排到时间轴上，供当前帧查找 | gyro 时间乱跳会导致补偿忽大忽小 |
| `frame_ts` | 视频帧对应的时间 | 防抖必须知道“这帧画面拍下来的时候，相机在哪里动” | 用它查找这一帧曝光期间的 gyro 数据 | 如果拿错成 `DQBUF` 时间，画面会明显慢半拍 |
| `clock_id` | 这个时间戳是用哪一块钟表记的 | 如果 gyro 用一块钟、video 用另一块钟，两个时间数字不能直接比较 | 决定是否需要做时钟换算 | 两边时钟没统一时，看起来都有 timestamp，但对齐全是错的 |
| `time_offset` | gyro 时间和视频时间之间固定差了多少 | 真实系统里 gyro 采样、FIFO 上报、sensor 曝光、ISP 出帧都有延迟 | 对齐时把这个固定差值加回去或减掉 | offset 错时，防抖相位不对，快速转动时尤其明显 |

调试检查点：

- gyro timestamp 是否单调递增
- gyro 采样间隔是否稳定
- frame timestamp 是否对应 SOF、EOF 还是 buffer done
- gyro 和 frame 是否使用同一时钟源

## 4. calibration model

职责：

- 把原始 gyro 和原始图像坐标转换到统一模型
- 修正零偏、比例、坐标轴、相机内参和相机-陀螺仪外参

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `gyro_bias` | gyro 静止不动时仍然冒出来的“假速度” | 如果不去掉，算法会以为相机一直在慢慢转 | 每个 gyro 样本先减掉 `gyro_bias`，再进入积分 | 静止画面会慢慢漂，视频越长越明显 |
| `accel_bias` | accel 静止时相对真实重力多出来或少掉的偏差 | 用 accel 判断姿态和安装方向时，需要知道它有没有偏 | 静止检查、姿态辅助和安装方向确认时使用 | 明明静止，合成加速度却明显偏离 `1g`，容易误判姿态 |
| `gyro_scale` | 把 gyro 原始数字换成真实角速度的比例，例如 `raw × scale = rad/s` | 算法不能直接拿寄存器原始值算，必须知道这个数字到底代表多快的转动 | 把 `gx/gy/gz` 从 LSB 或驱动原始值转换成 `deg/s` 或 `rad/s` | scale 偏小会补偿不够，scale 偏大会补偿过头 |
| `accel_scale` | 把 accel 原始数字换成 `g` 或 `m/s^2` 的比例 | 用来确认静止时重力是否接近 `1g`，也能辅助检查量程配置 | 把 `ax/ay/az` 从 raw 转成可读物理量 | scale 错时，板子静止也可能看起来不像受 1g 重力 |
| `temp_bias_model` | 描述温度变化时 gyro bias 如何变化 | 长时间运行时，温度变化会让零偏慢慢变 | 第一版可以先只记录温度；后续再按温度修正 bias | 冷启动和热机后的防抖漂移表现不一致 |
| `axis_map` | 说明 gyro 的 X/Y/Z 轴分别对应设备或相机的哪个方向 | 芯片贴在板子上可能是旋转过的，不能假设 gyro X 就是画面水平 | 把 `gx/gy/gz` 重新排列、改符号，变成算法理解的方向 | 水平晃动可能被当成垂直晃动，或者方向反了 |
| `R_gyro_to_cam` | 把“gyro 眼里的旋转”翻译成“相机眼里的旋转”的旋转关系 | gyro 和 camera 不一定朝同一个方向安装，防抖最终要补偿的是相机画面 | 算法用它把 gyro 运动转换到 camera 坐标系 | 某些方向有效，某些方向更抖，典型是外参不准 |
| `K` | 相机内参，简单说就是“相机如何把真实世界投到像素平面” | 同样转 1 度，不同焦距下画面移动的像素量不同 | 把相机旋转转换成图像上的像素位移或 homography | 焦距/主点错，画面补偿幅度和中心都会偏 |
| `distortion` | 镜头畸变参数，描述画面边缘被镜头拉弯了多少 | 广角镜头边缘不是理想直线映射，防抖 warp 时会受影响 | 畸变明显时，需要先校正或在 warp 里一起考虑 | 画面中心还行，边缘拉扯、变形明显 |
| `time_offset` | gyro 时间和 video 时间之间固定差了多少 | 传感器采样、FIFO、ISP 出帧都有延迟，两个时间戳不一定天然对齐 | 对齐 gyro 和 frame 时修正这个差值 | 补偿慢半拍或提前，快速运动时越明显 |

这里的“标定得到”不要理解成玄学。它的意思是：通过静止测试、已知角度转动、棋盘格拍摄、离线拟合等方法，把这些参数测出来。比如真实转了 10 度，但算法用当前 `scale` 算出来只有 8 度，就说明比例关系不准，需要把这个换算比例校正回来。

## 5. stabilization state

职责：

- 保存防抖算法的历史状态
- 把单帧补偿放进连续视频轨迹里处理

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `raw_rotation` | gyro 算出来的原始相机晃动轨迹 | 它记录“相机实际怎么动了”，里面既有手抖，也有主动转动 | 和 `smooth_rotation` 对比，找出需要抵消的高频抖动 | 原始轨迹错，后面所有补偿都建立在错的运动上 |
| `smooth_rotation` | 算法希望画面呈现的平滑运动轨迹 | 防抖不是把所有运动清零，而是保留慢慢转动，去掉小抖动 | 用它作为目标轨迹，计算当前帧应该补多少 | 平滑过强会像锁死画面，平滑过弱防抖不明显 |
| `correction` | 当前帧要额外补的那一下 | 它就是最终让画面稳定的补偿量 | 传给 warp/crop 模块，生成图像变换 | 补偿方向或大小错，画面会拉扯、跳变或越补越抖 |
| `crop_margin` | 为防抖预留的一圈“安全边” | 画面被移动/旋转后边缘可能露黑，需要提前多采一点再裁掉 | 决定最大能补多少抖动 | 太小会露黑边，太大会损失视角 |
| `history_window` | 算法看多长一段历史运动来判断什么是“平滑” | 需要用一段时间的运动趋势区分手抖和主动转动 | 控制平滑强度和实时延迟 | 窗口太短不稳，窗口太长延迟大 |

## 6. warp/crop result

职责：

- 表示当前帧应该如何被变换和裁剪
- 是算法输出和图像处理模块之间的接口

字段建议：

| 字段 | 人话理解 | 为什么防抖需要它 | 后续怎么用 | 错了会怎样 |
| --- | --- | --- | --- | --- |
| `transform` | 告诉图像处理模块“这一帧该怎么挪、怎么转、怎么变形” | gyro 算出来的是运动，最终必须变成图像上的操作 | 可能表现为旋转矩阵、homography、mesh 或逐行变换 | transform 错会导致画面变形、反向移动或局部撕裂 |
| `crop_rect` | 最终从补偿后的大图里裁哪一块作为输出 | 防止画面移动后露出黑边或无效区域 | 输出稳定视频时只保留这个有效区域 | crop 太小会露黑边，crop 太大损失画面 |
| `valid` | 当前帧的补偿结果是否可信 | gyro 丢样、timestamp 异常或补偿过大时，不应该硬套错误补偿 | 不可信时跳过防抖、减弱补偿或重置状态 | 错误数据继续使用，会导致画面突然大跳 |
| `quality` | 当前帧防抖质量的调试分数或状态 | 方便日志里判断这帧为什么稳或不稳 | 可用于统计丢样、过大补偿、crop 超限等情况 | 没有质量指标时，问题只能靠肉眼猜 |

## 本章边界

本章只建立对象和字段模型，不展开具体采集驱动、V4L2 buffer 机制和防抖数学公式。后续章节会分别说明这些对象从哪里被填充、怎样被消费。

## 一句话总结

陀螺仪防抖的核心不是单个算法函数，而是 `gyro sample + video frame + timestamp + calibration + stabilization state` 这些对象能否在同一个时空模型里闭合。
