# 陀螺仪数据采集链路

## 导读

### 本章定位

这一章讲 IMU 数据如何从硬件进入内核和用户态/算法层。MPU6050 这类器件通常同时输出 `accel`、`gyro`、`temp` 三类数据，其中防抖主输入是高频、低延迟、带时间戳的 `gyro` 角速度流，但 `accel` 和 `temp` 对安装方向、单位检查和温漂排查同样重要。

### 核心对象

- gyro hardware
- accel / gyro / temp channels
- I2C/SPI controller
- gyro driver private data
- `struct iio_dev` 或 input device
- FIFO / trigger / IRQ
- gyro sample buffer

### 关键函数

当前先按通用入口记录：

- `i2c_driver.probe()` / `spi_driver.probe()`
- `devm_iio_device_alloc()`
- `iio_device_register()`
- `iio_triggered_buffer_setup()`
- `request_threaded_irq()`
- `iio_push_to_buffers_with_timestamp()`

### 主流程

```text
I2C/SPI 匹配 gyro driver             // 总线根据 compatible/id_table 绑定陀螺仪驱动
-> probe 初始化芯片                  // 配置量程、采样频率、滤波、FIFO、中断
-> 注册 IIO/input 设备               // 把 gyro 数据变成内核里可读取的传感器设备
-> FIFO/trigger/IRQ 触发采样          // 用硬件节奏持续拿样本，不靠手动一笔一笔读
-> 读取 accel/gyro/temp raw           // 从寄存器或 FIFO 批量读加速度、角速度和温度
-> 换算成物理量                       // 把 raw 转成 g、deg/s 或 rad/s、摄氏度
-> 添加 timestamp                    // 给样本打上时间戳
-> 推入 buffer                       // 交给算法线程、用户态或中间件消费
```

## 这一章按什么逻辑展开

本章按“驱动绑定 -> 芯片配置 -> 采样触发 -> 时间戳 -> buffer 输出”的顺序展开。这个顺序符合真实 gyro 驱动从 probe 到持续采样的生命周期。

## 1. 驱动绑定和 probe

陀螺仪通常挂在 I2C 或 SPI 上。驱动入口不是防抖算法，而是普通总线驱动 probe。

```text
DTS compatible / ACPI / id_table       // 板级描述或设备 ID 声明 gyro 存在
-> i2c_driver.probe() / spi_driver.probe() // 总线匹配成功后进入 gyro 驱动
-> 读取 WHOAMI                         // 确认芯片身份
-> 分配私有结构                         // 保存 regmap、irq、量程、ODR、校准状态
-> 初始化寄存器                         // 配置采样率、量程、滤波、FIFO
-> 注册 IIO/input                       // 向内核暴露传感器数据接口
```

工程上最先确认的是：

- 驱动是否 probe
- WHOAMI 是否正确
- I2C/SPI 读写是否稳定
- 采样率和量程是否符合防抖需求

## 2. IIO 模型为什么适合 gyro

Linux 里很多 IMU/gyro 使用 IIO 子系统，因为它天然适合传感器采样流：

| IIO 能力 | 人话理解 | 对防抖有什么用 |
| --- | --- | --- |
| channel | 把 `ax/ay/az`、`gx/gy/gz`、`temp` 这种传感器量描述成标准通道 | 算法或用户态能知道每列数据代表哪个轴、哪种物理量 |
| scale / offset | 描述原始值怎么换成物理量，以及是否有偏移 | 避免算法拿原始寄存器值直接积分，也避免 accel/temp 被误读 |
| sampling_frequency | 采样频率，也就是每秒采多少次 | 防抖需要稳定高频采样，采样太低会漏掉手抖 |
| trigger | 触发采样的节奏来源，可以是定时器、数据就绪中断等 | 决定样本什么时候产生，影响采样间隔是否稳定 |
| buffer | 内核里存放一串连续传感器样本的队列 | 防抖算法可以按时间窗口取一段 gyro 数据 |
| timestamp | 每个样本或每批样本的时间 | 用来和视频帧对齐，是 gyro 能用于防抖的关键 |

典型关系是：

```text
struct iio_dev                       // IIO 子系统看到的传感器设备
-> channels                          // 描述 accel、angular_velocity、temp 等通道
-> info->read_raw                    // 读取 scale、offset、采样率或单次原始值
-> triggered buffer                  // 持续采样时把样本推入 buffer
-> userspace / algorithm             // 用户态或算法线程消费样本
```

如果某个项目没有走 IIO，而是走 input、misc 或私有 HAL，也要保留同样的问题意识：

- 数据单位是什么
- 时间戳来自哪里
- 采样间隔是否稳定
- 是否会丢样

## 3. FIFO、trigger 和 IRQ

防抖通常需要高频 gyro 数据。常见实现不是每帧读一次，而是让 gyro 以固定 ODR 采样，再通过 FIFO/IRQ 批量取走。

这几个词第一次看容易混：

| 术语 | 人话理解 | 为什么防抖需要关心 |
| --- | --- | --- |
| `ODR` | output data rate，gyro 自己每秒产生多少笔数据 | ODR 太低会漏掉手抖，太高会增加总线和 CPU 压力 |
| `FIFO` | gyro 芯片内部的小队列，先把多笔样本攒起来 | CPU 不用每来一笔就读一次，但要防止 FIFO 溢出和时间还原错误 |
| `trigger` | 告诉驱动“该采一笔数据了”的触发条件 | trigger 不稳，样本间隔就不稳，积分出来的角度也会抖 |
| `IRQ` | 硬件中断，芯片主动通知 CPU“有数据了” | 可以降低轮询开销，但 IRQ 延迟会影响 timestamp 精度 |

```text
gyro ODR 配置                         // 设置 gyro 自己每秒产生多少笔 gx/gy/gz
-> gyro FIFO                           // 芯片内部先缓存多笔样本，避免 CPU 来不及读
-> watermark IRQ                       // FIFO 攒到一定数量后通知 CPU 来取
-> threaded irq handler                // 驱动在线程化中断里批量读取 FIFO，避免中断里干太久
-> iio_push_to_buffers_with_timestamp() // 把样本和 timestamp 推给 IIO buffer
```

这里最重要的不是 IRQ 名字，而是能不能回答：

- 每个样本是否有独立 timestamp
- 如果只给 FIFO 批次一个 timestamp，如何还原每个样本时间
- FIFO 溢出时是否有日志或计数
- IRQ 线程是否会被视频链路或其他高负载阻塞

## 4. 时间戳是采集链路的核心产物

gyro sample 没有可靠 timestamp，就很难用于视频防抖。

时间戳来源大致有几种：

- 硬件 sample timestamp
- FIFO 批次 timestamp 加采样间隔反推
- IRQ 到达时间
- 软件读取时间

可靠性通常从高到低排列：

```text
硬件 sample timestamp               // 最理想，直接知道每个样本采样时刻
-> FIFO 批次 timestamp 反推          // 可用，但要处理 FIFO 深度和 ODR
-> IRQ timestamp                     // 有中断延迟和调度抖动
-> 软件 read timestamp               // 最容易受系统负载影响
```

## 5. accel / gyro / temp 三类通道怎么理解

以 MPU6050 这类 IMU 的实机输出为例，常见日志会同时出现：

```text
raw accel=(..., ..., ...) temp=... gyro=(..., ..., ...)
val accel=(..., ..., ...)g temp=(...)C gyro=(...)dps
```

这两行表达的是同一批传感器数据的两个层级：

| 数据 | `raw` 是什么 | `val` 是什么 | 在防抖项目里的用法 |
| --- | --- | --- | --- |
| `accel` | 加速度寄存器里的原始整数 | 按当前加速度量程换算后的 `g` 或 `m/s^2` | 静止时主要用来检查重力方向、芯片安装方向和加速度 scale 是否合理 |
| `gyro` | 角速度寄存器里的原始整数 | 按当前 gyro 量程换算后的 `deg/s` 或 `rad/s` | 防抖主输入，后续减零偏、转单位、按时间积分成相机旋转 |
| `temp` | 温度寄存器里的原始整数 | 按芯片公式换算后的摄氏度 | 先作为调试日志保存，后续可用来观察温度变化是否导致 gyro 零偏漂移 |

第一版防抖算法可以只使用 `gyro` 做补偿，但采集程序最好把 `accel` 和 `temp` 一起保存。这样做不是贪多，而是为了后面能回答三个基础问题：

- `accel` 静止时合成值是否接近 `1g`，说明加速度量程和换算大体可信。
- `gyro` 静止时是否接近 `0 dps`，不接近的平均值就是后续要减掉的零偏。
- `temp` 是否随运行时间变化，变化时 `gyro` 零偏是否也跟着漂。

不要把三类数据混成同一个概念。`accel` 描述受力或重力方向，`gyro` 描述旋转速度，`temp` 描述芯片温度。视频防抖最直接需要的是旋转速度，但工程排查不能只看旋转速度。

## 6. 单位、坐标轴和量程

IMU/gyro 采集链路至少要确认：

| 检查项 | 为什么需要确认 | 后续有什么用 | 如果错了会出现什么现象 |
| --- | --- | --- | --- |
| `accel` 单位 | 加速度可能是 raw、`g` 或 `m/s^2`，静止时应主要反映重力 | 用来判断板子是否倾斜、轴向是否符合预期、scale 是否大体正确 | 静止合成值明显不是 `1g`，姿态和安装方向判断会失真 |
| 原始值单位 | gyro 寄存器读出来的通常是 ADC/LSB 原始值，不一定已经是物理角速度 | 决定后续是否还要乘 `scale` 才能进入积分算法 | 补偿量整体过大或过小，甚至看起来完全没有防抖效果 |
| `scale` 是否已经应用 | IIO、HAL 或算法层可能已经做过一次单位换算，重复应用会把角速度放大或缩小 | 保证算法拿到的是同一层语义的数据，避免重复换算 | 同一段运动在日志里幅度异常，调算法参数也无法稳定 |
| 输出是 `deg/s` 还是 `rad/s` | 防抖积分和旋转矩阵/四元数计算通常按 `rad/s` 处理 | 决定角速度积分时是否需要从角度制转弧度制 | 如果把 `deg/s` 当 `rad/s`，补偿量会放大约 57.3 倍，画面会剧烈反向或拉扯 |
| 量程是否会饱和 | 手持快速转动时角速度可能超过当前 gyro 量程，饱和后真实运动被截断 | 判断是否要调大量程，或在算法里检测并降级处理饱和样本 | 快速转动时曲线削顶，防抖突然失效或产生大跳变 |
| 坐标轴和相机坐标轴是否一致 | gyro 的 X/Y/Z 轴、板子安装方向和相机图像坐标通常不是天然一致 | 决定 `gx/gy/gz` 如何映射到相机 pitch/yaw/roll，再生成正确补偿 | 水平转动触发垂直补偿，或者补偿方向反了，越防越抖 |
| 安装方向是否在 DTS、驱动或 HAL 中描述 | 有些平台会在 DTS、驱动私有配置或 HAL 中描述 sensor mount matrix | 决定坐标转换应该在哪一层完成，避免驱动和算法层重复旋转 | 离线数据和实时数据方向不一致，不同应用看到的 gyro 轴定义不同 |
| `temp` 是否保存 | 温度变化会影响 gyro 静止零偏，尤其是热机过程 | 用来分析“刚开机还好，运行一段时间开始漂”的问题 | 日志里没有温度时，很难判断漂移是否和温漂有关 |

这些问题如果没确认，后面算法看起来再合理，画面也可能朝反方向补偿。

这里的重点是：IMU 采集链路输出的不是“随便一组三轴数字”，而是一组需要分清角色的物理量。`gyro` 决定积分幅度和补偿方向，`accel` 帮助检查姿态和安装方向，`temp` 帮助解释零偏随时间变化。任何一项含糊，后面都会表现成防抖效果问题，但根因其实还停在采集链路。

## 本章边界

本章只讲 IMU 数据如何稳定、连续、带时间戳地出来，重点说明防抖主输入 `gyro`，同时保留 `accel` 和 `temp` 的工程检查价值。不展开视频帧如何采集，也不展开防抖算法如何使用这些样本。

## 一句话总结

陀螺仪采集链路的达标标准不是“能读到寄存器值”，而是能稳定输出单位明确、坐标清楚、时间戳可靠、采样间隔可验证的 `accel/gyro/temp` 数据，并能说明其中 `gyro` 如何进入防抖主流程。
