# 时间戳同步机制

## 导读

### 本章定位

这一章讲 gyro sample 和 video frame 如何对齐。陀螺仪防抖里，时间同步往往比滤波算法更先决定效果上限。

### 核心对象

- gyro timestamp
- frame timestamp
- clock source
- time offset
- sample interpolation
- exposure center

### 关键函数

当前先按概念入口记录：

- gyro sample timestamp 生成点
- frame SOF/EOF/done timestamp 生成点
- clock conversion
- gyro interpolation
- delay calibration

### 主流程

```text
采集 gyro sample timestamp             // 得到每个角速度样本的时间
-> 采集 frame timestamp                 // 得到每帧对应时间
-> 确认 clock source                    // 判断两边时间戳是否同一时钟
-> 估计 time offset                     // 修正传感器和视频链路延迟
-> 查找帧曝光窗口内 gyro 样本             // 找到该帧对应的运动数据
-> 插值或积分                            // 得到帧时刻或行时刻的姿态
```

## 这一章按什么逻辑展开

本章按“时间戳来源 -> 时钟统一 -> 延迟修正 -> 样本插值”的顺序展开。这个顺序符合排查防抖抖动、反向补偿和相位滞后的工程路径。

## 1. 先问两个 timestamp 是否同源

时间同步第一问：

```text
gyro_ts 和 frame_ts 是否来自同一个 clock source
```

这里的 `clock source` 可以先理解成“用哪块表计时”。如果 gyro 用的是一块表，video 用的是另一块表，即使两个 timestamp 都是数字，也不能直接相减。

```text
gyro_ts = 1000000                   // gyro 那块表上的 1000000
frame_ts = 1000000                  // video 那块表上的 1000000
```

这两个数看起来一样，但如果两块表的起点、速度或单位不同，它们就不是同一个真实时刻。

如果不是同源，就要处理：

- 时钟偏移
- 时钟漂移
- 单位换算
- 采样时刻和记录时刻的差异

常见风险：

- gyro timestamp 用 `ktime_get_boottime_ns()`
- video timestamp 用硬件 SOF 计数器
- 用户态又用 `CLOCK_MONOTONIC`
- 三者没有统一换算关系

## 2. frame timestamp 要映射到曝光中心

防抖算法更关心曝光期间的相机姿态，而不是 buffer 什么时候被用户态拿到。

这里的关键是：视频帧不是“瞬间生成”的。sensor 先曝光，再读出，再经过 ISP/DMA，最后 buffer 才完成。防抖真正想知道的是“画面被拍下来的那段时间，相机怎么动了”，不是“CPU 什么时候拿到这帧”。

```text
SOF timestamp                      // 一帧开始的时间，可能接近读出开始
-> exposure_time                   // 这一帧曝光持续多久
-> exposure_center                 // 曝光时间段的中间点，常当作整帧参考时刻
-> gyro samples around center       // 查找这个时间点附近相机怎么转
```

如果 frame timestamp 实际是 EOF 或 DMA done，需要再减去对应延迟。

## 3. time offset 是常见隐藏误差

即使两个时间戳来自同一时钟，也可能有固定延迟：

- gyro 硬件采样到驱动读取有延迟
- FIFO watermark 引入批量上报延迟
- sensor 曝光到 ISP buffer done 有延迟
- 算法线程处理有排队延迟

`time offset` 可以先理解成“gyro 看到动作”和“video 看到动作”之间固定差了几毫秒。

例如真实情况是：

```text
手在 100ms 时开始抖                  // 真实运动发生
gyro 在 100ms 记录到                 // gyro 很快看到
video buffer 在 120ms 才完成          // 图像链路晚 20ms 才把这一帧交出来
```

如果算法不知道这 20ms 差值，就可能拿“晚了一点”的 gyro 数据去补“早一点”的视频帧，效果就是慢半拍。

防抖效果上的表现：

- 补偿总是慢半拍
- 快速转动时画面反而更晃
- 静态还好，运动时边缘拉扯明显

## 4. gyro 样本需要插值或积分到帧时刻

gyro 采样率通常高于视频帧率。

“插值”可以理解成：目标时刻刚好落在两笔 gyro 数据中间时，估一个中间值。

```text
10.0ms gyro = 1.0                   // 前一笔样本
11.0ms gyro = 3.0                   // 后一笔样本
10.5ms 需要 gyro 值                 // 视频帧刚好要用中间时刻
-> 估成大约 2.0                     // 这就是最简单的线性插值理解
```

“积分”可以理解成：把很多小段“转得多快 × 持续多久”加起来，得到这段时间总共转了多少。

```text
frame exposure time range             // 当前帧或某一行对应的时间窗口
-> select gyro samples                 // 选择窗口内或附近样本
-> interpolate angular velocity        // 必要时插值到目标时刻
-> integrate angular velocity          // 得到姿态变化
```

简单整帧模型：

```text
frame center timestamp                 // 以帧中心作为姿态参考
-> integrate gyro between frames        // 得到相邻帧之间的旋转
-> generate frame-level transform       // 每帧使用一个补偿矩阵
```

rolling shutter 模型：

```text
line timestamp                          // 每一行都有自己的曝光时刻
-> integrate gyro for each line          // 得到逐行姿态
-> generate line-varying warp            // 使用按行变化的补偿
```

## 5. 时间同步调试方法

- 打印 gyro sample 间隔，确认采样率稳定
- 打印 frame timestamp 间隔，确认帧率稳定
- 同时记录 gyro 和视频 timestamp，画到同一时间轴
- 人为做一次快速水平转动，看补偿相位是提前还是滞后
- 调整 time offset，观察防抖效果是否明显改善
- 检查 FIFO 丢样和视频丢帧是否同时发生

## 本章边界

本章只讲时间对齐，不讲坐标系、相机内参和具体平滑算法。时间对齐正确后，后续标定和算法才有意义。

## 一句话总结

陀螺仪防抖里，timestamp 不是日志字段，而是把传感器运动和图像内容连接起来的主轴。
