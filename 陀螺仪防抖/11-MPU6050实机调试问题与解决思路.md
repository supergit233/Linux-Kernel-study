# MPU6050 实机调试问题与解决思路

## 学习目标

- 区分“控制器已启用”和“从设备已通信成功”这两个不同层级。
- 理解 MPU6050 不一定需要内核驱动，也可以通过 `/dev/i2c-X` 在用户态直读寄存器。
- 遇到 `read WHO_AM_I failed`、`I/O error`、`wait idle timeout`、`wait rx no empty timeout` 时，能按层级排查。
- 建立从硬件连线到应用测试程序的最小验证闭环。

## 导读

### 本章定位

这一章记录一次真实 MPU6050 上板调试过程。目标不是马上做视频防抖，而是先回答一个更基础的问题：

```text
板子是否真的能通过 I2C 读到 MPU6050 的寄存器？
```

只要这个问题还没验证通过，就不要急着改防抖算法、MP4 封装或视频链路。

### 关键结论

- `/dev/i2c-X` 存在，只说明 I2C 控制器驱动已经注册了 adapter。
- 设备树里某个 `i2c_busX status = "okay"`，只说明控制器节点启用了。
- MPU6050 是否在线，要看它是否在 `0x68` 或 `0x69` 地址 ACK，并且能读到 `WHO_AM_I(0x75)`。
- 对 MPU6050 做用户态直读时，不需要先写内核驱动，也不需要一定添加 `mpu6050@68` 设备树节点。
- 本次实机最终确认：MPU6050 在 `/dev/i2c-2`，有效地址是 `0x68`。
- 如果读 `WHO_AM_I` 或扫描地址时内核打印 `wait idle timeout`、`wait rx no empty timeout`，优先按 I2C 总线层或物理层问题排查。
- “配置目标 100Hz”不等于最终 CSV 一定有 100Hz；还要区分 MPU6050 芯片内部采样率、应用线程轮询周期、Linux 用户态定时精度和 I2C 读取耗时。

## 1. 先分清两个层级

### 控制器层

控制器层验证的是 SoC 里的 I2C 控制器是否工作：

```sh
ls /dev/i2c*
i2cdetect -l
ls /sys/bus/i2c/devices
```

如果能看到：

```text
/dev/i2c-0
/dev/i2c-1
/dev/i2c-2
```

说明 Linux 已经创建了 I2C adapter，但这还不能证明 MPU6050 接通了。

### 从设备层

从设备层验证的是 MPU6050 是否真的挂在总线上。最初排查时以 `/dev/i2c-1` 为例：

```sh
i2cdetect -y 1
i2cget -y 1 0x68 0x75
i2cget -y 1 0x69 0x75
```

命令含义：

- `i2cdetect -y 1`：扫描 `/dev/i2c-1` 这条总线，看哪些 I2C 地址有设备响应。
- `i2cget -y 1 0x68 0x75`：从 `/dev/i2c-1` 上地址 `0x68` 的设备读取 `WHO_AM_I(0x75)`。
- `i2cget -y 1 0x69 0x75`：同样读取 `WHO_AM_I(0x75)`，但测试 AD0 拉高时的地址 `0x69`。

成功时，`WHO_AM_I` 应该读到：

```text
0x68
```

如果读不到，问题通常还在 I2C 通信建立阶段，不要直接跳到防抖算法层。

本次实机后续确认目标总线不是 `/dev/i2c-1`，而是 `/dev/i2c-2`，所以最终验证命令应切换为：

```sh
i2cget -y 2 0x68 0x75
```

## 2. 为什么“不需要 MPU6050 驱动”也能测试

MPU6050 本质上是一个 I2C 从设备。Linux 下有两种使用方式：

| 方式 | 入口 | 是否需要设备树子节点 | 适合阶段 |
| --- | --- | --- | --- |
| 用户态直读 | `/dev/i2c-X` + `ioctl(I2C_SLAVE)` | 不需要 | 早期通信验证 |
| 内核 IIO 驱动 | `inv_mpu6050` + `/sys/bus/iio/devices/` | 需要 | 正式传感器框架接入 |

当前调试阶段优先选择用户态直读，因为它最小、直接、变量少：

```text
open("/dev/i2c-X")
-> ioctl(fd, I2C_SLAVE, 0x68)
-> read WHO_AM_I
-> write PWR_MGMT_1 = 0
-> read 0x3B 起始的 14 字节传感器数据
```

等用户态能稳定读数后，再考虑是否整理成内核 IIO 驱动或接入防抖主程序。

## 3. 典型错误：read WHO_AM_I failed

实机运行测试程序时出现：

```text
MPU6050 I2C test
device: /dev/i2c-1, addr: 0x68
bsp-i2c 11061000.i2c: wait idle timeout, RIS: 0x10, SR: 0xa0000
read WHO_AM_I failed: I/O error
```

这个现象说明：

- 程序已经打开了 `/dev/i2c-1`。
- 失败发生在第一次访问 MPU6050 寄存器时。
- 内核 I2C 控制器在等待总线空闲或完成传输时超时。
- 这不是“缺少 MPU6050 内核驱动”的典型表现。

优先怀疑：

- SDA/SCL 没有接到当前 I2C bus 对应的真实引脚。
- SDA/SCL 接反。
- GND 没共地。
- MPU6050 没有 3.3V 供电，或模块电平不匹配。
- SDA/SCL 没有上拉，或上拉太弱。
- 某根线被拉低，导致总线无法进入 idle。
- 设备地址不是 `0x68`，而是 AD0 拉高后的 `0x69`。
- 当前测试的 bus 不对。
- 400kHz 对飞线、弱上拉或面包板连接不稳定，频率改成100kHz.

一般可能跟下面的现象同时出现：

io复用i2c-2的命令
```text
bspmm 0x11130050 0x1103    I2C_SDA
bspmm 0x11130054 0x1103


0x11130044 0x1202
echo 11 > /sys/class/gpio/export
echo in > /sys/class/gpio/gpio11/direction
cat /sys/class/gpio/gpio11/value
```

## 4. 实机定位：从 i2c-1 切换到 i2c-2

### 现象

最初按 `/dev/i2c-1` 测试时失败，日志里出现：

```text
bsp-i2c 11061000.i2c: wait idle timeout, RIS: 0x10, SR: 0xa0000
read WHO_AM_I failed: I/O error
```

后续确认当前 MPU6050 实际使用的是 `i2c-2` 这组引脚，因此测试对象应从：

```text
/dev/i2c-1
i2c@11061000
&i2c_bus1
```

切换为：

```text
/dev/i2c-2
i2c@11062000
&i2c_bus2
```

设备树里 `i2c_bus2` 已经启用：

```dts
&i2c_bus2 {
    status = "okay";
    clock-frequency = <100000>;
};
```

但是执行：

```sh
i2cdetect -y 2
```

仍然没有直观看到 `0x68` 地址，甚至可能夹杂超时或 abort 日志。

### 关键验证命令

不要只依赖 `i2cdetect` 扫描结果，直接读取 MPU6050 的真实身份寄存器：

```sh
i2cget -y 2 0x68 0x75
```

实机返回：

```text
0x68
```

再测试另一个可能地址：

```sh
i2cget -y 2 0x69 0x75
```

实机返回失败：

```text
bsp-i2c 11062000.i2c: wait idle abort!, RIS: 0x611
i2cget: read failed: I/O error
```

### 结论

这个结果说明：

```text
MPU6050 已经在 /dev/i2c-2 总线上响应
当前有效地址是 0x68
AD0 当前等效为低电平
0x69 地址没有设备响应
```

其中：

```text
0x75 是 MPU6050 的 WHO_AM_I 寄存器
WHO_AM_I 正常默认值就是 0x68
```

因此，`i2cget -y 2 0x68 0x75` 返回 `0x68`，就是当前阶段最可靠的“设备在线”证据。

### 为什么 i2cdetect 没显示也不能否定设备在线

`i2cdetect` 是地址扫描工具，不是 MPU6050 的功能验证工具。它的探测方式可能和具体 I2C 控制器、设备响应习惯有关。部分控制器驱动对空地址扫描处理得比较重，扫到没有 ACK 的地址时可能打印 timeout/abort。

相比之下：

```sh
i2cget -y 2 0x68 0x75
```

是一次明确的寄存器读取：

```text
总线：i2c-2
地址：0x68
寄存器：0x75 WHO_AM_I
期望返回：0x68
```

这个命令比 `i2cdetect` 更接近真实业务访问方式，也更适合作为 MPU6050 是否通信成功的判断标准。

### 识别成功标准

最小成功标准：

```sh
i2cget -y 2 0x68 0x75
```

返回：

```text
0x68
```

更完整的成功标准：

```sh
# 唤醒 MPU6050
i2cset -y 2 0x68 0x6B 0x00

# 读回电源管理寄存器，确认写入成功
i2cget -y 2 0x68 0x6B

# 读取加速度和陀螺仪相关寄存器
i2cget -y 2 0x68 0x3B
i2cget -y 2 0x68 0x43
```

如果后续独立测试程序能输出：

```text
WHO_AM_I: 0x68
raw accel=(...)
val accel=(...)g temp=(...)C gyro=(...)dps
```

就说明从用户态到 MPU6050 的独立读数链路已经打通。

### 读数成功后怎么看 accel / gyro / temp

当前实机输出已经不是“只能读 WHO_AM_I”的阶段，而是能连续读到三类传感器数据：

```text
raw accel=(   504, -7100,-16744) temp= -2480 gyro=(   150,   -19,   146)
val accel=(  0.031, -0.433, -1.022)g temp= 29.24C gyro=(  1.145, -0.145,  1.115)dps
```

这两行要分层看：

| 数据 | 当前输出说明 | 对防抖的意义 |
| --- | --- | --- |
| `raw accel` / `val accel` | 原始加速度已经被换算成 `g`。静止时三轴合成值应大致接近 `1g`，哪个轴接近 `1g` 或 `-1g`，说明重力主要落在哪个方向 | 用来检查板子姿态、安装方向、加速度 scale 和轴向是否合理 |
| `raw gyro` / `val gyro` | 原始角速度已经被换算成 `dps`。如果板子静止不动，长期不接近 `0 dps` 的均值就是 gyro 零偏 | 这是防抖主输入，后续需要减 bias，再从 `dps` 转成 `rad/s` 进入积分 |
| `raw temp` / `temp` | 温度已经换算成摄氏度，当前约 `29.2C` | 用来记录当前零偏对应的温度，后续观察热机后 bias 是否变化 |

从这一组数据可以先得到几个判断：

- `gyro=(1.145, -0.145, 1.115)dps` 如果是在静止状态下采到的，X/Z 轴大约有 `1 dps` 级别零偏，后续不能直接拿来积分，必须先做静止均值并减掉。
- `accel=(0.031, -0.433, -1.022)g` 说明重力主要落在 Z 负方向，同时 Y 方向也有明显分量，板子大概率不是完全水平放置，或者安装方向需要记录清楚。
- `temp=29.24C` 当前只是辅助信息，但后续做长时间测试时应和 gyro bias 一起记录。

到这里仍然还没达到“可用于视频防抖”的标准，因为日志里还缺少防抖最关键的工程字段：

```text
timestamp_ns                         // 每笔 IMU 样本发生的时间
seq                                  // 样本序号，用于检查丢样和重复
sampling_interval                    // 相邻样本间隔，用于确认 ODR 是否稳定
```

也就是说，当前阶段可以认为“MPU6050 读数链路打通，单位换算大体可检查”，下一阶段才是“把 accel/gyro/temp 变成带时间戳、可对齐视频帧的 IMU 数据流”。

### Linux 设备模型里的“识别”与 I2C 总线在线不是一回事

当前阶段通过 `/dev/i2c-2` 直接访问 MPU6050，没有在设备树中添加 `mpu6050@68` 子节点，也没有绑定内核 IIO 驱动。因此：

```sh
ls /sys/bus/i2c/devices/
```

可能只看到：

```text
i2c-0  i2c-1  i2c-2
```

而不会自动出现：

```text
2-0068
```

这不代表总线上没有 MPU6050，只代表 Linux 还没有把它注册成一个内核 I2C client 设备。

如果将来要让内核设备模型识别它，才需要在设备树中增加类似节点：

```dts
&i2c_bus2 {
    #address-cells = <1>;
    #size-cells = <0>;

    mpu6050@68 {
        compatible = "invensense,mpu6050";
        reg = <0x68>;
    };
};
```

并确保内核中有对应驱动，例如 `inv_mpu6050`。但对当前用户态通信测试来说，这一步不是必需条件。

### 本次排查后的修正

后续独立测试程序应默认使用：

```text
/dev/i2c-2
addr 0x68
```

推荐运行：

```sh
./mpu6050_i2c_test /dev/i2c-2 0x68
```

若程序版本已改成默认 `/dev/i2c-2`，也可以直接运行：

```sh
./mpu6050_i2c_test
```

## 5. 典型错误：wait rx no empty timeout

执行 `i2cdetect -y 1` 或 `i2cdetect -y 2` 时，扫描输出中可能夹杂大量内核报错：

```text
-- bsp-i2c 11061000.i2c: wait rx no empty timeout, RIS: 0x10, SR: 0xa0000
--
40:
50: bsp-i2c 11061000.i2c: wait rx no empty timeout, RIS: 0x10, SR: 0xa0000
-- bsp-i2c 11061000.i2c: wait rx no empty timeout, RIS: 0x10, SR: 0xa0000
-- bsp-i2c 11061000.i2c: wait rx no empty timeout, RIS: 0x10, SR: 0xa0000
--
60:
70:
```

这条日志来自当前测试的 I2C 控制器。它的含义是：

```text
驱动等待 RX FIFO 变成非空，但一直没有收到可读数据，最终超时。
```

拆开理解：

- `rx`：receive，接收方向。
- `no empty`：不是空的，也就是 RX FIFO 里应该出现数据。
- `timeout`：超过等待时间仍然没有等到。

这个现象可以判断为：当前 bus 上没有从设备完成一次正常读响应，至少 MPU6050 没有在扫描过程中正常 ACK 并返回数据。

它比普通的地址无响应更值得关注。普通 NACK 可能只是某个地址没有设备，而大量 `wait rx no empty timeout` 更像是总线读过程没有正常结束，优先检查：

- SDA/SCL 是否接到了当前 I2C bus 的真实管脚。
- SDA/SCL 是否接反。
- SDA/SCL 空闲时是否都是高电平。
- 模块是否 3.3V 供电，GND 是否共地。
- SDA/SCL 是否缺少上拉或被外部拉低。
- 管脚复用是否真的切到了 I2C 功能。
- I2C 频率是否过高，可先降到 100kHz 验证。

当前项目中，判断 MPU6050 是否在线，不应只依赖 `i2cdetect`，而应优先看：

```sh
i2cget -y 2 0x68 0x75
```

是否返回：

```text
0x68
```

在达到这个结果前，问题还停留在 I2C 建链阶段，不应该继续接入防抖主流程。

## 6. 典型错误：目标 100Hz 但 CSV 只有约 50Hz

### 调试日期

```text
2026-05-14
```

### 起因

在 `venc_rtsp_mp4_gyro` 项目中，视频主链路保持不变，额外增加了 MPU6050 数据采集线程和三份 CSV：

```text
mpu6050_imu.csv     IMU 采样时间线
video_pts.csv       VENC 每帧输出时间线
mpp_pts_map.csv     MPP PTS 与 CLOCK_MONOTONIC 的映射时间线
```

原始设计里，IMU 采集线程默认使用：

```c
SAMPLE_IMU_INTERVAL_US_DEFAULT = 10000
```

直觉上 `10000us = 10ms`，应该得到：

```text
1s / 10ms = 100Hz
```

但是实测 CSV 只有约 50Hz，因此需要判断问题到底出在：

```text
MPU6050 芯片内部采样率
I2C 读取速度
CSV 写入
应用线程 sleep 调度
还是视频编码主链路干扰
```

### 现象

60 秒长测试中，视频和 IMU 同时运行，结果如下：

```text
H.265: 1796 frames, fps=30.00
H.264: 1795 frames, fps=30.00
IMU:   2987 rows, freq=49.30Hz
MPP map: 3009 rows, freq=49.64Hz
```

关键现象不是“IMU 单独慢”，而是：

```text
mpu6050_imu.csv 约 50Hz
mpp_pts_map.csv 约 50Hz
video_pts.csv 仍然稳定 30fps
```

其中 `mpp_pts_map.csv` 线程不读 I2C，只是周期性调用：

```c
ss_mpi_sys_get_cur_pts()
```

所以如果 IMU 和 MPP map 都从目标 100Hz 变成约 50Hz，就不能优先怀疑 MPU6050 芯片本身，也不能优先怀疑 I2C 总线。

### 初始代码问题

当时 MPU6050 采集线程的主体逻辑是：

```text
读当前时间
读 MPU6050 14 字节数据
写一行 CSV
usleep(interval_us)
```

也就是相对延时：

```c
usleep(ctx->interval_us);
```

这有两个问题。

第一，`usleep(10000)` 只是告诉内核“至少睡 10ms 左右”，不是硬实时定时器。线程什么时候真正再次被调度，取决于内核 tick、调度器、系统负载和当前平台的定时精度。

第二，循环体本身也要耗时。实际周期不是：

```text
sleep interval
```

而是：

```text
I2C read + CSV write + flush判断 + 线程调度 + sleep interval
```

另外，早期代码只唤醒 MPU6050：

```c
PWR_MGMT_1 = 0x00
```

还没有显式配置：

```text
SMPLRT_DIV
CONFIG
GYRO_CONFIG
ACCEL_CONFIG
```

所以当时“100Hz”其实只是应用层轮询目标，并不是完整的 MPU6050 芯片配置。

### 第一轮验证：把轮询周期改成 5000us

为了区分“芯片慢”还是“线程睡眠慢”，不先改芯片，只把两个应用线程的环境变量改成：

```sh
SAMPLE_IMU_INTERVAL_US=5000
SAMPLE_MPP_PTS_MAP_INTERVAL_US=5000
```

20 秒测试结果：

```text
IMU:     freq=97.63Hz
MPP map: freq=99.23Hz
video:   fps=30.00
```

这个结果说明：

- MPU6050/I2C/CSV 链路具备接近 100Hz 的输出能力。
- MPP map 线程也能接近 100Hz，说明不是视频主链路把所有线程压死。
- `10000us` 得到约 50Hz，`5000us` 得到约 100Hz，问题更像是当前板子用户态 sleep 被调度粒度向上取整。

这一步很关键：如果只看 IMU 50Hz，很容易误判为 MPU6050 配置错了；但 MPP map 也 50Hz，把怀疑范围明显推向了用户态定时。

### 第二轮措施：改成绝对时间调度

为了减少相对 `usleep()` 的累计误差，把 IMU 线程和 MPP map 线程改成基于 `CLOCK_MONOTONIC` 的绝对时间调度：

```c
clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next_ts, NULL);
```

核心思想是：

```text
不是“每次处理完再睡 interval”
而是“下一次目标时间固定是上一目标时间 + interval”
```

这样可以避免循环体耗时不断累加到采样周期里。

但是实测发现，在当前板子上，即使改成绝对时间调度：

```text
interval=10000us
```

仍然约为：

```text
IMU:     freq=49.91Hz
MPP map: freq=50.04Hz
```

这说明当前平台用户态定时很可能存在 10ms 级别的调度粒度或 tick 取整现象。请求 10ms 时，实际经常被调度成约 20ms；请求 5ms 时，实际落到约 10ms。

### 第三轮措施：显式配置 MPU6050 芯片内部采样率

为了让“目标 100Hz”不只停留在应用层，还需要显式配置 MPU6050：

```text
PWR_MGMT_1    = 0x01  使用 X gyro PLL 作为时钟源
CONFIG        = 0x02  DLPF 约 94Hz
SMPLRT_DIV    = 9     1kHz / (9 + 1) = 100Hz
GYRO_CONFIG   = 0x00  +-250 dps
ACCEL_CONFIG  = 0x00  +-2g
```

这里要注意 MPU6050 的采样率关系：

```text
Sample Rate = Gyroscope Output Rate / (1 + SMPLRT_DIV)
```

当 DLPF 开启时，gyro output rate 通常按 1kHz 计算，所以：

```text
1kHz / (1 + 9) = 100Hz
```

这一步解决的是“芯片内部数据更新率”的问题；用户态线程还需要按合适周期去读。

### 最终修正

最终代码采取两层修正：

```text
1. MPU6050 芯片内部显式配置为 100Hz
2. 应用层默认轮询间隔改为 5000us
```

之所以应用层不是默认 `10000us`，是因为当前板子上：

```text
10000us 请求 -> 实测约 50Hz
5000us 请求  -> 实测约 98-100Hz
```

也就是说，应用层写 `5000us` 并不是想让 MPU6050 输出 200Hz，而是为了抵消当前系统用户态调度取整后实际落到约 10ms 的现象。

修改后默认日志能看到：

```text
mpp pts map csv: ... interval=5000us
[imu] capture started: ... interval=5000us ...
```

默认短测试输出行数接近 100Hz 量级：

```text
mpp_pts_map.csv 约 1250 行
mpu6050_imu.csv 约 1240 行
video_pts.csv 约 706 行
```

这次短测包含程序启动、交互和退出开销，所以不能只用 `行数 / sleep秒数` 做严格频率判断；正式判断仍然以离线脚本统计的 `monotonic_ns` 相邻间隔为准。

#### 1. 先读现有代码，确认“100Hz”到底在哪里

测试结果：


为什么这个结果重要：

```text
IMU 能接近 100Hz，说明 I2C 读 14 字节和 CSV 写入不是主要瓶颈。
MPP map 也接近 100Hz，说明非 I2C 周期线程也能到 100Hz。
所以“10000us -> 50Hz”的主要嫌疑转向用户态 sleep 调度。
```

#### 3. 修改线程调度方式：从相对 sleep 改成绝对时间

然后修改了两个线程：

```text
sample_imu_mpu6050.c 的 IMU 采样线程
sample_venc.c 的 mpp_pts_map 线程
```

把原来的相对延时：

```c
usleep(interval_us);
```

改成基于 `CLOCK_MONOTONIC` 的绝对时间调度：

```c
clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next_ts, NULL);
```

为什么这么改：

```text
相对 sleep 的实际周期 = 处理耗时 + sleep 时间 + 调度延迟。
绝对时间调度可以减少处理耗时造成的累计漂移。
```

这一步主要是工程上更正确，避免长期运行时采样时间不断漂。

### 结论

这次问题的根因不是单一的“MPU6050 设置错了”，而是两层问题叠加：


### 经验教训

遇到“配置 100Hz 但实测只有 50Hz”时，不要只看配置值，要把链路拆成三层：

```text
传感器内部数据更新率
应用线程实际醒来频率
CSV/日志记录出来的时间戳频率
```

排查时建议同时放一个“不读 I2C 的周期线程”做对照。本项目里的 `mpp_pts_map.csv` 就起到了这个作用：

```text
如果 IMU 慢，MPP map 不慢 -> 优先查 I2C 或 MPU6050。
如果 IMU 慢，MPP map 也慢 -> 优先查用户态定时/调度。
```

后续如果要做更严肃的 Gyro EIS，最好不要长期依赖用户态 sleep 轮询。更稳的方向是：

```text
MPU6050 Data Ready 中断
或 MPU6050 FIFO
或内核 IIO/input 驱动
或更高精度的实时调度策略
```

但对当前阶段的离线对齐验证来说，默认 `5000us` 轮询已经可以得到接近 100Hz 的 IMU CSV，足够继续验证视频帧与 IMU 数据的时间对齐关系。

## 7. 推荐排查顺序

### 第一步：只确认控制器

```sh
i2cdetect -l
ls -l /dev/i2c-2
```

如果 `/dev/i2c-2` 不存在，先回到设备树、内核配置和控制器驱动。

如果 `/dev/i2c-2` 存在，继续下一步。

### 第二步：确认总线空闲电平

用万用表或示波器看 SDA/SCL 空闲电平：

```text
SDA idle: 高电平
SCL idle: 高电平
```

如果任意一根线长期为低，先查接线、短路、上拉和模块供电。

### 第三步：确认地址

MPU6050 地址由 AD0 决定：

```text
AD0 = 0 -> 0x68
AD0 = 1 -> 0x69
```

测试时两个地址都要试：

```sh
./mpu6050_i2c_test /dev/i2c-2 0x68
./mpu6050_i2c_test /dev/i2c-2 0x69
```

当前实机结论是 `0x68` 有效，`0x69` 无响应。

### 第四步：降低 I2C 频率

如果设备树里当前是：

```dts
clock-frequency = <400000>;
```

可以先临时降到：

```dts
clock-frequency = <100000>;
```

这样能排除飞线、上拉和信号质量导致的 400kHz 不稳定问题。

### 第五步：使用 repeated-start 读寄存器

用户态测试程序建议使用 `I2C_RDWR`：

```text
msg0: 写 1 字节寄存器地址
msg1: 读 N 字节寄存器数据
```

这样中间是 repeated-start，比简单 `write()+read()` 更接近传感器手册里的标准寄存器读取方式。

如果 `I2C_RDWR` 仍然报 `wait idle timeout`，就更应该回到总线物理层排查。

## 8. 现象到原因速查表

| 现象 | 更可能的层级 | 下一步 |
| --- | --- | --- |
| 没有 `/dev/i2c-2` | 控制器/设备树/内核配置 | 查 `status`、I2C 驱动、`CONFIG_I2C_CHARDEV` |
| `/dev/i2c-2` 存在但 `i2cget` 失败 | 从设备通信 | 查接线、地址、上拉、供电 |
| `wait idle timeout` | 总线状态/物理层 | 查 SDA/SCL 是否被拉低 |
| `wait rx no empty timeout` | 读过程未收到数据 | 查从设备 ACK、SDA/SCL、上拉、管脚复用 |
| `WHO_AM_I` 不是 `0x68` | 地址或读法异常 | 试 `0x69`，检查 repeated-start |
| 能读 `WHO_AM_I`，但数据全 0 或不变 | 初始化或采样 | 写 `PWR_MGMT_1=0`，再读 14 字节 |
| 静止时 gyro 值一直很大 | 零偏/量程/单位 | 做静止均值，后续用于 bias 校准 |
| 目标 100Hz 但 CSV 约 50Hz | 用户态 sleep 调度/定时精度 | 用无 I2C 对照线程验证，尝试绝对时间调度和更短轮询间隔 |
| 只改 `SAMPLE_IMU_INTERVAL_US` 仍不稳 | 芯片内部采样率未配置或用户态轮询不可靠 | 配置 `SMPLRT_DIV/CONFIG`，后续考虑 Data Ready/FIFO |

## 9. 建议的最小验证闭环

```text
确认 /dev/i2c-2 存在
-> 确认 SDA/SCL 空闲高电平
-> 试 0x68 和 0x69
-> 读 WHO_AM_I = 0x68
-> 写 PWR_MGMT_1 = 0
-> 配置 CONFIG / SMPLRT_DIV / GYRO_CONFIG / ACCEL_CONFIG
-> 连续读 0x3B 起始 14 字节
-> 静止观察 accel/gyro/temp 是否合理
-> 用 monotonic_ns 统计真实采样频率
```

只有这个闭环通过后，才进入下一阶段：

```text
MPU6050 独立读数
-> 数据加时间戳
-> 静止零偏估计
-> 与视频帧时间戳对齐
-> 防抖算法验证
-> 接入实时视频链路
```

## 10. 调试原则

- 一次只改变一个变量，例如只换地址、只换 bus、只降频率。
- 先验证 `WHO_AM_I`，再验证传感器数据。
- 先独立测试 MPU6050，不要一开始就接入 `sample_venc.c`。
- 先解决 I2C 通信，再讨论防抖算法。
- 日志里同时记录 bus、addr、寄存器、错误码和内核 dmesg。
- 采样频率不要只看配置值，要用时间戳相邻间隔统计真实频率。
- 对频率问题要增加对照线程，避免把用户态调度问题误判成传感器问题。

## 回答的问题

- 为什么 `/dev/i2c-1` 存在但 MPU6050 仍然可能读不到。
- 为什么当前实机应切换到 `/dev/i2c-2`。
- 为什么用户态直读不需要 MPU6050 内核驱动。
- `wait idle timeout` 应该优先从哪里排查。
- `wait rx no empty timeout` 表示 I2C 读过程没有收到数据。
- 为什么 `WHO_AM_I=0x68` 是第一个成功标准。
- 为什么 `i2cdetect` 没显示也不能否定 MPU6050 在线。
- 为什么当前应以 `i2cget -y 2 0x68 0x75` 返回 `0x68` 作为识别成功标准。
- 为什么 `/sys/bus/i2c/devices/` 没有 `2-0068` 不等于总线上没有 MPU6050。
- 什么时候才应该把 gyro 数据接入防抖视频链路。
- 为什么 `SAMPLE_IMU_INTERVAL_US=10000` 在当前板子上会实测成约 50Hz。
- 为什么最终默认用 `5000us` 轮询来获得接近 100Hz 的 CSV。
