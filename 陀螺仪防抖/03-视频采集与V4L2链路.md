# 视频采集与 V4L2 链路

## 导读

### 本章定位

这一章讲视频帧如何从 sensor/CSI/ISP 进入 V4L2 buffer。陀螺仪防抖不是只需要图像数据，还需要知道每帧图像对应的准确时间。

### 核心对象

- image sensor
- CSI/MIPI 接收器
- ISP
- media graph
- `struct video_device`
- `struct vb2_queue`
- `struct vb2_buffer`
- V4L2 buffer timestamp

### 关键函数

- `VIDIOC_S_FMT`
- `VIDIOC_REQBUFS`
- `VIDIOC_QBUF`
- `VIDIOC_STREAMON`
- `VIDIOC_DQBUF`
- sensor `s_stream`
- vb2 `buf_queue`
- frame done IRQ / ISP done callback

### 主流程

```text
配置 sensor / ISP / video node          // 设置分辨率、格式、帧率和 pipeline
-> reqbuf / qbuf                         // 用户态或中间件准备视频 buffer
-> streamon                              // 启动 sensor、CSI、ISP 和 DMA
-> frame start / exposure                // sensor 开始一帧曝光和读出
-> frame done / DMA done                 // ISP 或 DMA 完成当前帧写入
-> vb2_buffer_done                       // V4L2 buffer 完成并带 timestamp 返回
-> dqbuf                                 // 用户态或算法层拿到一帧图像
```

## 这一章按什么逻辑展开

本章按 V4L2 视频采集生命周期展开：先配置 pipeline，再准备 buffer，再启动 stream，最后分析 timestamp 在哪个时刻被填充。

## 1. 视频链路不是单个节点

真实摄像头链路通常是多个模块串起来：

```text
image sensor                     // 曝光和输出 RAW/YUV 数据
-> MIPI CSI receiver              // 接收 sensor 输出的数据流
-> ISP                            // 做去噪、AE/AWB、色彩、缩放等处理
-> video node                     // 通过 V4L2 把帧交给用户态或算法模块
```

如果系统使用 media controller，还要看：

- entity
- pad
- link
- subdev
- video node

防抖项目里，这条链决定两个关键问题：

- 图像什么时候曝光
- buffer timestamp 到底表示哪个阶段的时间

## 2. V4L2 buffer 是算法看到的视频载体

典型采集路径：

```text
VIDIOC_REQBUFS                  // 申请一批装视频帧的 buffer
-> VIDIOC_QBUF                  // 把空 buffer 交给驱动，等硬件往里写图像
-> VIDIOC_STREAMON              // 启动视频流，让 sensor/CSI/ISP 开始工作
-> vb2 buf_queue                // 驱动把 buffer 放入 DMA 等待队列
-> frame done IRQ               // 硬件通知“一帧已经写完”
-> vb2_buffer_done              // 驱动告诉 V4L2：这个 buffer 可以取走了
-> VIDIOC_DQBUF                 // 用户态或算法层取出这一帧图像和 timestamp
```

这里的 buffer 可以先理解成“给硬件装一帧图像的盒子”。用户态先把空盒子排队给驱动，硬件把图像写进去，写完后驱动再把装满的盒子还给用户态。

对防抖来说，`VIDIOC_DQBUF` 拿到的 buffer 至少要关注：

| 字段 | 人话理解 | 为什么防抖需要它 |
| --- | --- | --- |
| `sequence` | 第几帧 | 用来检查是否丢帧，以及和 gyro 日志对齐 |
| `timestamp` | 这一帧对应的时间 | 用来找这帧曝光时相机怎么动 |
| `bytesused` | 这个 buffer 里实际写了多少数据 | 判断帧数据是否完整 |
| `field` | 隔行/逐行等场信息 | 普通逐行视频通常不用太纠结，但异常时要确认 |
| metadata | 额外帧信息，例如曝光、增益、SOF/EOF 时间 | 防抖常需要 exposure、frame duration 或更准确时间 |
| 图像格式和 stride | 像素怎么排列、每行占多少字节 | warp/crop 时要按正确格式处理图像 |

## 3. timestamp 不能只看字段名

V4L2 buffer 有 timestamp 字段，但它不一定等于图像曝光开始时间。

常见情况：

```text
SOF timestamp                    // start of frame，一帧开始，通常更接近曝光或读出开始
EOF timestamp                    // end of frame，一帧结束
DMA done timestamp               // DMA 把这一帧写进内存 buffer 的时刻
DQBUF timestamp                  // 用户态取到 buffer 的时刻，已经晚了，不适合当曝光时间
```

防抖最理想的时间点是每帧曝光期间的相机姿态。如果拿到的是 EOF 或 DMA done，就需要知道它和曝光中心之间的固定偏移。

## 4. rolling shutter 会改变问题形态

如果 sensor 是 rolling shutter，一帧内不同图像行不是同一时刻曝光。

简化理解：

```text
frame timestamp                   // 只能代表某个参考时刻
-> line readout time               // 每一行相对参考时刻有偏移
-> per-line gyro pose              // 每行可能对应不同相机姿态
-> rolling-shutter warp            // 防抖变换可能需要按行变化
```

第一版学习可以先按 global shutter 或整帧同一姿态理解，后续再扩展 rolling shutter。

## 5. 视频链路调试检查点

- sensor 是否 stream on
- 帧率是否稳定
- V4L2 buffer 是否连续返回
- `sequence` 是否跳变
- timestamp 是否单调
- timestamp 间隔是否接近帧周期
- timestamp 来源是 SOF、EOF 还是 done
- 是否有 metadata 能提供 exposure、gain、frame duration

## 本章边界

本章只讲视频帧和 timestamp 如何出来，不展开 gyro 如何采集，也不展开补偿算法如何 warp 图像。

## 一句话总结

视频链路对防抖最重要的产物不是“拿到一帧图像”本身，而是拿到一帧带可信时间含义的图像。
