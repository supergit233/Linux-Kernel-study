# 嵌入式实时落地

## 导读

### 本章定位

这一章讲离线防抖如何落到嵌入式实时系统。实时落地关注的不只是算法正确，还包括 buffer、线程、延迟、算力、带宽和 ISP/硬件 warp 能力。

### 核心对象

- IMU / gyro sampling thread / IRQ
- IMU ring buffer
- video capture thread
- stabilization algorithm context
- warp engine
- output buffer
- latency budget

### 关键函数

当前先按工程入口记录：

- IMU / gyro buffer dequeue
- V4L2 `DQBUF/QBUF`
- ISP frame done callback
- hardware warp submit
- output encoder/display submit

### 主流程

```text
启动 IMU 采样                       // 保存 accel/gyro/temp，其中 gyro 要保证高频、连续、低延迟
-> 启动 video stream                 // V4L2/ISP 开始输出帧
-> 每帧取 timestamp                  // 确定当前帧对应的曝光时间
-> 查找 gyro 窗口                    // 从 ring buffer 中取出对应 gyro 样本
-> 计算补偿参数                      // 算法线程输出 transform/crop
-> 提交 warp                         // CPU/GPU/DSP/ISP/硬件模块做图像变换
-> 输出编码或显示                    // 稳定后帧进入后级链路
```

## 这一章按什么逻辑展开

本章按实时 pipeline 展开：先保证数据不断流，再控制延迟，最后考虑硬件加速和异常回退。

## 1. 实时系统要有 ring buffer

gyro 采样频率通常远高于视频帧率，所以需要保存一段历史样本。工程上可以把一次 IMU 样本完整保存为 `accel/gyro/temp/timestamp/seq`，算法主线程主要查 `gyro`，调试和标定线程可以同时使用 `accel` 和 `temp`。

```text
IMU IRQ/thread                      // 持续写入 accel/gyro/temp sample
-> IMU ring buffer                  // 保存最近一段时间的数据和 timestamp
-> video frame arrives              // 当前帧到达
-> query samples by timestamp       // 按帧时间查找对应样本窗口
```

ring buffer 要检查：

- 容量是否覆盖最大延迟
- 写读是否加锁或无锁安全
- 是否统计丢样
- 是否能按 timestamp 快速查找
- 是否保留 accel/temp，方便后续排查安装方向和温漂

## 2. latency budget

实时防抖每帧都有时间预算。

```text
frame interval                      // 例如 30fps 约 33.3ms
-> video capture delay              // sensor/ISP/DMA 延迟
-> algorithm delay                  // gyro 查询、积分、平滑、变换计算
-> warp delay                       // 图像变换耗时
-> encode/display delay             // 输出链路延迟
```

如果使用未来帧做平滑，效果会更稳，但会引入算法延迟。实时预览场景通常不能接受太大延迟。

## 3. warp 放在哪里做

可选位置：

- CPU OpenCV/NEON
- GPU
- DSP/NPU
- ISP 内部 EIS/warp 模块
- 编码前的图像处理模块

选择时关注：

- 是否支持 YUV 格式
- 是否支持 homography 或 mesh
- 是否支持 rolling shutter 行变换
- 是否会额外拷贝 buffer
- 是否影响编码链路

## 4. 异常回退策略

实时系统必须处理异常：

```text
gyro 丢样                            // 当前帧防抖质量下降或跳过防抖
-> accel 异常                         // 可能是量程、安装方向或剧烈线性运动，需要标记样本质量
-> temp 快速变化                      // 观察 gyro bias 是否需要重新估计
-> frame timestamp 异常               // 不使用错误补偿
-> gyro/video 时间范围不重叠           // 当前帧回退为无补偿
-> warp 超时                          // 丢帧、降级或关闭防抖
```

不要让错误 gyro 数据持续污染姿态积分。发现异常时，应该能重置状态或重新估计 bias。

## 5. 嵌入式落地检查点

- IMU/gyro 采样线程是否被低优先级阻塞
- accel/gyro/temp 是否都带同一套 timestamp
- gyro 和 video 是否使用同一时钟源
- frame timestamp 是否来自 SOF/EOF/done
- 算法是否会额外复制大图像 buffer
- warp 是否能跟上帧率
- crop margin 是否足够
- 关闭防抖后视频链路是否稳定

## 本章边界

本章只讲工程落地框架，不绑定具体 SoC 的 ISP/EIS 模块。后续拿到平台源码后，应把本章每个入口映射到具体驱动和 HAL。

## 一句话总结

实时防抖的难点是让 IMU/gyro、video、算法和 warp 在同一时间预算内稳定闭环，而不是只让离线算法跑出一个好看的结果。
