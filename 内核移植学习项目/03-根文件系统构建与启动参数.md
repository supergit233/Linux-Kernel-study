# 根文件系统构建与启动参数

## 导读

### 本章定位

这一章看 Linux 内核起来以后，如何找到根文件系统、挂载根文件系统，并最终进入 `/sbin/init`。

### 核心对象

- `bootargs`
  - 传给内核的启动参数
- `root=`
  - 根文件系统位置
- `init=`
  - 首个用户态程序
- `initramfs / initrd`
  - 内存中的根文件系统形式
- `rootfs`
  - 磁盘或 flash 上的根文件系统内容

### 关键函数

- `parse_early_param()`
- `parse_args()`
- `prepare_namespace()`
- `mount_root()`
- `run_init_process()`

### 主流程

接收 `bootargs` -> 解析 `root=`、`console=`、`init=` -> 识别块设备或内存根文件系统 -> 挂载根文件系统 -> 执行 `init`。

## 新人先读：本章先认识这些概念

| 概念 | 先这样理解 |
| --- | --- |
| rootfs | 根文件系统，也就是 Linux 启动后看到的 `/` 目录内容 |
| VFS | Linux 的虚拟文件系统层，负责统一处理挂载和文件访问 |
| 块设备 | 像 eMMC、SD、硬盘分区这类按块读写的存储设备，常见名字如 `/dev/mmcblk0p2` |
| MTD | 面向 NAND/NOR flash 的存储抽象，不一定表现成普通块设备 |
| UBI / UBIFS | NAND flash 上常见的卷管理和文件系统组合 |
| `root=` | 告诉内核根文件系统在哪里 |
| `rootfstype=` | 告诉内核根文件系统是什么类型，例如 ext4、ubifs、squashfs |
| `init=` | 指定第一个用户态程序，不写时内核会尝试默认路径 |
| BusyBox | 嵌入式里常用的小型用户态工具集合，很多最小 rootfs 都靠它提供 shell 和基础命令 |
| 动态库 | 用户态程序运行时依赖的 `.so` 文件，缺失时程序存在也可能跑不起来 |

## 这一章按什么逻辑展开

这一章按“内核启动交接 -> 参数解析 -> rootfs 定位 -> 文件系统挂载 -> 执行 init”的顺序展开。

这样拆的原因是：根文件系统失败通常不是一个点的问题，而是 `bootargs`、存储驱动、块设备命名、文件系统类型、用户态内容多层共同决定的结果。

## 1. 根文件系统层真正解决什么

- 提供 `/sbin/init`
- 提供 `/dev`、`/proc`、`/sys`
- 提供 shell、动态库、基础命令
- 提供业务程序运行所需环境

## 2. 入口从哪里来：内核何时开始挂 rootfs

根文件系统挂载发生在内核主初始化后半段，入口来自上一章的 `kernel_init()` 主线：

```text
rest_init()                          // 创建 kernel_init 等内核线程
-> kernel_init()                     // 继续执行用户态启动前的内核初始化
-> kernel_init_freeable()            // 完成可释放初始化阶段
-> prepare_namespace()               // 根据 root= 等参数准备挂载根文件系统
-> mount_root()                      // 选择具体块设备、NFS、initramfs 或 flash 文件系统
-> run_init_process()                // 执行 /sbin/init、/etc/init、/bin/init 或 /bin/sh
```

因此，看到 `VFS: Cannot open root device` 时，说明内核已经跑到相当靠后的阶段，不应再按“内核完全没启动”处理。

## 3. bootargs 参数消费表

| 参数 | 来源/填充时机 | 内核消费结果 | 常见问题 |
| --- | --- | --- | --- |
| `console=` | U-Boot `bootargs` 或 DTS `chosen.bootargs` | 决定内核控制台输出到哪个串口/终端 | 系统在跑但看不到日志 |
| `root=` | U-Boot 启动参数 | 决定根文件系统设备、NFS 路径或 initramfs 行为 | `VFS: Cannot open root device` |
| `rootfstype=` | U-Boot 启动参数 | 指定 ext4、squashfs、ubifs、jffs2、nfs 等类型 | 文件系统驱动不匹配或未编译 |
| `rootwait` / `rootdelay=` | U-Boot 启动参数 | 等待慢速块设备出现 | eMMC/SD/USB 存储还没枚举完 |
| `init=` | U-Boot 启动参数 | 指定第一个用户态程序 | 路径不存在或程序不可执行 |
| `ubi.mtd=` / `ubi.block=` / `mtdparts=` | U-Boot 启动参数或板级分区设计 | 帮助 MTD/UBI/UBIFS 找到分区和卷 | flash 分区错、UBI attach 失败 |

调试 rootfs 时，必须同时看“参数是否正确”和“内核是否具备对应驱动”。例如 `root=/dev/mmcblk0p2` 正确，也要求 MMC host、分区解析、ext4 等路径都可用。

## 4. rootfs 三种挂载路线

实际移植时，rootfs 建议按三条路线理解，而不是只记“根文件系统”这个词。

| 路线 | rootfs 在哪里 | U-Boot 要做什么 | Linux 要做什么 | 适合阶段 |
| --- | --- | --- | --- | --- |
| initramfs / initrd | RAM 中，可能内置进内核，也可能由 U-Boot 额外加载 | 加载 kernel、`dtb`，外部 initrd 还要加载 `rootfs.cpio.gz` | 解包内存 rootfs，执行 `/init` 或 `rdinit=` | 最早期验证用户态 |
| NFS root | PC/NFS 服务器目录 | 加载 kernel、`dtb`，传 `root=/dev/nfs nfsroot=... ip=...` | 初始化网卡，通过网络挂载 rootfs | 开发调试 |
| 本地存储 rootfs | eMMC/SD/NAND/SPI flash 分区或 UBI 卷 | 加载 kernel、`dtb`，传 `root=`、`rootfstype=`、`rootwait` 等参数 | 初始化存储驱动，识别分区或 UBI 卷并挂载 | 产品落地 |

### 4.1 initramfs / initrd

这条路线的核心是：先不依赖本地存储，尽快验证内核能执行第一个用户态程序。

```text
U-Boot 加载 kernel + dtb              // 如果 initramfs 内置，只需要这两个
-> 可选加载 rootfs.cpio.gz            // 如果使用外部 initrd
-> bootargs 使用 rdinit=/init         // 指定内存 rootfs 里的 init
-> Linux 解包 initramfs               // 不需要等待 eMMC/NAND 分区
-> 执行 /init                         // 进入最小用户态
```

### 4.2 NFS root

这条路线的核心是：rootfs 放在 PC 上，方便频繁修改文件，不用反复烧录。

```text
U-Boot 加载 kernel + dtb              // rootfs 不由 U-Boot 搬到 RAM
-> bootargs 指定 root=/dev/nfs        // 告诉 Linux 根文件系统来自网络
-> bootargs 指定 nfsroot=server:path  // 告诉 Linux NFS 服务器和路径
-> Linux 初始化网卡                   // 网卡驱动必须内建或足够早可用
-> Linux 挂载 NFS root                // 通过网络进入完整 rootfs
```

### 4.3 本地存储 rootfs

这条路线的核心是：rootfs 放在板子自己的存储介质里，最接近最终产品启动方式。

```text
U-Boot 加载 kernel + dtb              // 通常不把整个 rootfs 搬进 RAM
-> bootargs 指定 root=/dev/mmcblk0p2  // eMMC/SD/ext4 这类块设备路线
-> 或指定 ubi.mtd/root=ubi0:rootfs    // NAND/UBI/UBIFS 这类 flash 路线
-> Linux 初始化存储控制器             // MMC、NAND、SPI flash 等驱动要可用
-> Linux 识别分区或 UBI 卷            // root= 指向的对象必须真实存在
-> mount_root() 挂载 rootfs           // 文件系统驱动必须内建或足够早可用
```

## 5. 最容易出的问题

- `VFS: Cannot open root device`
- 找不到 `/sbin/init`
- 动态库缺失
- 设备节点不完整
- `console=` 错误导致系统起来但没有输出

## 6. 关键参数

- `console=`
- `root=`
- `rootfstype=`
- `rootwait`
- `init=`
- `ubi.mtd=` / `ubi.block=` / `mtdparts=`

## 7. 全流程总结

```text
U-Boot bootargs                      // 上一层传入 console/root/init/rootfstype 等参数
-> parse_early_param() / parse_args() // 内核早期和普通参数解析
-> kernel_init()                     // 内核完成主初始化后准备进入用户态
-> prepare_namespace()               // 根据 root=、rootwait、rootfstype 准备 rootfs
-> rootfs 路线选择                   // 按 initramfs、NFS root、本地存储三类分开判断
   -> initramfs / initrd             // 解包内存 rootfs，重点检查 /init 和 cpio/initrd 格式
   -> NFS root                       // 初始化网卡并挂载服务器目录，重点检查 nfsroot= 和网络驱动
   -> 本地存储 rootfs                 // 等待块设备或 UBI 卷，重点检查 root=、rootfstype=、rootwait
-> mount_root()                      // 挂载 ext4/squashfs/ubifs/jffs2/nfs/initramfs 等
-> run_init_process()                // 尝试执行 init= 或默认 init 路径
-> /sbin/init                        // 第一个用户态进程启动，系统进入用户态
```

## 8. 一句话总结

根文件系统层决定“内核起来以后，系统到底能不能进入真正可交互的用户态”。
