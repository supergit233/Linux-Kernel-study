# 内核移植学习达标检查清单

## 学习目标

- 判断是否已经建立 `BootROM/U-Boot -> Linux -> rootfs -> init` 的完整启动链模型
- 判断是否能把镜像、`dtb`、`bootargs`、rootfs、`init` 等交接物放回正确层级
- 判断是否能解释 initramfs/initrd、NFS root、本地存储 rootfs 三条路线的适用阶段和排查边界
- 判断是否能按现象反推问题停在 U-Boot、Linux early、driver model、rootfs 还是用户态
- 判断是否具备独立整理新板 Bring-up 日志和排查路径的能力

## 导读

### 本章定位

本章是整个内核移植学习项目的验收章。它不是普通总结，而是用来检查学习完成后是否已经能独立画流程、解释交接物、定位失败阶段和设计排查顺序。

### 核心对象

- `U-Boot`
  - 负责早期硬件初始化、加载 image/`dtb`、组织 `bootargs`
- Linux 内核
  - 负责解析设备树、建立内存/中断/驱动模型、进入 rootfs 挂载阶段
- `rootfs`
  - 负责提供 `/sbin/init`、shell、动态库、设备节点和用户态基础环境，可能来自 initramfs、NFS 或本地存储
- `bootargs`
  - 连接 U-Boot、内核、rootfs 的关键参数桥梁

### 关键函数

- `board_init_f()` / `board_init_r()`
- `bootm` / `bootz`
- `start_kernel()`
- `setup_arch()`
- `do_initcalls()`
- `prepare_namespace()`
- `mount_root()`
- `run_init_process()`

### 主流程

```text
BootROM / reset                      // 上电后最早入口
-> U-Boot early                      // 串口、时钟、DDR、重定位
-> U-Boot runtime                    // 存储、环境变量、bootcmd、bootargs
-> bootm / bootz                     // image/dtb/initramfs 交给 Linux
-> start_kernel()                    // Linux 通用初始化开始
-> setup_arch() / parse_args()       // 消费 dtb、memory、bootargs
-> do_initcalls() / driver probe     // 驱动注册、设备匹配和 probe
-> prepare_namespace()               // 根据 root=、rootwait、nfsroot= 等准备 rootfs
   -> initramfs / initrd             // 内存 rootfs，优先验证能否执行第一个用户态程序
   -> NFS root                       // 网络 rootfs，适合开发阶段频繁改文件
   -> 本地存储 rootfs                 // eMMC/SD/NAND/SPI flash 等产品化路径
-> mount_root()                      // 挂载或切换到目标根文件系统
-> run_init_process()                // 启动第一个用户态进程
```

## 新人使用方式

如果是第一次学内核移植，不建议一开始就做完整自测。更稳的顺序是：

1. 先读 [[00-内核移植学习项目总览]] 的“最小概念词典”
2. 每读一章，先看该章的“新人先读”表格
3. 只要求先能说清“这个词属于哪一层”，不要急着背函数细节
4. 学完整套后，再回到本章做清单检查

新人阶段最重要的不是记住所有函数，而是能把概念放回层级：

| 层级 | 先记住哪些词 |
| --- | --- |
| U-Boot 前后 | BootROM、SPL/TPL、U-Boot、SRAM、DDR、bootcmd、bootargs |
| Linux 早期 | image、`dtb`、`start_kernel()`、`setup_arch()`、memory、console |
| 驱动模型 | DTS、platform device、platform driver、initcall、probe |
| rootfs | `root=`、`rootfstype=`、initramfs/initrd、NFS root、本地存储 rootfs、VFS、块设备、MTD/UBI、`init`、BusyBox |
| 排查方法 | 现象、层级、对象、主流程阶段、检查点 |

## 这一章按什么逻辑展开

本章按“达标效果 -> 必须知道的内容清单 -> 自检题 -> 不达标信号 -> 参考答案”的顺序展开。

这样拆的原因是：内核移植学习不能只看懂概念，还要能在真实日志里判断当前停在哪一层，并知道下一步应该查哪类对象。

## 1. 学完后应该达到的效果

### 1.1 能画出完整启动链

至少能不看笔记画出：

```text
BootROM
-> SPL/TPL 或 U-Boot
-> board_init_f()
-> DDR / clock / UART
-> board_init_r()
-> bootcmd / bootargs
-> bootm / bootz
-> start_kernel()
-> setup_arch()
-> do_initcalls()
-> prepare_namespace()
-> initramfs/initrd 或 NFS root 或 本地存储 rootfs
-> mount_root()
-> run_init_process()
-> /sbin/init
```

### 1.2 能解释每层交接物

需要能说明：

- U-Boot 交给 Linux 的是 image、`dtb`、`bootargs`、可选 initramfs
- Linux 早期消费 `dtb`、memory、`chosen.bootargs` 和命令行参数
- driver model 消费 DTS 节点、`compatible`、`reg`、`interrupts`、clock、reset 等资源
- rootfs 阶段消费 `root=`、`rootfstype=`、`rootwait`、`nfsroot=`、`rdinit=`、`init=` 和具体存储、网络或 initramfs 内容

### 1.3 能按现象归层

看到一个启动失败现象时，第一反应不是马上改驱动，而是先判断：

- 串口完全没输出：BootROM/U-Boot early
- U-Boot 正常但跳内核无输出：U-Boot 到 Linux 交接
- 内核 early panic：memory、timer、IRQ、DT early
- 驱动不 probe：DTS、Kconfig、driver model、initcall
- rootfs 挂载失败：bootargs、initramfs 内容、网络/NFS、块设备、文件系统、存储驱动
- 找不到 `init`：rootfs 内容、动态库、权限或 `init=` 参数

## 3. 能独立回答的问题

- U-Boot 能启动，为什么不代表 Linux 一定能启动
- Linux 有日志，为什么不代表系统已经进入用户态
- `dtb` 和 `bootargs` 分别在哪些阶段被消费
- `compatible` 错误为什么会导致 driver probe 不进
- `root=/dev/mmcblk0p2` 正确，为什么仍然可能挂载失败
- 新板移植为什么要先找最接近的参考板，而不是从空 DTS 和空 defconfig 开始
- 如何确认 U-Boot 实际加载的是新编译出来的 `dtb`
- `kernel_addr_r`、`fdt_addr_r` 这些加载地址为什么不能随便写
- `olddefconfig` 和 `menuconfig -> savedefconfig` 两条配置链路有什么区别
- 为什么推荐按 initramfs/initrd -> NFS root -> 本地存储 rootfs 的顺序调试
- `VFS: Cannot open root device` 和 `No init found` 的排查方向有什么不同
- 新板 Bring-up 为什么要先稳定串口、DDR、存储，再调网络、USB、WiFi

## 4. 不达标信号

出现下面情况，说明还需要回到对应章节补读：

- 只会说“启动失败”，不能判断停在哪一层
- 看到 `bootargs` 只知道是启动参数，说不清 `console=`、`root=`、`init=` 的消费位置
- 驱动不 probe 时，只看驱动 C 文件，不查 DTS、Kconfig、initcall 和模块加载
- 不会说明参考板 DTS、defconfig、DTS Makefile、最终 `dtb` 之间的关系
- rootfs 挂载失败时，不能按路线同时检查 root 参数、initramfs 内容、网络/NFS、存储驱动、块设备命名和文件系统类型
- 说不清 initramfs/initrd、NFS root、本地存储 rootfs 三条路线各自减少或增加了哪些变量
- 串口无输出时，仍然去改 Linux rootfs 或业务驱动
- 能进入 shell 后，就认为板级移植全部完成，忽略 IRQ、DMA、clock、regulator 等外设稳定性

## 6. 一句话标准

真正达标的标准是：看到一条启动日志或一个卡死现象时，能先判断它停在 `U-Boot -> Linux early -> driver model -> rootfs -> init -> 用户态服务` 的哪一段，再决定看哪份源码、哪段 DTS、哪组启动参数或哪份 rootfs 内容。

## 7. 参考答案

本节按第 2 节和第 3 节的问题顺序组织。复盘时先定位题目所属小节，再查看对应答案，避免把 U-Boot、内核、rootfs 和 Bring-up 问题混在一起。

### 7.1 对 2.1 U-Boot 清单的参考答案

- `board_init_f()` 偏早期初始化，常在重定位前或受限内存环境中完成串口、时钟、DDR 等基础工作；`board_init_r()` 发生在重定位后，继续初始化控制台、存储、环境变量、命令框架和后续 boot 流程。
- U-Boot 目录分工可以按“配置、板级、架构、驱动、环境、命令”理解：`configs/` 保存默认配置，`board/` 保存板级初始化和板级差异，`arch/` 保存 CPU/SoC 架构相关启动代码与 DTS，`drivers/` 保存串口、存储、网络等控制器驱动，`env/` 管理环境变量来源与保存方式，`cmd/boot*` 提供 `bootm`、`bootz` 等跳转内核命令。
- 串口、DDR、时钟、存储的验证顺序通常是先确认串口可输出，再确认 DDR 可稳定运行，然后确认关键时钟和复位配置正确，最后确认启动介质或下载介质可读写。这个顺序的意义是先拿到日志，再保证代码能可靠运行，再保证镜像能被重复加载。
- `bootcmd` 是 U-Boot 自动启动脚本，负责按顺序加载 kernel image、`dtb`、可选 initramfs/initrd，并执行 `bootz` 或 `bootm` 跳入 Linux。排查启动失败时，需要把 `bootcmd` 展开成一条条命令确认每一步实际读了什么文件、放到了什么地址。
- `bootargs` 是传给 Linux 的命令行参数，常见影响点包括 `console=`、`root=`、`rootfstype=`、`rootwait`、`init=`、`ip=`、`nfsroot=`。U-Boot 阶段只负责传递字符串，Linux 阶段才真正解析并执行这些策略。
- kernel image、`dtb`、initramfs/initrd 地址不能互相覆盖，因为跳转内核前它们都位于 DDR 中。覆盖 image 会导致内核解压或执行异常，覆盖 `dtb` 会导致设备树解析错误，覆盖 initramfs/initrd 会导致 rootfs 内容损坏。
- `kernel_addr_r`、`fdt_addr_r`、`ramdisk_addr_r` 通常来自 U-Boot 当前环境变量、板级默认环境或手动规划。检查时先用 `printenv kernel_addr_r fdt_addr_r ramdisk_addr_r` 看实际值，再用 `bdinfo` 确认 DDR 起始地址、大小和保留区域，最后确认各镜像大小相加后不会重叠。

### 7.2 对 2.2 Linux 内核清单的参考答案

- `bootm/bootz` 完成镜像检查、参数准备和跳转后，CPU 会进入 Linux 架构入口代码，随后经过解压、早期汇编入口和架构初始化，最终进入通用 C 入口 `start_kernel()`。学习时不需要一开始记住所有汇编细节，但必须知道 U-Boot 跳转后不是直接进入普通驱动，而是先进入内核启动主线。
- `setup_arch()` 和 `dtb`、memory 强相关，因为它会处理架构启动参数、设备树指针、内存布局、reserved-memory、machine 描述等早期信息。`memory` 节点、`chosen` 节点、保留内存或 `dtb` 版本错误，经常在这一阶段或后续早期内存初始化中暴露。
- `parse_args()` 消费来自 `bootargs` 的内核命令行参数。`console=` 影响串口日志输出，`root=` 和 `rootfstype=` 影响 rootfs 挂载目标，`init=` 影响第一个用户态程序路径，调试类参数则会改变日志等级或等待行为。
- `do_initcalls()` 负责按 initcall 等级调用内建驱动和子系统初始化函数。驱动注册发生在 initcall 中，设备树节点生成的 platform device 也在启动过程中被创建，二者满足匹配条件后才会进入对应 driver 的 `probe`。
- DTS 节点会在内核设备树解析和 platform 总线创建阶段变成 platform device；driver 侧通过 `of_match_table` 中的 `compatible` 与节点匹配。匹配成功后，platform core 才调用 driver 的 `probe`，驱动再读取 `reg`、`interrupts`、clock、reset、pinctrl 等资源。

### 7.3 对 2.3 rootfs 清单的参考答案

- `prepare_namespace()` 位于内核启动后半段，发生在驱动初始化和块设备发现之后、执行第一个用户态程序之前。它的核心任务是根据启动参数找到并挂载根文件系统。
- `root=` 指定根文件系统来源，`rootfstype=` 指定或约束文件系统类型，`rootwait` 让内核等待异步出现的块设备，`init=` 指定挂载 rootfs 后执行的第一个用户态程序。
- initramfs/initrd 适合早期验证内核到用户态的最短路径，NFS root 适合开发阶段频繁修改 rootfs，本地存储 rootfs 适合最终产品路线和量产启动验证。
- initramfs/initrd 路线下，U-Boot 加载 kernel、`dtb` 和可选 initrd，并传递必要的 `console=`、`rdinit=` 或 `init=`；NFS root 路线下，U-Boot 加载 kernel 和 `dtb`，并传 `root=/dev/nfs nfsroot=... ip=...`；本地存储 rootfs 路线下，U-Boot 加载 kernel 和 `dtb`，并传 `root=/dev/mmcblk...`、`rootfstype=`、`rootwait` 或 UBI/MTD 相关参数。
- initramfs/initrd 路线要求内核支持 initramfs 或 initrd 解包和最小用户态执行；NFS root 路线要求内建网卡驱动、PHY/MDIO、IP 自动配置、NFS client 和相关网络协议；本地存储 rootfs 路线要求内建存储控制器、分区解析、块设备、对应文件系统和必要的 rootfs 内容。
- `VFS: Cannot open root device` 通常说明内核已经跑到 rootfs 挂载阶段，但无法找到、打开或识别 `root=` 指向的设备或文件系统。排查方向应集中在启动参数、存储驱动、分区、文件系统类型和设备出现时序。
- `No init found` 和 rootfs 挂载失败不是同一类问题。前者通常表示 rootfs 已经挂载或至少已经进入查找 init 的阶段，但 `/sbin/init`、`/init`、`init=` 指定路径、执行权限、动态库或解释器存在问题。
- BusyBox 安装决定最小用户态命令是否存在，动态库决定动态链接程序能否执行，设备节点决定用户态能否访问控制台、存储、串口等设备。最小 rootfs 常见问题不是内核没起来，而是用户态文件内容不完整。

### 7.4 对 2.4 Bring-up 清单的参考答案

- 新板优先点亮串口和 DDR，是因为串口提供可观察日志，DDR 提供稳定运行空间。基础链路不稳定时直接调业务外设，会把日志缺失、内存异常、加载失败和外设问题混在一起。
- 选择参考板时应优先匹配 SoC 或同系列 SoC，其次匹配启动介质、DDR 类型、串口控制器、存储控制器、网络方案和 rootfs 路线。参考板越接近，需要同时改动的变量越少，问题归因越清楚。
- 复制 DTS 后，Linux 或 U-Boot 的设备树编译流程会消费它并生成 `dtb`；复制 defconfig 后，Kconfig 配置流程会消费它生成 `.config`；修改 DTS Makefile 后，构建系统才知道要把新 DTS 编译成目标 `dtb`。
- `olddefconfig` 适合在已有 defconfig 基础上非交互式补齐新增或缺省 Kconfig 选项；`menuconfig` 适合交互式打开、关闭或确认功能；`savedefconfig` 适合把当前 `.config` 中真正有效的差异沉淀回可维护的默认配置。
- 每个阶段的成功信号要具体：U-Boot 阶段看串口、DDR、存储加载和跳转命令；内核早期看解压、`start_kernel()` 日志、machine/model、console；驱动阶段看关键控制器 probe；rootfs 阶段看挂载成功和 init 执行；用户态阶段看 shell、服务或业务程序启动。
- “内核可启动”只表示 Linux 主线跑起来并能输出日志；“rootfs 可挂载”表示内核找到了并挂载了根文件系统；“业务外设可用”还要求对应驱动 probe、资源正确、用户态节点或服务正常。三者不是同一个验收点。
- DTS 资源错误要归到具体层级：`compatible` 错误影响匹配和 probe，`reg` 错误影响寄存器映射，`interrupts` 错误影响 IRQ，clock/reset 错误影响模块时钟和复位，pinctrl 错误影响引脚复用，regulator 错误影响供电依赖。

### 7.5 对 2.5 工程排查清单的参考答案

- 串口无输出时，优先查供电、串口线、波特率、复用、U-Boot 早期串口和 `console=`，不要直接查 rootfs。rootfs 只有在内核已经能挂载并进入用户态后才相关。
- 跳内核无输出时，优先查 image 格式、加载地址、`dtb` 地址、`bootz/bootm` 参数、`console=`、earlycon 和内核是否实际跳转成功。这个现象通常还没有进入 rootfs 阶段。
- 驱动不 probe 时，优先查 DTS 节点是否启用、`compatible` 是否匹配、Kconfig 是否编进内核或模块、initcall 是否执行、模块是否加载、依赖资源是否导致 defer probe。
- 中断不来时，按 DTS IRQ 描述、irqdomain 映射、`request_irq()` 返回值、触发类型、硬件中断线、电平/边沿状态和中断屏蔽寄存器顺序排查。不能只看驱动 handler 有没有写。
- rootfs 失败时，要先确认当前属于 initramfs/initrd、NFS root 还是本地存储 rootfs 路线，再同时检查启动参数、initramfs 内容、网络和 NFS 参数、存储控制器、块设备、分区、文件系统类型、`/sbin/init` 和动态库。

### 7.6 对第 3 节独立问题的参考答案

#### 7.6.1 U-Boot 能启动，为什么不代表 Linux 一定能启动

U-Boot 能启动只说明早期串口、DDR、启动介质读取和 U-Boot 自身命令环境基本可用。Linux 还依赖正确的 kernel image、`dtb`、`bootargs`、内存布局、控制台参数、驱动模型、rootfs 路线和用户态内容，所以 U-Boot 成功只是第一段链路成功。

#### 7.6.2 Linux 有日志，为什么不代表系统已经进入用户态

Linux 有日志说明内核已经开始运行，但系统进入用户态还需要完成驱动初始化、`prepare_namespace()` 挂载 rootfs，并由 `run_init_process()` 成功执行 `/sbin/init`、`/init` 或 `init=` 指定的程序。只看到内核日志不能等同于 shell 或业务系统已经启动。

#### 7.6.3 `dtb` 和 `bootargs` 分别在哪些阶段被消费

`dtb` 在内核早期被用于识别板级信息、内存、reserved-memory 和 chosen 节点，后续还会用于创建设备对象、解析 IRQ、clock、reset、pinctrl 等资源。`bootargs` 在内核命令行解析阶段被消费，影响 console、rootfs、init、网络启动和调试参数。

#### 7.6.4 `compatible` 错误为什么会导致 driver probe 不进

设备树节点变成 device 后，platform core 需要用节点里的 `compatible` 和 driver 的 `of_match_table` 做匹配。`compatible` 写错时，device 可能存在，但找不到对应 driver，结果就是注册了 driver 也不会进入该设备的 `probe`。

#### 7.6.5 `root=/dev/mmcblk0p2` 正确，为什么仍然可能挂载失败

`root=` 只指定目标路径，不保证目标设备已经出现或能被识别。MMC host 可能没 probe，分区可能没识别，缺少 `rootwait` 可能导致等待不够，ext4 等文件系统可能没内建，设备名可能变化，rootfs 内容或超级块也可能损坏。

#### 7.6.6 新板移植为什么要先找最接近的参考板，而不是从空 DTS 和空 defconfig 开始

最接近的参考板已经验证过 SoC 基础初始化、串口、时钟、存储、内核配置和部分 rootfs 路线。基于它修改可以一次只替换少量差异点；从空 DTS 和空 defconfig 开始会同时引入太多未知量，排查难度会迅速放大。

#### 7.6.7 如何确认 U-Boot 实际加载的是新编译出来的 `dtb`

可以从三处交叉验证：检查 U-Boot 加载命令中的文件名和地址是否指向新 `dtb`；用 `fdt addr ${fdt_addr_r}`、`fdt print /model` 或打印特定节点确认内存中的内容；再看 Linux 启动日志中的 machine/model 或新 DTS 中特意修改过的节点行为是否生效。

#### 7.6.8 `kernel_addr_r`、`fdt_addr_r` 这些加载地址为什么不能随便写

这些地址都指向 DDR 中的运行时加载位置。地址必须落在可用 DDR 范围内，避开 U-Boot 自身、堆栈、malloc 区、kernel image 解压区、`dtb`、initrd/initramfs 和 reserved-memory。随便写可能导致镜像互相覆盖，表现为解压失败、`dtb` 解析失败、随机 panic 或 rootfs 损坏。

#### 7.6.9 `olddefconfig` 和 `menuconfig -> savedefconfig` 两条配置链路有什么区别

`olddefconfig` 是非交互式复现链路，适合在已有 defconfig 基础上让 Kconfig 自动补齐新增或缺省选项。`menuconfig -> savedefconfig` 是交互修改和沉淀链路，适合打开某个功能后，把当前 `.config` 中真正有效的最小差异保存为新的默认配置。

#### 7.6.10 为什么推荐按 initramfs/initrd -> NFS root -> 本地存储 rootfs 的顺序调试

这个顺序能逐步增加变量。initramfs/initrd 先绕开网络和本地存储，验证内核能否进入最小用户态；NFS root 再加入网络、IP、NFS 和完整 rootfs；本地存储 rootfs 最后验证分区、文件系统、烧录、存储驱动和产品启动路径。

#### 7.6.11 `VFS: Cannot open root device` 和 `No init found` 的排查方向有什么不同

`VFS: Cannot open root device` 偏向 root 设备或文件系统挂载失败，应查 `root=`、`rootfstype=`、`rootwait`、存储驱动、分区和文件系统支持。`No init found` 偏向 rootfs 内容问题，应查 `/sbin/init`、`/init`、`init=` 路径、执行权限、动态库、解释器和 BusyBox 安装。

#### 7.6.12 新板 Bring-up 为什么要先稳定串口、DDR、存储，再调网络、USB、WiFi

串口提供日志，DDR 提供稳定运行空间，存储提供可重复加载和后续 rootfs 路线。网络、USB、WiFi 都依赖这些基础层；基础层不稳定时调业务外设，会让现象混合在一起，难以判断问题来自外设驱动、板级资源还是启动链路本身。
