# 内核移植学习达标检查清单

## 学习目标

- 判断是否已经建立 `BootROM/U-Boot -> Linux -> rootfs -> init` 的完整启动链模型
- 判断是否能把镜像、`dtb`、`bootargs`、rootfs、`init` 等交接物放回正确层级
- 判断是否能解释 initramfs/initrd、NFS root、本地存储 rootfs 三条路线的适用阶段和排查边界
- 判断是否能按现象反推问题停在 U-Boot、Linux early、driver model、rootfs 还是用户态
- 判断是否具备独立整理新板 Bring-up 日志和排查路径的能力

## 导读

### 本章定位

本章是整个内核移植学习项目的验收章。它不是普通总结，而是用来检查学习完成后是否已经能独立画流程、解释交接物、定位失败阶段和设计排查顺序。

### 核心对象

- `U-Boot`
  - 负责早期硬件初始化、加载 image/`dtb`、组织 `bootargs`
- Linux 内核
  - 负责解析设备树、建立内存/中断/驱动模型、进入 rootfs 挂载阶段
- `rootfs`
  - 负责提供 `/sbin/init`、shell、动态库、设备节点和用户态基础环境，可能来自 initramfs、NFS 或本地存储
- `bootargs`
  - 连接 U-Boot、内核、rootfs 的关键参数桥梁

### 关键函数

- `board_init_f()` / `board_init_r()`
- `bootm` / `bootz`
- `start_kernel()`
- `setup_arch()`
- `do_initcalls()`
- `prepare_namespace()`
- `mount_root()`
- `run_init_process()`

### 主流程

```text
BootROM / reset                      // 上电后最早入口
-> U-Boot early                      // 串口、时钟、DDR、重定位
-> U-Boot runtime                    // 存储、环境变量、bootcmd、bootargs
-> bootm / bootz                     // image/dtb/initramfs 交给 Linux
-> start_kernel()                    // Linux 通用初始化开始
-> setup_arch() / parse_args()       // 消费 dtb、memory、bootargs
-> do_initcalls() / driver probe     // 驱动注册、设备匹配和 probe
-> prepare_namespace()               // 根据 root=、rootwait、nfsroot= 等准备 rootfs
   -> initramfs / initrd             // 内存 rootfs，优先验证能否执行第一个用户态程序
   -> NFS root                       // 网络 rootfs，适合开发阶段频繁改文件
   -> 本地存储 rootfs                 // eMMC/SD/NAND/SPI flash 等产品化路径
-> mount_root()                      // 挂载或切换到目标根文件系统
-> run_init_process()                // 启动第一个用户态进程
```

## 新人使用方式

如果是第一次学内核移植，不建议一开始就做完整自测。更稳的顺序是：

1. 先读 [[00-内核移植学习项目总览]] 的“最小概念词典”
2. 每读一章，先看该章的“新人先读”表格
3. 只要求先能说清“这个词属于哪一层”，不要急着背函数细节
4. 学完整套后，再回到本章做清单检查

新人阶段最重要的不是记住所有函数，而是能把概念放回层级：

| 层级 | 先记住哪些词 |
| --- | --- |
| U-Boot 前后 | BootROM、SPL/TPL、U-Boot、SRAM、DDR、bootcmd、bootargs |
| Linux 早期 | image、`dtb`、`start_kernel()`、`setup_arch()`、memory、console |
| 驱动模型 | DTS、platform device、platform driver、initcall、probe |
| rootfs | `root=`、`rootfstype=`、initramfs/initrd、NFS root、本地存储 rootfs、VFS、块设备、MTD/UBI、`init`、BusyBox |
| 排查方法 | 现象、层级、对象、主流程阶段、检查点 |

## 这一章按什么逻辑展开

本章按“达标效果 -> 必须知道的内容清单 -> 自检题 -> 不达标信号 -> 参考答案”的顺序展开。

这样拆的原因是：内核移植学习不能只看懂概念，还要能在真实日志里判断当前停在哪一层，并知道下一步应该查哪类对象。

## 1. 学完后应该达到的效果

### 1.1 能画出完整启动链

至少能不看笔记画出：

```text
BootROM
-> SPL/TPL 或 U-Boot
-> board_init_f()
-> DDR / clock / UART
-> board_init_r()
-> bootcmd / bootargs
-> bootm / bootz
-> start_kernel()
-> setup_arch()
-> do_initcalls()
-> prepare_namespace()
-> initramfs/initrd 或 NFS root 或 本地存储 rootfs
-> mount_root()
-> run_init_process()
-> /sbin/init
```

### 1.2 能解释每层交接物

需要能说明：

- U-Boot 交给 Linux 的是 image、`dtb`、`bootargs`、可选 initramfs
- Linux 早期消费 `dtb`、memory、`chosen.bootargs` 和命令行参数
- driver model 消费 DTS 节点、`compatible`、`reg`、`interrupts`、clock、reset 等资源
- rootfs 阶段消费 `root=`、`rootfstype=`、`rootwait`、`nfsroot=`、`rdinit=`、`init=` 和具体存储、网络或 initramfs 内容

### 1.3 能按现象归层

看到一个启动失败现象时，第一反应不是马上改驱动，而是先判断：

- 串口完全没输出：BootROM/U-Boot early
- U-Boot 正常但跳内核无输出：U-Boot 到 Linux 交接
- 内核 early panic：memory、timer、IRQ、DT early
- 驱动不 probe：DTS、Kconfig、driver model、initcall
- rootfs 挂载失败：bootargs、initramfs 内容、网络/NFS、块设备、文件系统、存储驱动
- 找不到 `init`：rootfs 内容、动态库、权限或 `init=` 参数

## 2. 必须知道的内容清单

### 2.1 U-Boot 清单

- [ ] `board_init_f()` 和 `board_init_r()` 分别处在哪个阶段
- [ ] 串口、DDR、时钟、存储在 U-Boot 阶段的验证顺序
- [ ] `bootcmd` 如何加载 kernel image、`dtb`、initramfs
- [ ] `bootargs` 如何影响 Linux 控制台和 rootfs
- [ ] image、`dtb`、initramfs 地址为什么不能互相覆盖

### 2.2 Linux 内核清单

- [ ] `bootm/bootz` 跳内核后如何进入 `start_kernel()`
- [ ] `setup_arch()` 为什么和 `dtb`、memory 强相关
- [ ] `parse_args()` 如何消费 `bootargs`
- [ ] `do_initcalls()` 和驱动注册、probe 的关系
- [ ] DTS 节点如何变成 platform device 并匹配 driver

### 2.3 rootfs 清单

- [ ] `prepare_namespace()` 在启动链中的位置
- [ ] `root=`、`rootfstype=`、`rootwait`、`init=` 分别解决什么
- [ ] initramfs/initrd、NFS root、本地存储 rootfs 分别适合什么阶段
- [ ] 三条 rootfs 路线下，U-Boot 分别需要加载什么、传什么参数
- [ ] 三条 rootfs 路线下，Linux 分别依赖哪些内建驱动、协议或文件系统能力
- [ ] `VFS: Cannot open root device` 通常说明已经跑到哪个阶段
- [ ] `No init found` 和 rootfs 挂载失败不是同一类问题
- [ ] 动态库、设备节点、BusyBox 安装对用户态启动有什么影响

### 2.4 Bring-up 清单

- [ ] 新板优先点亮串口和 DDR，而不是直接调业务外设
- [ ] 能把每个阶段的成功信号说清楚
- [ ] 能区分“内核可启动”“rootfs 可挂载”“业务外设可用”
- [ ] 能把 DTS 资源错误归到具体层级，例如 `compatible`、`reg`、`interrupts`、clock、reset、pinctrl

### 2.5 工程排查清单

- [ ] 串口无输出时，不直接看 rootfs
- [ ] 跳内核无输出时，先查 image/`dtb`/`console=`
- [ ] 驱动不 probe 时，先查 DTS 节点、Kconfig、initcall 和模块加载
- [ ] 中断不来时，先查 DTS IRQ、irqdomain、request_irq 和硬件线
- [ ] rootfs 失败时，能按路线同时查启动参数、initramfs 内容、网络/NFS、存储驱动、块设备和文件系统类型

## 3. 能独立回答的问题

- U-Boot 能启动，为什么不代表 Linux 一定能启动
- Linux 有日志，为什么不代表系统已经进入用户态
- `dtb` 和 `bootargs` 分别在哪些阶段被消费
- `compatible` 错误为什么会导致 driver probe 不进
- `root=/dev/mmcblk0p2` 正确，为什么仍然可能挂载失败
- 为什么推荐按 initramfs/initrd -> NFS root -> 本地存储 rootfs 的顺序调试
- `VFS: Cannot open root device` 和 `No init found` 的排查方向有什么不同
- 新板 Bring-up 为什么要先稳定串口、DDR、存储，再调网络、USB、WiFi

## 4. 不达标信号

出现下面情况，说明还需要回到对应章节补读：

- 只会说“启动失败”，不能判断停在哪一层
- 看到 `bootargs` 只知道是启动参数，说不清 `console=`、`root=`、`init=` 的消费位置
- 驱动不 probe 时，只看驱动 C 文件，不查 DTS、Kconfig、initcall 和模块加载
- rootfs 挂载失败时，不能按路线同时检查 root 参数、initramfs 内容、网络/NFS、存储驱动、块设备命名和文件系统类型
- 说不清 initramfs/initrd、NFS root、本地存储 rootfs 三条路线各自减少或增加了哪些变量
- 串口无输出时，仍然去改 Linux rootfs 或业务驱动
- 能进入 shell 后，就认为板级移植全部完成，忽略 IRQ、DMA、clock、regulator 等外设稳定性

## 5. 达标复盘方式

1. 不看笔记，画出从 BootROM 到 `/sbin/init` 的完整链路
2. 对每个阶段写出“上一层交给下一层什么”
3. 随机选择三个现象，用 `现象 -> 层级 -> 对象 -> 主流程阶段 -> 检查点` 写排查路径
4. 回到 [[00-内核移植学习项目总览]] 检查总流程是否漏阶段
5. 回到 [[05-内核移植调试与工程问题问答]] 校准工程排查顺序

## 6. 一句话标准

真正达标的标准是：看到一条启动日志或一个卡死现象时，能先判断它停在 `U-Boot -> Linux early -> driver model -> rootfs -> init -> 用户态服务` 的哪一段，再决定看哪份源码、哪段 DTS、哪组启动参数或哪份 rootfs 内容。

## 7. 参考答案

### 7.1 内容清单参考答案

- `board_init_f()` 偏早期初始化，常在受限内存环境中完成串口、时钟、DDR 等基础工作；`board_init_r()` 发生在重定位后，初始化控制台、存储、环境变量和命令框架。
- `bootcmd` 负责执行加载 image、`dtb`、initramfs 和跳转内核的命令序列；`bootargs` 是传给 Linux 的参数字符串，影响 console、rootfs、init 和调试行为。
- `setup_arch()` 会消费架构相关参数、设备树和内存描述，所以 memory 节点、`dtb` 匹配、reserved-memory 等错误经常表现为 early panic 或随机异常。
- `do_initcalls()` 负责调用内建驱动初始化函数；驱动注册以后，设备树节点生成的 platform device 才能和 driver 的 `of_match_table` 匹配并进入 probe。
- initramfs/initrd 适合最早期验证用户态，U-Boot 只需要加载 kernel、`dtb` 和可选 initrd；NFS root 适合开发调试，U-Boot 传 `root=/dev/nfs nfsroot=... ip=...`，Linux 必须内建网卡和 NFS client；本地存储 rootfs 适合产品落地，U-Boot 传 `root=`、`rootfstype=`、`rootwait` 或 `ubi.mtd=`，Linux 必须内建对应存储和文件系统能力。
- `prepare_namespace()` / `mount_root()` 是 rootfs 挂载主线。`VFS: Cannot open root device` 表示内核已经跑到挂 rootfs 阶段，但没找到或打不开目标 root 设备。
- `run_init_process()` 负责执行第一个用户态程序。`No init found` 往往说明 rootfs 可能已经挂上了，但 `/sbin/init`、`init=` 指定路径、权限或动态库有问题。

### 7.2 独立问题参考答案

- U-Boot 能启动只说明早期串口、DDR、存储和加载命令基本可用；Linux 还需要正确 image、`dtb`、`bootargs`、内存布局、控制台和驱动模型。
- Linux 有日志只说明内核主线开始跑；系统进入用户态还依赖 `prepare_namespace()` 挂 rootfs 和 `run_init_process()` 执行 init。
- `dtb` 在 Linux 早期和 driver model 阶段被消费，影响 memory、chosen、platform device、IRQ、clock、reset 等；`bootargs` 被参数解析路径消费，影响 console、root、init 等运行策略。
- `compatible` 错误会导致设备树节点和 driver 的 `of_match_table` 对不上，platform device 可能存在，但具体 driver probe 不会进入。
- `root=/dev/mmcblk0p2` 正确仍可能失败，因为 MMC host 可能没 probe、分区没识别、rootwait 不够、ext4 未编译、设备名变化或 rootfs 内容损坏。
- 推荐按 initramfs/initrd -> NFS root -> 本地存储 rootfs 调试，是因为每一步只新增一类变量：initramfs 先绕开存储和网络，NFS root 再验证网络和完整用户态，本地存储最后验证分区、文件系统、烧录和量产启动路径。
- `VFS: Cannot open root device` 偏 root 设备和文件系统挂载问题；`No init found` 偏 rootfs 内容、init 路径、权限和动态库问题。
- 新板 Bring-up 先稳定串口、DDR、存储，是为了让后续每个问题都有可见日志、可靠运行内存和可重复加载路径；业务外设依赖这些基础层，不能倒着调。
