# 内核移植调试与工程问题问答

## 学习目标

- 把启动链常见故障现象映射回 `U-Boot`、内核、根文件系统三层
- 建立一套适合板级移植的固定排查顺序
- 形成面向工程落地的问题问答清单

## 导读

### 本章定位

这一章是“内核移植学习项目”的工程问答章，负责把启动链问题重新组织成可直接落地的排查入口。

### 核心对象

- `U-Boot`
  - 镜像加载、DDR、启动参数
- Linux 内核
  - 设备树、内存、中断、驱动模型
- 根文件系统
  - 挂载路径、`init`、用户态基础环境

### 关键函数

- `bootm / bootz`
- `start_kernel()`
- `panic()`
- `prepare_namespace()`
- `mount_root()`
- `run_init_process()`

### 主流程

现象 -> 判断停在哪一层 -> 找到对应入口函数 -> 再回到板级配置、镜像、参数和驱动。

## 新人先读：先学会看现象

| 概念 | 先这样理解 |
| --- | --- |
| 现象 | 串口看到的输出、卡死位置、panic、错误码，或者“完全没输出”这种外部表现 |
| 层级 | 问题大概停在哪一段，例如 U-Boot、Linux early、driver probe、rootfs、用户态 |
| 对象 | 当前最该检查的东西，例如串口、DDR、`dtb`、`bootargs`、块设备、`init` |
| 检查点 | 可以验证的具体项，例如地址是否覆盖、参数是否正确、DTS 节点是否 okay |
| panic | 内核遇到无法继续运行的问题后主动停止，并打印错误信息 |
| 卡死 | 没有继续输出，不一定等于真的死机，也可能是控制台参数错或日志没打开 |
| first bad point | 第一处异常点，比后面连锁错误更重要；排查时要尽量找最早异常日志 |

## 这一章按什么逻辑展开

这一章按“先分层，再定位对象，再回到主流程阶段和检查点”的逻辑展开。

这样拆的原因是：内核移植问题经常表现成同一句“启动不了”，但实际可能停在 BootROM、U-Boot、内核早期、驱动 probe、rootfs 挂载或用户态 init。先分层，才能避免在错误源码里越挖越深。

## 1. 常见问题分层

### 1.1 串口完全没输出

优先看：

- `U-Boot` 早期串口
- pinmux
- 时钟
- 串口基址

### 1.2 `U-Boot` 正常，但跳内核后无输出

优先看：

- 内核镜像地址
- `dtb` 地址
- `bootargs` 中 `console=`
- 内核解压入口是否正确

### 1.3 内核有输出，但 panic 在早期

优先看：

- 内存布局
- 设备树内存节点
- 中断控制器
- 定时器

### 1.4 内核起来，但挂载根文件系统失败

优先看：

- `root=`
- `rootfstype=`
- `rdinit=` / `nfsroot=` / `ubi.mtd=`
- 当前走的是 initramfs、NFS root 还是本地存储 rootfs
- 对应存储驱动
- 对应网络驱动和 NFS client
- 设备节点命名

### 1.5 进入 rootfs 后找不到 `init`

优先看：

- `/sbin/init`
- `init=`
- 动态库
- 根文件系统内容完整性

## 2. 工程排查模板

遇到现象时，先按下面模板归档：

```text
现象                                  // 看到的日志、卡死位置、panic 信息或串口表现
-> 层级                               // BootROM/U-Boot/Linux early/driver/rootfs/user space
-> 对象                               // 串口、DDR、image、dtb、bootargs、block device、init
-> 主流程阶段                         // 对应 00-04 章里的哪条流程
-> 检查点                             // 地址、参数、DTS、驱动配置、文件系统内容、日志开关
```

这套模板的作用是把“猜问题”变成“按证据归层”。

## 3. 固定排查顺序

1. 串口是否通
2. DDR 是否稳
3. `U-Boot` 是否能读镜像
4. 内核日志是否出现
5. 设备树是否匹配当前板子
6. 存储或网络驱动是否起来
7. `rootfs` 路线是否明确，initramfs、NFS root 或本地存储后端是否可用
8. `init` 是否存在并可执行

## 4. 常见现象定位表

| 现象 | 优先层级 | 重点对象 | 第一批检查点 |
| --- | --- | --- | --- |
| 串口完全没输出 | BootROM / U-Boot early | UART、pinmux、clock、DDR | 波特率、UART 基址、时钟、复位、启动介质 |
| U-Boot 能输出但随机死 | U-Boot / DDR | DDR、cache、重定位地址 | DDR 训练、内存测试、供电、重定位范围 |
| U-Boot 能读镜像但跳内核无输出 | U-Boot -> Linux 交接 | image、`dtb`、`bootargs`、console | image 格式、加载地址、`dtb` 地址、`console=` |
| 内核早期 panic | Linux early | memory、MMU、timer、interrupt-controller | `memory` 节点、reserved-memory、timer、GIC/中断控制器 |
| 驱动不 probe | driver model / DT | platform device、`compatible`、driver config | DTS `status`、`compatible`、Kconfig、initcall、模块加载 |
| probe 了但中断不来 | driver / IRQ | `interrupts`、IRQ controller、handler | DTS IRQ 属性、irqdomain、request_irq、硬件中断线 |
| rootfs 挂载失败 | rootfs | `root=`、initramfs、NFS root、块设备、文件系统驱动 | `rdinit=`、`nfsroot=`、`rootwait`、设备名、分区、`rootfstype`、存储/网络驱动 |
| 找不到 `init` | user space | `/sbin/init`、动态库、权限 | `init=`、可执行权限、动态链接器、BusyBox 安装 |

## 5. 全流程排查总结

```text
串口无输出                            // 先停在 BootROM/U-Boot early，不看 Linux rootfs
-> U-Boot 有输出但不稳                 // 看 DDR、时钟、复位、重定位和存储基础
-> U-Boot 能 bootm/bootz               // 看 image/dtb/bootargs 是否正确交给 Linux
-> Linux 有早期日志                    // 看 setup_arch、memory、timer、IRQ、console
-> driver model 开始工作               // 看 DTS 节点、compatible、initcall、probe
-> prepare_namespace()                 // 看 root=、rdinit=、nfsroot=、rootwait、rootfstype
-> mount_root()                        // 看 initramfs、NFS、分区、UBI、文件系统和后端驱动
-> run_init_process()                  // 看 /sbin/init、init=、动态库和 rootfs 内容
```

## 6. 一句话总结

内核移植问题的关键不是立刻深入某一层源码，而是先判断启动链到底停在 `U-Boot`、内核还是根文件系统。
