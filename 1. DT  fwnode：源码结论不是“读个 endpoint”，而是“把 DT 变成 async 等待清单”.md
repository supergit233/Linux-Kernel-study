# 1. DT / fwnode：源码结论不是“读个 endpoint”，而是“把 DT 变成 async 等待清单”

`rkisp1_subdev_notifier_register()` 里最关键的动作不是注册 notifier，而是先 `fwnode_graph_for_each_endpoint()` 遍历 **本设备的每个 endpoint**，然后根据 endpoint 父节点的 `reg` 区分 port 0 和 port 1：port 0 被当成 MIPI CSI-2，port 1 则要求 DT 里显式给出并解析成 parallel/BT656。接着驱动调用 `v4l2_fwnode_endpoint_parse()` 解析 bus type、lane、flags 等，再通过 `fwnode_graph_get_remote_endpoint()` 找远端 endpoint，并用 `v4l2_async_nf_add_fwnode()` 把远端加入 notifier 的等待列表，同时把 `port`、`mbus_type`、`mbus_flags`、`lanes` 存进 `rkisp1_sensor_async`。这说明在 5.10 里，**DT 的 endpoint 信息没有直接变成“立刻可用的 subdev 指针”，而是先被翻译成“将来要匹配的 async 描述符 + 链路参数缓存”**。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

配套看 5.10 的 `v4l2_fwnode_endpoint_parse()` / `v4l2_fwnode_parse_link()` 更清楚：前者会把 firmware endpoint 解析成 `v4l2_fwnode_endpoint`，自动区分 CSI1/CSI2/parallel/BT656 等总线类型，并填入 endpoint base 信息；后者则从本地 endpoint 出发，拿到 `local_port/local_id` 和远端 `remote_port/remote_id` 以及两端的父节点。也就是说，**fwnode 层在内核里的职责是：把“图上的边”和“边的总线属性”拆出来，供上层驱动做异步匹配和 link 建立**。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/v4l2-core/v4l2-fwnode.c))

**我对第 1 层的源码结论：**

- DT 在 5.10 平台 camera 驱动里，首先是**媒体拓扑描述**，其次才是配置来源。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/v4l2-core/v4l2-fwnode.c))
- `rkisp1` 不是简单“读 sensor 节点”，而是**从自己本地 endpoint 出发**去找 remote endpoint，再把它变成 async match 项。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- DT 里 bus/lane/flags 这些信息不会等到 stream on 才看，而是在 notifier 注册阶段就被缓存进 `rkisp1_sensor_async`，后面 `.bound()`/链路建立时直接用。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

# 2. probe 主容器：`rkisp1` 的真实顺序是“先资源，再容器，再内部实体，再等外部 sensor”

`rkisp1_probe()` 的顺序非常值得记。它先 `devm_kzalloc()` 分配 `rkisp1_device`，设置 `info` 和 `drvdata`，然后申请 IRQ、bulk clocks、打开 runtime PM、短暂 resume 读出硬件 `CIF_ID`。做完这些基础资源和硬件可达性检查之后，才进入 V4L2/media 世界：先 `v4l2_device_register()`，再 `media_device_register()`，然后按 feature 初始化内部 CSI，再调用 `rkisp1_entities_register()` 注册 ISP、resizer、capture、stats、params、CSI 等内部实体，最后才 `rkisp1_subdev_notifier_register()` 去等外部 subdev。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

这一顺序非常能说明驱动作者的设计意图：**probe 的目标不是“一步把整条 camera pipeline 带起来”，而是先把主设备的世界建完整，再把外部 sensor 作为异步边接进来。** 换句话说，在 `rkisp1` 里，主设备的内部 graph 是“先天存在”的，外部 sensor 是“后天绑定”的。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

`rkisp1_entities_register()` 的顺序也很有信息量：先 `rkisp1_isp_register()`，再 `rkisp1_resizer_devs_register()`，再 `rkisp1_capture_devs_register()`，再 stats、params，最后才是可选的 `rkisp1_csi_register()`，随后统一调用 `rkisp1_create_links()` 建内部链路。这个顺序说明它把 ISP 看成核心处理块，把 resizer/capture/stats/params 看成挂在 ISP 周边的内部节点；**也就是说，video node 不是起点，而是内部处理图的末端接口。**([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**我对第 2 层的源码结论：**

- `probe()` 的正确阅读法不是看“调了哪些 API”，而是看依赖顺序：**资源 → v4l2/media 容器 → 内部 entities/video nodes → async notifier**。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- `rkisp1` 明确把“内部图搭建”和“外部 sensor 绑定”拆成两个阶段，这对定位 probe/绑定类 bug 很关键。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- 从 remove 的反向顺序也能印证这个依赖：先 `v4l2_async_nf_unregister/cleanup()`，再 `entities_unregister()`，再 CSI cleanup，最后 `media_device_unregister()` 和 `v4l2_device_unregister()`。这说明对象所有权关系是非常清晰、严格反向拆除的。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

# 3. async notifier：`rkisp1` 的源码证明 `.bound()` 不是注册 subdev，而是“补上外部边”

先看 5.10 `v4l2-async.c` 的核心事实：`v4l2_async_match_notify()` 里，**先** `v4l2_device_register_subdev(v4l2_dev, sd)`，**再**调用 notifier 的 `.bound()`，然后把 `asd` 从 waiting 删掉、把 `sd` 移到 done list。`v4l2_async_notifier_try_complete()` 则明确要求 waiting 为空、整棵 notifier 树都 complete，才调用 `.complete()`。这意味着：**`.bound()` 运行时，subdev 已经属于主 `v4l2_device`，它不是“注册时机”，而是“装配时机”**。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/v4l2-core/v4l2-async.c))

对应到 `rkisp1_subdev_notifier_bound()`，它做的三件事很典型。第一，保存 `s_asd->sd = sd`。第二，通过 `media_entity_get_fwnode_pad()` 结合 `source_ep` 去找到 sensor 实体上真正的 source pad。第三，根据 `port` 决定接法：如果是 port 0，则调用 `rkisp1_csi_link_sensor()` 先把 sensor 接到内部 CSI；如果不是 port 0，就直接 `media_create_pad_link(sd -> isp sink)`。这说明 `rkisp1` 的 `.bound()` 逻辑不是抽象的，而是非常具体地把 **外部 sensor 的哪个 pad 接到内部哪个 pad** 补上。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235&utm_source=chatgpt.com))

`rkisp1_subdev_notifier_complete()` 更有意思：它没有再去注册 capture 节点，而是只调用 `v4l2_device_register_subdev_nodes()`。结合 5.10 `v4l2-device.c` 可知，这个函数的职责是给带 `V4L2_SUBDEV_FL_HAS_DEVNODE` 标志的 subdev 创建设备节点，并且在 media controller 打开时，还会给 subdev entity 和 devnode interface 建 immutable/enabled interface link。也就是说，在 `rkisp1` 里，**complete 不是“主视频节点出生”，而是“整条异步链完整后，给 subdev 暴露 devnode”**。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235&utm_source=chatgpt.com))

**我对第 3 层的源码结论：**

- 在 5.10 async core 里，`bound` 的前提是 subdev 已被 `v4l2_device_register_subdev()` 接纳，所以 `.bound()` 主要负责**保存指针、补 media link、做 topology 装配**。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/v4l2-core/v4l2-async.c))
- `rkisp1` 的异步匹配对象是 **remote endpoint 的 fwnode**，不是随便按名字/I2C 地址猜设备；这比老式匹配更稳，也更能表达 pad 级连接关系。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- `complete()` 在这里的意义不是“整套主设备首次创建”，而是“外部 subdev 全部补齐后，异步层面的 graph 已闭合，可以对 subdev 再做进一步暴露”。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235&utm_source=chatgpt.com))

# 4. media graph：`rkisp1` 源码把 graph 分成“内部静态链”和“外部异步链”两半

`rkisp1_create_links()` 很漂亮地把内部 graph 写死出来了。若硬件有 MIPI CSI2，就先建 `CSI.src -> ISP.sink_video`；然后对两个路径循环建立 `ISP.source_video -> Resizer.sink`，再建 `Resizer.src -> Capture` 的 immutable+enabled 链；之后再建 `params -> ISP.sink_params` 和 `ISP.source_stats -> stats`。这一段源码直接说明：**`rkisp1` 的内部 topology 不是动态发现的，而是 probe 时由驱动按硬件结构主动创建的。**([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

而 `rkisp1_subdev_notifier_bound()` 则补的是外部 sensor 到内部图的第一条边。所以整个 graph 的建立分两类：

- **内部静态链**：probe 时通过 `rkisp1_entities_register()` + `rkisp1_create_links()` 建立
- **外部异步链**：在 `.bound()` 中用 sensor 的 source pad 连到 CSI 或 ISP sink([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

再看 5.10 `mc-entity.c`：`media_entity_pads_init()` 会给 entity 的 pad 逐个设置 `entity/index` 并在 mdev 存在时创建 graph object；`media_create_pad_link()` 则同时在 source 和 sink 两侧创建正反 link，并挂进 media graph。这个核心源码印证了你读驱动时的一个重要判断标准：**只要驱动调用了 `media_entity_pads_init()` 和 `media_create_pad_link()`，它就不是“逻辑上有个链路”，而是“media graph 里真的长出了一条边”。**([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/mc/mc-entity.c))

**我对第 4 层的源码结论：**

- `rkisp1` 的 media graph 明显分成两层：内部固定块之间的链路由主驱动静态建，sensor 相关链路由 async `.bound()` 动态补。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- 这意味着调 graph 问题时，先分清是**内部链没建好**还是**外部 sensor 没 bound 上**，不要混查。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
- `video_device` 在这个 graph 里只是 capture/params/stats 的接口节点，不是图的中心。图的中心是 ISP / CSI / sensor 这些 entity。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

# 5. vb2 / runtime：从源码能得出的硬结论，是“数据面和拓扑面是两条并行线”

这次我拿到的 `rkisp1` 可直接读的源码里，runtime 最有价值的入口是 `rkisp1_isr()`。它明确要求 **先调用 `rkisp1_capture_isr()`，再 `rkisp1_isp_isr()`，最后 `rkisp1_csi_isr()`**，理由是 capture 中断处理要先拿当前 `frame_sequence` 对应的已完成帧，避免被后续 ISP vertical sync 先把序号推进。这个顺序本身已经说明：**在 `rkisp1` 里，真正把一帧“结算”给 vb2 的关键时点，是 capture 中断路径，而不是 async 或 media link 层。**([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

再结合 5.10 `videobuf2-core.c`，可以得到 runtime 的通用硬约束：`vb2_core_streamoff()` 最终会走 `__vb2_queue_cancel()`；如果 `start_streaming_called` 为真，就调用驱动的 `stop_streaming()`；而源码里专门有 warning 提醒，如果 stop path 没有把驱动持有的 buffer 全部归还，vb2 会认为驱动清理不正确。同样，`start_streaming` 失败路径也会检查驱动是否还非法持有 buffer。也就是说，**vb2 在 5.10 里把驱动的 buffer 生命周期纪律钉得很死：start 失败要还，stop 时必须全还。**([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/common/videobuf2/videobuf2-core.c))

`vb2-core.c` 里 done queue 的逻辑也能对上用户态：`__vb2_get_done_vb()` 会等待 `done_list` 非空，然后从 done_list 头部取出已完成 buffer。这正对应你前面那个用户态例子里 `select()/DQBUF` 的真实语义：**用户能 `DQBUF`，不是因为硬件“好像采完了”，而是因为驱动在 ISR/DMA 完成路径里已经把 buffer 放进了 vb2 的 done_list。**([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/common/videobuf2/videobuf2-core.c))

从 `rkisp1_probe()` / `remove()` 与 PM 路径还能看出另一层 runtime 结论：runtime PM 的 enable/resume 在 probe 前期就建立好了，suspend/resume 只负责 `clk_bulk_disable_unprepare()` / `clk_bulk_prepare_enable()` 和 pinctrl state 切换。也就是说，**PM 负责“设备能不能活”，vb2/runtime capture 负责“帧能不能流”**，它们是两条并行但分层清晰的线。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**我对第 5 层的源码结论：**

- 在 `rkisp1` 里，拓扑建立和数据完成是两条不同主线：前者靠 DT/async/media，后者靠 IRQ/DMA/vb2。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235&utm_source=chatgpt.com))
- 真正决定 `DQBUF` 是否返回的，不是 graph 是否存在，而是 capture 完成路径是否把 buffer 放进 vb2 done queue。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/common/videobuf2/videobuf2-core.c))
- 停流/错误恢复必须按 vb2 纪律把 buffer 全收回来，否则 5.10 核心层会直接把问题指向 `stop_streaming()`。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/common/videobuf2/videobuf2-core.c))

# 把第 19 点压成最终结论

如果把你第 19 点那条“源码阅读路线图”真正按 5.10 `rkisp1` 读完，我认为最重要的结论有 6 个：

**第一，DT/fwnode 在平台 camera 驱动里本质上是在描述 graph，不是在“给单个驱动配参数”。** `rkisp1` 明确是从本地 endpoint 出发，解析 bus，再找 remote endpoint，把它变成 async 等待项。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**第二，probe 并不会把整条链一次性拉齐。** `rkisp1_probe()` 只负责资源、容器、内部实体和内部链路；外部 sensor 连接被故意推迟到 async 阶段。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**第三，async 的价值不是“方便晚点注册”，而是把不确定的设备出现顺序收敛成确定的 `.bound()` / `.complete()` 时机。** 5.10 core 里 `v4l2_device_register_subdev()` 先于 `.bound()`，这点非常关键。([GitHub](https://raw.githubusercontent.com/torvalds/linux/v5.10/drivers/media/v4l2-core/v4l2-async.c))

**第四，media graph 在 `rkisp1` 里明显分成“内部静态图”和“外部异步边”。** 这直接决定了你排查 graph 问题时的切分方法。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**第五，用户态视频流只是生命周期的后半段。** 在用户看到 `/dev/videoX` 前，大部分拓扑和对象早就已经出生了；真正的运行态核心是 vb2 和中断完成路径。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))

**第六，stop/remove 的逆序清理和 probe 的正序建立一样重要。** `rkisp1_remove()` 和 vb2 core 都强调：notifier、entity、media、v4l2、buffer 必须按依赖反向拆掉，否则就会留下典型“第一次能跑、第二次起不来”的 bug。([git.zx2c4.com](https://git.zx2c4.com/linux-rng/tree/drivers/media/platform/rockchip/rkisp1/rkisp1-dev.c?h=jd%2Fvdso-test-harness&id=12b68040a5e468068fd7f4af1150eab8f6e96235))
